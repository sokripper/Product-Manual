# 结算基础配置 详细设计（F-CFG-01~04）

本文覆盖结算基础配置的**维护（写）侧**设计，与各消费链 LLD（结算/付款/服务商索赔单只读取配置）互补：本文只写「谁能配、配哪张表哪些列、生效范围如何写入」。字段级保存校验阈值（索赔系数取值范围、付款金额档区间连续/不重叠、审批链非空等）已登记 **D-配置-B1（延到开发按合理默认定稿具体阈值）**，本文只标指针、不编具体阈值。effective_from 录入/生效日精确口径与「覆盖更新 vs 追加新生效行」策略登记 **D-配置-B2（待确认）**，本文按「追加新生效行 + 写 effective_from」标指针。

> 权威源：PRD S4.5（F-CFG-01~04）、S2 角色（索赔业务管理员）、原型 `pages/12-结算基础配置.html`、`docs/db-schema/表结构文档.md`（settle_period_config / invoice_whitelist / payment_approval_flow_config / claim_coefficient）。只引用表结构文档已登记真表真列。
> 只读消费侧的选取谓词见：结算 LLD OP1（settle_period_config「有效配置」谓词、invoice_whitelist 开票权限快照谓词）、服务商索赔单 LLD 系数快照（claim_coefficient「当前生效值」谓词）。本文维护侧写入的 effective_from/enabled 即为那些谓词的数据来源。

## 一、先约定：这一页公共的东西

- **谁在操作**：全部四个页签的维护均为 **索赔业务管理员**（PRD S2「索赔业务管理员 / 结算基础配置 / 全局配置」；P11 菜单归属索赔业务管理员）。接口进来先 RBAC 鉴权（未登录 401、越权 403），非该角色一律拦下；配置为全局数据、无区域/服务商数据范围收窄。
- **节点目录**：系统 - 系统配置 - 结算基础配置（原型面包屑「系统配置 / 结算基础配置」，页面 P11）。
- **页面结构（原型 12）**：单页四页签/分区——`结算周期`（顶部页签，页级「保存更改/取消」）、`开票权限白名单`（表格 + ＋添加服务商 / 编辑 / 停用）、`付款审批流（分级）`（表格 + 编辑）、`服务商索赔系数`（默认系数 + 按服务商表格）。
- **不回溯总则（A-16/A-32/A-2）**：所有配置变更只对「下一周期 / 新单」生效，进行中周期与历史数据不回溯（原型顶部 note-bar 明示）。落地手段=消费侧按 `effective_from<=当前/发起日 取最新有效行` 选取（谓词已写入各消费 LLD），维护侧只负责写入 `effective_from` 与 `enabled`，不回改已生成的业务单快照。
- **审计列**：每次写配置表按通用约定填 `create_by/created_at`（新增）或 `update_by/updated_at`（更新）、`version`（乐观锁，`WHERE id=? AND version=?`，不命中报「数据已变更请刷新」）；软删 `deleted_at`。
- **effective_from 写入口径（D-配置-B2 待确认）**：本文对含 `effective_from` 列的三张配置表（周期/白名单/系数）按「保存时写入生效日期、并以追加新生效行方式保留历史供追溯」标指针；生效日是否人工录入、默认值（下一周期起始日 vs 保存次日）、以及是否改为覆盖更新同一行，原型 12 未提供生效日录入控件、PRD 未给精确算法，登记 D-配置-B2 待产品定稿。

## F-CFG-01 结算周期配置

### 功能说明：
- 维护结算周期的频率、自动生成日、周期起止日、手动提前结算开关等，供结算链 OP1 定时生成结算单读取。
- 变更仅对下一周期/新单生效，进行中周期与历史不回溯（A-16）。

### 开发类型：
- 新增节点（配置维护侧，原读侧已有）。

### 节点目录：
- 系统 - 系统配置 - 结算基础配置 - 结算周期（页签）。

### 业务流转说明：
- 索赔业务管理员在「结算周期」页签修改表单项，点「保存更改」→ 校验后写 `settle_period_config`（按 D-配置-B2 追加新生效行、写 `effective_from`）→ 结算链 OP1 下次触发时按「有效配置」谓词读到当期生效行。点「取消」不保存、还原表单。

### 涉及数据表：
- `settle_period_config`（结算周期配置）：本节点写入 `frequency / gen_day / period_start_day / period_end_day / manual_advance / effective_from`（+审计列）——表结构见 `docs/db-schema/`。

### 解决方案：
- **查询结果（当前生效配置回显）**：读 `settle_period_config` 当前生效行（谓词同 OP1：`deleted_at IS NULL AND COALESCE(effective_from,'1000-01-01')<=CURRENT_DATE`，取最新），回填各表单项。
- **表单字段（对应原型 12「结算周期」六项）**：
    - `默认结算频率` → `frequency`：可编辑；本期固定 `MONTHLY`（PRD F-CFG-01「本期固定按月」；是否开放多频率待挂载）。
    - `自动生成结算单日` → `gen_day`：可编辑，整数（每月第 N 日自动生成）。
    - `结算起止日期` → `period_start_day` / `period_end_day`：可编辑，整数（原型示例「1 日 ~ 月末」；月末钳制口径见结算 LLD OP1 周期算法）。
    - `手动提前结算` → `manual_advance`：可编辑开关（原型「允许 ✓」；0/1）。
    - `结算归属依据` → `belong_basis`：**只读**（原型 `input ro`，恒=闪欣审批通过时间 `APPROVED_TIME`，本期不可改）。
    - `零金额结算单` → `zero_amount_rule`：**只读**（原型 `input ro`，恒=`NOT_GENERATE` 不生成、0 元核销，A-34）。
- **操作按钮**：
    - **保存更改**：
        - 校验：字段级阈值（gen_day/起止日合法范围等）按 **D-配置-B1** 合理默认，本文不编具体阈值。
        - 写入 `settle_period_config`（D-配置-B2：追加新生效行 / 或覆盖更新，待定稿）：
            - `frequency = 表单值（本期 MONTHLY）`
            - `gen_day = 表单值`
            - `period_start_day = 表单值`
            - `period_end_day = 表单值`
            - `manual_advance = 表单值(0/1)`
            - `belong_basis = APPROVED_TIME`（只读不变）
            - `zero_amount_rule = NOT_GENERATE`（只读不变）
            - `effective_from = 生效日期`（D-配置-B2 待确认精确取值）
        - 生效：仅对下一周期/新单，进行中周期不回改（A-16，靠消费侧谓词兜底）。
    - **取消**：不保存，还原为当前生效配置。

### 编码规则：无。
### 外部接口：无。
### 导入：无。
### 防重复与并发：乐观锁 `version`（后端维护、接口不传）；并发保存以 version 不匹配拦截。

## F-CFG-02 服务商开票权限（白名单）配置

### 功能说明：
- 维护「哪些服务商在正数结算时可开票」的白名单（白名单=开票权限，A-32）。
- 中途取消对在开票中的单不回溯（A-16/A-32），仅影响后续新结算单的开票权限快照。

### 开发类型：
- 新增节点（维护侧）。

### 节点目录：
- 系统 - 系统配置 - 结算基础配置 - 开票权限白名单（分区）。

### 业务流转说明：
- 索赔业务管理员在白名单表格「＋添加服务商 / 编辑 / 停用」→ 写 `invoice_whitelist`（`enabled` / `effective_from`）→ 结算链 OP5 生单时按谓词把 `invoice_permission_flag` 快照进结算单；OP6 恢复挂起单按当前 enabled+effective_from 重判。

### 涉及数据表：
- `invoice_whitelist`（开票权限白名单）：本节点写入 `provider_id / provider_name / region / enabled / effective_from`（+审计列）——`uk_provider(provider_id)` 每服务商一行。

### 解决方案：
- **查询结果（表格，对应原型 12 白名单表）**：
    - 数据范围：`deleted_at IS NULL`；全局配置无数据权限收窄。
    - `服务商名称` = `invoice_whitelist.provider_name`
    - `所属区域` = `invoice_whitelist.region`
    - `状态` = 由 `enabled` 渲染（1=已启用 / 0=未启用）
- **操作按钮**：
    - **＋添加服务商**（表单）：
        - `provider_id`：从服务商模块选取（**待挂载/Mock**）；`provider_name / region` 由服务商模块带出。
        - `enabled`：默认 1（添加即启用，原型新增行为「已启用」）。
        - 保存：写入 `invoice_whitelist`：
            - `provider_id = 选取值`
            - `provider_name = 带出值`
            - `region = 带出值`
            - `enabled = 1`
            - `effective_from = 生效日期`（D-配置-B2 待确认）
        - 判重：同 `provider_id` 已存在有效行则拒绝（`uk_provider`）。
    - **编辑**：改 `provider_name / region / effective_from`（provider_id 不可改）；写 `update_by/updated_at/version`。
    - **停用**：`enabled = 0`（A-32 白名单停用；不软删、保留行）。停用对「在开票中」的单不回溯（A-16），仅影响后续新结算单快照。

### 编码规则：无。
### 外部接口：服务商基础数据（服务商模块，待挂载/Mock）——添加服务商时取 provider_id/name/region；本期 Mock，真库到位回填。
### 导入：无。
### 防重复与并发：`uk_provider` 唯一 + 乐观锁 `version`。

## F-CFG-03 付款审批流配置

### 功能说明：
- 维护付款分级三档金额线（<5万 / 5万~20万 / ≥20万）与各档审批节点角色链（A-24）。
- 付款单提交时按申请金额匹配档位、读取当档 node_chain 生成审批链。

### 开发类型：
- 新增节点（维护侧）。

### 节点目录：
- 系统 - 系统配置 - 结算基础配置 - 付款审批流（分区）。

### 业务流转说明：
- 索赔业务管理员在「付款审批流（分级）」表格逐档「编辑」金额线与审批链 → 写 `payment_approval_flow_config`（`amount_min / amount_max / node_chain`）→ 付款链在付款单提交时按金额匹配 `grade_level`、读当档 `node_chain` 落审批节点。

### 涉及数据表：
- `payment_approval_flow_config`（付款审批流配置）：本节点写入 `grade_level / amount_min / amount_max / node_chain / enabled`（+审计列）。

### 解决方案：
- **查询结果（表格，对应原型 12 审批流表）**：
    - `金额档位` = 由 `grade_level` + `amount_min` / `amount_max` 渲染（L1/L2/L3；`amount_max=NULL`=最高档无上限）
    - `审批链` = `node_chain`（JSON 有序角色链）解析展示
- **操作按钮**：
    - **编辑**（逐档）：
        - `amount_min` / `amount_max`：金额线（`amount_max=NULL` 表示最高档无上限）。
        - `node_chain`：有序审批节点角色链（JSON 数组）。
        - 保存校验：三档区间连续/不重叠、审批链非空——**按 D-配置-B1 合理默认**，本文不编具体阈值。
        - 写入 `payment_approval_flow_config`：`amount_min / amount_max / node_chain = 表单值`；`enabled` 维持/置 1。
- **生效范围**：`payment_approval_flow_config` **无 effective_from 列**——配置对「新提交的付款单」生效（付款单提交时点读当时配置匹配、落审批链快照进 `payment_approval_node`），已进入审批的付款单已快照审批链、不回改（不涉及 A-16 的 effective_from 机制）。

### 编码规则：无。
### 外部接口：无。
### 导入：无。
### 防重复与并发：乐观锁 `version`；三档区间不重叠约束在保存校验（D-配置-B1）。

## F-CFG-04 服务商索赔系数配置

### 功能说明：
- 维护服务商索赔系数：全局默认系数 + 按服务商维度差异化系数（A-2）。
- 新建索赔取发起时点值快照存单，变更不影响历史索赔单（消费侧谓词见服务商索赔单 LLD 系数快照）。

### 开发类型：
- 新增节点（维护侧）。

### 节点目录：
- 系统 - 系统配置 - 结算基础配置 - 服务商索赔系数（分区）。

### 业务流转说明：
- 索赔业务管理员维护默认系数（`provider_id IS NULL` 全局行）与按服务商系数行 → 写 `claim_coefficient`（`coefficient / effective_from / enabled`）→ 新建服务商索赔单时按「当前生效值」谓词取快照写 `claim_order.coefficient_snapshot`。

### 涉及数据表：
- `claim_coefficient`（服务商索赔系数）：本节点写入 `provider_id / coefficient / effective_from / enabled`（+审计列）；`provider_id IS NULL`=全局默认行。

### 解决方案：
- **查询结果（对应原型 12 系数区）**：
    - `当前生效系数（默认）` = `claim_coefficient` 中 `provider_id IS NULL` 的当前生效行 `coefficient`
    - 按服务商表格：`服务商` = 关联服务商名（provider_id 带出，待挂载/Mock）、`系数` = `coefficient`
- **操作按钮**：
    - **维护默认系数**（`provider_id IS NULL` 行）：
        - `coefficient`：系数值；校验范围（系数>0 等）按 **D-配置-B1** 合理默认，本文不编阈值。
        - 保存写入：`provider_id = NULL`、`coefficient = 表单值`、`enabled = 1`、`effective_from = 生效日期`（D-配置-B2 待确认）。
    - **维护/新增按服务商系数行**：
        - `provider_id`：从服务商模块选取（待挂载/Mock）。
        - `coefficient / effective_from / enabled`：同上。
        - 保存写入 `claim_coefficient`：`provider_id = 选取值`、`coefficient = 表单值`、`enabled = 1`、`effective_from = 生效日期`。
    - **启停**：`enabled = 0/1`（停用行不参与消费侧「当前生效值」选取）。
- **生效范围**：变更仅对「发起日在 effective_from 之后」的新建索赔生效，历史快照不回改（A-2；消费侧谓词 `enabled=1 AND effective_from<=发起日 取最新`）。

### 编码规则：无。
### 外部接口：服务商基础数据（服务商模块，待挂载/Mock）——选服务商时取 provider_id/name。
### 导入：无。
### 防重复与并发：乐观锁 `version`；同 provider_id 生效行选取由消费侧「取最新 effective_from」保证，维护侧追加新生效行策略待 D-配置-B2 定稿。
