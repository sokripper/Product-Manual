# 详细设计 · 闪欣向欣旺达索赔单链（后端 LLD）

> 版本：v1.0 ｜ 冻结版本 v1.0 ｜ 冻结 commit 88ed0f850780

本链只管三张表：`xw_claim_order`（闪欣向欣旺达索赔单头）、`xw_claim_detail`（明细）、`claim_audit_record`（审核记录，本链写 `biz_type='XW'` 的行）。附件走公共 `attachment` 表（`biz_type='XW'`）。所有列名一律取自 `docs/db-schema/表结构文档.md` 已登记真列，未登记的一律标「待挂载」，不自造。

只读上游：服务商索赔单链（`claim_order` 及其明细）在业务审核通过时把数据推给本链；本链不回写服务商单。下游：结算链（闪欣结算 F-SET-06）读本链 `PENDING_SETTLE` 的单子做结算，`SETTLED` 由结算链写，本链不写。

对接口径（约定稿）：
- 欣旺达审批：本期**欣旺达角色在本系统内建操作**，无跨系统接口（A-22）。审核就是本系统一个内建角色点按钮，不是调外部系统。
- QMS 判责：本期 **Mock / 人工录入**（A-20）。`xw_claim_order` 上的 `liability_result / internal_dept / liability_ratio` 是 QMS 回显位，本期靠人工或 Mock 回填，生成时留空。

## 1. 状态机（xw_claim_order，7 态）

枚举 `XW_CLAIM_STATUS`：`DRAFT`=草稿 / `PENDING_AUDIT`=待审核 / `PENDING_SETTLE`=待结算 / `REJECTED`=已驳回 / `DECLINED`=已拒绝 / `SETTLED`=已结算 / `VOIDED`=已作废。
审核结果枚举 `XW_AUDIT_RESULT`：`PASS`=通过 / `REJECT`=驳回 / `DECLINE`=拒绝 / `PARTIAL`=部分通过。

| 现态 | 事件 | 新态 | 谁 |
|---|---|---|---|
| （无）| 服务商单业务审核通过且 claim_target=XW → 逐单自动生成 | `DRAFT` | 系统（随 F-CLAIM-07 同事务） |
| `DRAFT` / `REJECTED` | 编辑（补附件，不动明细） | 态不变 | 闪欣索赔专员 |
| `DRAFT` | 提交 | `PENDING_AUDIT` | 闪欣索赔专员 |
| `DRAFT` | 作废 | `VOIDED`（终） | 闪欣索赔专员 |
| `PENDING_AUDIT` | 欣旺达审核=通过 | `PENDING_SETTLE` | 欣旺达索赔专员（内建） |
| `PENDING_AUDIT` | 欣旺达审核=部分通过（写核准金额） | `PENDING_SETTLE` | 欣旺达索赔专员（内建） |
| `PENDING_AUDIT` | 欣旺达审核=驳回 | `REJECTED` | 欣旺达索赔专员（内建） |
| `PENDING_AUDIT` | 欣旺达审核=拒绝 | `DECLINED`（终） | 欣旺达索赔专员（内建） |
| `REJECTED` | 编辑后重新提交 | `PENDING_AUDIT` | 闪欣索赔专员 |
| `PENDING_SETTLE` | 结算完成 | `SETTLED`（终） | **下游结算链写，本链不写** |

拦截（一律先校验现态，不符直接报错回滚，不改任何字段）：
- 只有 `DRAFT`、`REJECTED` 能编辑 / 提交；其余态提交一律拒。（编辑前置态：docx 表15 权限矩阵「编辑仅草稿」系口径不精，以表7 / S2.2.2「已驳回可编辑重提」为准，取 `DRAFT`+`REJECTED`——见 S6 H3。）
- 只有 `PENDING_AUDIT` 能被欣旺达审核；重复审核（已流转到 `PENDING_SETTLE/REJECTED/DECLINED`）拒。
- 只有 `DRAFT` 能作废；`PENDING_AUDIT` 不允许撤回作废（D-向欣-B1 已确认）。
- `SETTLED / DECLINED / VOIDED` 为终态，任何写操作拒。
- 所有写操作走乐观锁：`WHERE id=? AND status=? AND version=? AND deleted_at IS NULL`，`version=version+1`；命中 0 行＝并发冲突，回滚提示刷新。

## 2. 编码规则

`xw_claim_order.claim_no` = `SXCO-yyyyMMdd-4位日内流水`，全局唯一（表上 `UNIQUE`）。按已确认的 D-编码-D1/D2 方案从 `doc_no_sequence` 以「前缀+日期」同事务行锁取号，同日从 0001 递增；业务 INSERT 成功才占号、回滚退号，表上 `UNIQUE` 兜底。生成在自动生成操作里完成；序列表 DDL 仍待研发组长审定。

## 3. 操作详解

### 3.1 逐单自动生成（A-36）

**谁能触发**：不是人点的。服务商索赔单在「业务审核」（F-CLAIM-07）被闪欣索赔专员点通过、且该服务商单 `claim_target='XW'`（欣旺达件）时，**系统在同一事务里逐单自动生成一张 `xw_claim_order` + 一条 `xw_claim_detail`**（非汇总，一张服务商单对一张闪欣单一条明细）。

**前置**：服务商单 `claim_order.status` 将由 `PENDING_HQ` 迁到 `PENDING_SETTLE`（欣旺达件不经判责，S6.1）；且 `claim_order.claim_target='XW'`。

**幂等校验**：同一 `claim_order.id` 只能生成一次——落 `xw_claim_detail.provider_claim_id`（`UNIQUE uk_provider_claim`）挡重复；若已存在则跳过生成（业务审核只会通过一次，正常不触发）。

**写 `xw_claim_order`（一行）**：
| 列 | 值 |
|---|---|
| id | 雪花算法生成 |
| claim_no | `SXCO-yyyyMMdd-流水`（见 S2） |
| provider_claim_id | = `claim_order.id`（触发的服务商单） |
| provider_claim_no | = `claim_order.claim_no` |
| work_order_no | = `claim_order.work_order_no` |
| repair_type / sub_repair_type | 工单库带出（**待挂载/Mock**；未提供写 NULL） |
| battery_sn / serial_no / vin | 工单库带出（**待挂载/Mock**；未提供写 NULL） |
| provider_code / provider_name / region | = `claim_order.provider_code / provider_name / region` |
| province_city | 工单/服务商库带出（**待挂载**；未提供写 NULL，与 schema/DDL 可空口径一致） |
| labor_fee_total | = `claim_order.labor_fee_total` |
| part_mgmt_fee | = `claim_order.part_mgmt_fee` |
| other_fee_total | = `claim_order.other_fee_total` |
| apply_total_amount | `labor_fee_total + part_mgmt_fee + other_fee_total`（含备件管理费，等于服务商单总额；不乘系数 A-37；D-向欣-B2 已确认） |
| approved_amount | NULL |
| currency | `'CNY'` |
| status | `'DRAFT'` |
| audit_result | NULL |
| liability_result / internal_dept / liability_ratio | NULL（QMS 回显位，本期人工/Mock 回填 A-20） |
| fee_transfer_started / provider_deduct_started | 0 |
| create_by | 触发的业务审核操作人（记为系统自动 + 操作人） |
| created_at / updated_at | NOW |
| version | 0 |

**写 `xw_claim_detail`（一条）**：
| 列 | 值 |
|---|---|
| id | 雪花算法生成 |
| xw_claim_id | = 新单 `xw_claim_order.id` |
| provider_claim_id | = `claim_order.id`（`UNIQUE uk_provider_claim`，占位，防被别的闪欣单再纳入） |
| provider_claim_no | = `claim_order.claim_no` |
| work_order_no | = `claim_order.work_order_no` |
| labor_fee_total | = `claim_order.labor_fee_total` |
| other_fee_total | = `claim_order.other_fee_total` |
| subtotal | = `labor_fee_total + other_fee_total` |

**新状态**：`DRAFT`。
**触发下游**：无（等专员补附件/提交）。
**同事务/回滚**：生成挂在 F-CLAIM-07 业务审核通过的**同一数据库事务**里——服务商单置 `PENDING_SETTLE`、闪欣单落 `DRAFT`、明细落一条，三者要么全成要么全滚。生成失败（如撞号重试仍失败）整笔业务审核回滚。

### 3.2 编辑（草稿 / 已驳回）

**谁能点**：闪欣索赔专员。
**前置状态**：`DRAFT` 或 `REJECTED`（已驳回可改后重提，原型列表「已驳回」行有编辑按钮；docx 表15「编辑仅草稿」口径不精，以表7 / S2.2.2「已驳回可编辑重提」为准，见 S6 H3）。其余态编辑拒。

**一张闪欣单严格对应一张服务商索赔单（A-36：逐单自动生成、非汇总）**：`xw_claim_detail` 承载的是本单对应那**一张**服务商索赔单的明细快照，不是多选池。编辑**不允许增删 / 合并明细**、不允许把别的服务商单勾选纳入本单——那会变成「多明细合一单 / 汇总生成」，直接违反已确认裁决 A-36。原型 04 的「勾选服务商索赔单明细 / 新增明细 / 删除已勾选 / 已纳入 SXCO-…0002」多选 UI，以 A-36 为准废止、不落地。

**能改什么**（只动附件，不动明细、不动金额）：
1. **补附件**：写 `attachment`（`biz_type='XW'`，`biz_id=` 本单 id，落 `file_name/file_path/file_type/file_size/upload_by/upload_time`，本地磁盘存相对路径）。

**写字段**（同事务）：
- 仅 `attachment` 表 insert；当前登记的 `xw_claim_order` 无备注列，本期不提供独立文本说明编辑，不借未登记列落库。`xw_claim_order` 的明细相关金额字段（`labor_fee_total / part_mgmt_fee / other_fee_total / apply_total_amount`）与 `xw_claim_detail` **一律不改**——金额在 3.1 生成时按对应服务商单一次性定稿，编辑阶段不重算。
- `version = version+1`，`update_by / updated_at` 记操作人 / 时间。

**新状态**：不变（`DRAFT` 还 `DRAFT`，`REJECTED` 还 `REJECTED`）。
**触发下游**：无。
**同事务/回滚**：附件写入 + 头表 version 递增同一事务；乐观锁冲突整体回滚。

### 3.3 提交

**谁能点**：闪欣索赔专员。
**前置状态**：`DRAFT` 或 `REJECTED`。

**校验**：
- 现态必须 `DRAFT/REJECTED`，否则拒。
- 明细 ≥ 1 条。
- 条件必填（A-39）：电池包维修 `battery_sn` 必填、电机电控 `serial_no` 必填、整车 `vin` 17 位（内部场景豁免）。这些字段本期由工单库 Mock 带出——**待挂载**：本期 Mock 未带出时放行不卡；真库接上后按 A-39 改为硬校验（D-向欣-B3 已确认）。

**写字段**：
- `status`：`DRAFT/REJECTED` → `PENDING_AUDIT`
- `version = version+1`，`update_by / updated_at` 记提交人/时间。

**新状态**：`PENDING_AUDIT`。
**触发下游**：把单子推给欣旺达索赔专员待办审核（内建角色，无跨系统调用 A-22）。
**同事务/回滚**：单表状态更新，乐观锁一把；冲突回滚。

### 3.4 欣旺达审核（通过 / 驳回 / 拒绝 / 部分通过）

**谁能点**：欣旺达索赔专员（本系统内建角色 A-22）。
**前置状态**：`PENDING_AUDIT`。

**入参**：`audit_result` ∈ {`PASS`,`REJECT`,`DECLINE`,`PARTIAL`}；审批意见 `remark`（驳回/拒绝必填）；部分通过时核准金额 `approved_amount`（必填）；附件可选。

**校验**：
- 现态必须 `PENDING_AUDIT`，否则拒（挡重复审核）。
- `audit_result='PARTIAL'` 时 `approved_amount` 必填，且 `0 < approved_amount ≤ apply_total_amount`（核准金额不得超过申请总额、不得为 0/负）。
- `audit_result ∈ {REJECT,DECLINE}` 时 `remark` 必填。

**写字段（按结果分支，均在同事务）**：

| 结果 | `xw_claim_order` 写 | 新状态 |
|---|---|---|
| 通过 `PASS` | `status='PENDING_SETTLE'`、`audit_result='PASS'`、`approved_amount` 保持 NULL（NULL＝全额通过，结算按 `apply_total_amount`） | `PENDING_SETTLE` |
| 部分通过 `PARTIAL`（A-40） | `status='PENDING_SETTLE'`、`audit_result='PARTIAL'`、`approved_amount=` 录入核准金额（**结算口径以此为准，覆盖 apply_total_amount**） | `PENDING_SETTLE` |
| 驳回 `REJECT` | `status='REJECTED'`、`audit_result='REJECT'` | `REJECTED`（可编辑重提） |
| 拒绝 `DECLINE` | `status='DECLINED'`、`audit_result='DECLINE'` | `DECLINED`（终） |

四种结果都写一条 **`claim_audit_record`**：
| 列 | 值 |
|---|---|
| id | 雪花 |
| biz_type | `'XW'` |
| biz_id | = 本 `xw_claim_order.id` |
| audit_stage | `'欣旺达审核'` |
| auditor | 欣旺达索赔专员 |
| audit_time | NOW |
| audit_result | `通过`/`驳回`/`拒绝`/`部分通过`（中文语义，对齐该列既有取值）|
| approved_amount | 部分通过＝核准金额；其余＝NULL |
| remark | 审批意见 / 驳回原因 / **部分通过的核减明细说明**（如「SPCO-…0044 工时费核减 ¥520.00」）；本期以文本记录、不建专表（D-向欣-B4 已确认） |
| attachment_ids | 审核说明附件 id 列表（可空） |

`xw_claim_order.version = version+1`。

**新状态**：`PENDING_SETTLE` / `REJECTED` / `DECLINED`。
**触发下游**：
- 进 `PENDING_SETTLE` 后，闪欣结算链（F-SET-06）汇总本链待结算单做欣旺达→闪欣结算，**按 `approved_amount`（部分通过）或 `apply_total_amount`（全额通过）取数**；结算完成由结算链把本单置 `SETTLED`。
- QMS 判责回显（`liability_result` 等）本期由下述 Mock 操作人工回填，不与欣旺达审核事务捆绑。

**同事务/回滚**：头状态更新 + 审核记录写入同一事务，乐观锁一把；冲突回滚，审核记录不落。

### 3.4a QMS 判责 Mock 人工回填（A-20）

**谁能点**：具有“QMS Mock 回填”权限的欣旺达索赔专员；真实 QMS 契约到位后该入口由适配层回调替换。

**允许状态**：`DRAFT/REJECTED/PENDING_AUDIT/PENDING_SETTLE`，且该单尚未进入任何有效 `claim_settle_detail`。先锁 `xw_claim_order` 行并复核状态/结算占用；已结算、已作废/拒绝或已纳入结算单时拦截，避免分账依据在结算中途被改写。

**入参与校验**：
- `liability_result` 必填，只能是 `SUPPLIER/PROVIDER/INTERNAL`；`liability_ratio` 必填且 `0 < liability_ratio <= 100`。
- `liability_result='INTERNAL'` 时 `internal_dept` 必填；其余结果显式写 `internal_dept=NULL`。判责说明必填，可带证据附件 id。

**写入（同事务）**：
- `xw_claim_order`：写 `liability_result/internal_dept/liability_ratio`；`PROVIDER` 时 `provider_deduct_started=1, fee_transfer_started=0`，`SUPPLIER/INTERNAL` 时 `fee_transfer_started=1, provider_deduct_started=0`。本期两个标志只表示“已记录后续抵扣/转嫁意图”，不伪造外部单据；OA/供应商子流程仍待挂载。刷新 `update_by/updated_at/version`。
- `claim_audit_record`：插入 `biz_type='XW'`、`biz_id`=本单 id、`audit_stage='QMS_MOCK'`、`audit_result`=上述判责枚举、`auditor/audit_time`=回填人/时间、`remark`=判责说明、`attachment_ids`=证据附件 id 列表。

**状态与回滚**：单据状态不变；头表与留痕记录同事务，任一失败整体回滚。重复回填会新增一条历史并覆盖头表当前值，保留可追溯的修改轨迹。

### 3.5 作废

**谁能点**：闪欣索赔专员。
**前置状态**：`DRAFT`（仅草稿；`PENDING_AUDIT` 不允许撤回作废，D-向欣-B1 已确认）。

**写字段**：
- `status = 'VOIDED'`（终态）
- `xw_claim_detail` 明细保留且 `deleted_at` 保持 NULL；`uk_provider_claim` 占用**不释放**，同一服务商索赔单本期不会再生成第二张向欣旺达索赔单。
- `version = version+1`，`update_by / updated_at`。

**新状态**：`VOIDED`。
**触发下游**：无。作废只取消“向欣旺达追偿”分支；上游 `claim_order` 仍保持 `PENDING_SETTLE`，继续进入闪欣向服务商付款的服务商结算链，不因追偿单作废而取消对服务商的应付。
**同事务/回滚**：仅头表乐观锁置 `VOIDED`；明细不删除、不释放占用。冲突则回滚。

### 3.6 查看

**谁能点**：闪欣索赔专员、欣旺达索赔专员（全状态可查看）。
**读**（无写）：
- 头：`xw_claim_order` 全字段。
- 明细：`xw_claim_detail`（`WHERE xw_claim_id=? AND deleted_at IS NULL`）。
- 审核记录：`claim_audit_record`（`WHERE biz_type='XW' AND biz_id=? ORDER BY audit_time`）。
- 附件：`attachment`（`WHERE biz_type='XW' AND biz_id=?`）。
- 关联服务商索赔单：按 `provider_claim_id` 反查 `claim_order`（读取，不写）。

**列表页**（原型 03）：按索赔单编号 / 关联服务商索赔单号 / 状态 / 创建时间筛选；导出按当前筛选条件。全部 `WHERE deleted_at IS NULL`。

## 4. 金额规则（本链，A-37 / A-40）

- **申请总额**：`apply_total_amount = labor_fee_total + part_mgmt_fee + other_fee_total`（工时费 + 备件管理费 + 其它费用），**一律不乘系数**（A-37）。备件管理费计入向欣旺达/闪欣结算金额，等于服务商单总额（D-向欣-B2 与 D-结算-B5 已联动确认）；原型 04 仅显示工时费+其它费用的旧口径废止。
- **明细小计**：`xw_claim_detail.subtotal = labor_fee_total + other_fee_total`（明细表未登记备件管理费列，备件管理费在头表 `part_mgmt_fee` 承载）。
- **结算口径**：全额通过按 `apply_total_amount` 结算；部分通过按 `approved_amount` 结算（A-40，核准金额覆盖申请总额，进待结算即以核准金额入结算汇总）。
- 币种恒 `CNY`（A-29）。

## 5. 事务、并发与幂等小结

- **逐单自动生成**挂在服务商单业务审核通过的同一事务里，服务商单状态 + 闪欣单 + 明细原子落库。
- **生成幂等**：`uk_provider_claim` 挡同一服务商单被重复生成（一张服务商单只对一张闪欣单，A-36 非汇总）。
- **核心写**（编辑、提交、审核、作废）全走乐观锁 `version`，命中 0 行即并发冲突回滚。
- **编码并发**：`SXCO` 日内流水用号段/序列 + `UNIQUE` 兜底重试防撞号。
