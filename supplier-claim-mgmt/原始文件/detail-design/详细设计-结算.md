## 一、业务流转总览

结算链有两条主体：**服务商结算**（`settle_type=PROVIDER`，闪欣结算给服务商，页面 P05/P06）与**闪欣结算**（`settle_type=SHANXIN`，欣旺达结算给闪欣，页面 P07/P08）。两条共用一张 `claim_settle_order` 与 `claim_settle_detail`，靠 `settle_type` 区分角色与流向。

**服务商结算的数据从哪来、到哪去。** 定时任务在每月 `settle_period_config.gen_day` 0 点触发，按「服务商」维度扫描 `claim_order` 中 `status=PENDING_SETTLE` 且尚未纳入任何有效结算单的索赔单，按归属时间落入当期周期窗口，再扫描同服务商同周期 `claim_deduct_order` 中 `status=UNSETTLED` 的服务商责任抵扣单。应付总金额 = 索赔单应付合计 − 抵扣合计【A-30/S9】。应付 > 0 时生成一张 `claim_settle_order`（`PENDING_CONFIRM`），把每张索赔单与每张抵扣单落成 `claim_settle_detail` 行，并把纳入的抵扣单置 `SETTLED`；应付 = 0 时**不生成结算单**，直接把参与的索赔单置 `SETTLED`（0 元核销）、抵扣单置 `SETTLED`【A-34】。同服务商同周期同 `settle_type` 只允许 1 张有效结算单。

结算单生成后，索赔专员核对可增删明细并确认结算（`PENDING_CONFIRM → PENDING_RECON`），服务商对账确认。对账确认时按应付正负分流：正数校验开票权限白名单后进 `PENDING_INVOICE`（无权限停在 `PENDING_INVOICE` 并标记挂起）【A-32】；负数关闭开票权限、进 `PENDING_FORCE`【A-33】。正数线：服务商开票上传发票（`→PENDING_VERIFY`）→ 闪欣发票确认（`→VERIFIED`）→ 闪欣侧人工申请付款触发付款链（`payment_apply`，本链之外）【A-28】→ 付款链回写付款完成，结算单与其名下索赔单同事务置 `SETTLED`。负数线采用两步：索赔专员先发起强制结算并上传扣减依据，财务二次确认后才从 `warranty_deposit_account` 执行扣减、写 `warranty_deposit_txn`；余额不足扣全部并把差额记 `shortfall_carryover`，再把结算单与索赔单同事务置 `SETTLED`（D-结算-B4 已确认）——**财务执行的质保金扣减与结算完结必须同一事务，任一步失败整体回滚**。

**闪欣结算的数据从哪来、到哪去。** 按「欣旺达 ↔ 闪欣」维度汇总 `xw_claim_order` 中 `status=PENDING_SETTLE` 的向欣旺达索赔单，统一生成一张不拆分开票的结算单，内部分账归属记在明细 `fee_owner`，交财务共享/OA（本期 Mock 记录）【A-35】。角色顺序与服务商结算相反：欣旺达确认结算（`PENDING_CONFIRM→PENDING_RECON`）→ 闪欣对账确认进 `PENDING_INVOICE` → 闪欣开票上传发票（`→PENDING_VERIFY`）→ 欣旺达发票确认（`→VERIFIED`）→ 欣旺达发起付款、财务线下办理（本期 Mock）→ 财务人工回填成功后结算单置 `SETTLED`【A-18/S6.5】。闪欣结算恒为正数、不走质保金扣减、不生成 `payment_apply`。

**表间数据流转顺序**：`claim_order`/`claim_deduct_order`（上游只读）→ 生成写 `claim_settle_order` + `claim_settle_detail`、回写 `claim_deduct_order.status`；对账正负分流读 `invoice_whitelist`；负数完结写 `warranty_deposit_account` + `warranty_deposit_txn`；正数完结由 `payment_apply`（下游）回写；最终把 `claim_order.status` 由 `PENDING_SETTLE` 推到 `SETTLED`。

## 三、状态机（claim_settle_order 8 态）

状态全集（`SETTLE_STATUS`）：`PENDING_CONFIRM 待确认 / PENDING_RECON 待对账 / PENDING_INVOICE 待开票 / PENDING_VERIFY 待核验 / VERIFIED 已核验 / PENDING_FORCE 待强制结算 / SETTLED 已结算 / VOIDED 已作废`。「账单挂起」不是独立状态，是 `PENDING_INVOICE` 下 `invoice_permission_flag=0` 的挂起标记【A-32】。

| 现态 | 迁移动作（谁能点·前置） | 新态 | 触发副作用 |
|---|---|---|---|
| —（不存在） | 定时/手动生成，应付>0 | `PENDING_CONFIRM` | 写结算单+明细，抵扣单置 SETTLED |
| —（不存在） | 定时/手动生成，应付=0 | 不建单 | 索赔单直接 SETTLED（0 元核销），抵扣单 SETTLED【A-34】 |
| `PENDING_CONFIRM` | 索赔专员确认结算 | `PENDING_RECON` | — |
| `PENDING_CONFIRM` | 索赔专员作废（仅此态可作废） | `VOIDED` | 删明细、抵扣单回滚 UNSETTLED |
| `PENDING_RECON` | 收款方对账确认·应付>0 | `PENDING_INVOICE` | 写 `recon_confirm_time`；校验白名单置 `invoice_permission_flag` |
| `PENDING_RECON` | 收款方对账确认·应付<0（仅服务商结算） | `PENDING_FORCE` | 写 `recon_confirm_time`；`invoice_permission_flag=0`、`force_settle_flag=1`【A-33】 |
| `PENDING_INVOICE` | 开票方开票上传发票（`invoice_permission_flag=1`） | `PENDING_VERIFY` | 写发票四字段+附件 |
| `PENDING_VERIFY` | 核验方发票确认 | `VERIFIED` | — |
| `VERIFIED` | 服务商结算：闪欣申请付款 | 保持 `VERIFIED` | 触发付款链 `payment_apply`（本链外）【A-28】 |
| `VERIFIED` | 服务商结算：付款链回写付款完成 | `SETTLED` | 写 `settle_time`；索赔单 SETTLED（同事务） |
| `VERIFIED` | 闪欣结算：财务人工回填付款成功 | `SETTLED` | 写 `settle_time`；索赔单 SETTLED；分账记财务共享/OA（Mock）【A-35】 |
| `PENDING_FORCE` | 索赔专员发起强制结算 | 保持 `PENDING_FORCE` | 上传扣减依据，等待财务二次确认 |
| `PENDING_FORCE` | 财务确认并执行扣减 | `SETTLED` | 质保金扣减+写流水+写 `settle_time`+索赔单 SETTLED（**同事务**）【A-31/D-结算-B4】 |

**联动状态机 · claim_order（只读上游，仅由本链推末态）**：`PENDING_SETTLE` 是本链的入口态。生成结算单**不改** `claim_order.status`（仍 `PENDING_SETTLE`），靠「已在有效 `claim_settle_detail` 中」把它从下一次生成扫描里排除（`claim_order` 无「结算中」列；快照/专列为待文档 B-1）。仅在结算单到 `SETTLED`（正数付款完成 / 负数强制结算完成）或 0 元核销时，`claim_order.status: PENDING_SETTLE → SETTLED`。结算单 `VOIDED` 时其名下索赔单不改状态、仅移出明细，恢复可被扫描。

**联动状态机 · claim_deduct_order**：`UNSETTLED`（未结算）→ 生成结算单纳入时置 `SETTLED`（已纳入结算）；结算单 `VOIDED` 时回滚 `SETTLED → UNSETTLED`。0 元核销时同样置 `SETTLED`。

**非法迁移拦截**：任何动作先校验 `现态 ∈ 允许集`，否则返回业务码「状态不允许该操作」、不写库。作废仅 `PENDING_CONFIRM` 可点，其余态点作废一律拦截。负数结算单不得进 `PENDING_INVOICE`；正数结算单不得进 `PENDING_FORCE`；强制结算仅 `PENDING_FORCE` 且 `payable_amount<0` 可用。闪欣结算（`settle_type=SHANXIN`）无 `PENDING_FORCE` 分支（恒正数）。所有更新走乐观锁 `WHERE id=? AND status=? AND version=?`，`version=version+1`；`version` 不匹配返回「数据已变更请刷新」。

## 四、操作副作用（每操作自包含 · 字段级 · 事务边界）

> 约定：下列「写 表: 字段=值」只列业务字段，审计列每次写入按通用约定填 `create_by/created_at` 或 `update_by/updated_at`、`version`；置空一律显式写 NULL。金额 `DECIMAL(18,2)`。

### OP1 自动生成结算单（定时任务，每月 gen_day 0 点）

**谁能点**：系统定时任务（Spring 调度 / XXL-Job，研发侧定），无人工。
**前置**：读 `settle_period_config`（取 `frequency=MONTHLY`、`gen_day`、`period_start_day`、`period_end_day`、`belong_basis`、`zero_amount_rule=NOT_GENERATE` 的有效配置）。「有效配置」选取谓词：`deleted_at IS NULL AND COALESCE(effective_from,'1000-01-01') <= CURRENT_DATE`，多行命中时按 `effective_from DESC, created_at DESC` 取最新一行（与白名单 effective_from 同口径）。任务启动时快照当时生效的配置；任务开始后的配置修改只对下一个未生成周期生效，不回改已生成结算单（`effective_from` 晚于当前的配置行在任务启动时点尚未生效，天然被谓词排除，即 A-16 的运行期兜底）。

**周期日期生成算法**：
1. 以任务触发时点为锚，在历史日期中找到“严格早于本次触发时间”的最近一个 `period_end_day` 作为 `settle_period_end`；如配置 31 而目标月只有 28/29/30 天，钳制为该月最后一天。
2. 若 `period_start_day <= period_end_day`，`settle_period_start` 取 `settle_period_end` 所在月的 start day；若 `period_start_day > period_end_day`（例如 26~25），取上一月的 start day；start day 同样按该月月末钳制。
3. 存储周期含首含尾；查询统一计算 `period_end_exclusive = settle_period_end + 1 日`，使用半开区间 `[settle_period_start, period_end_exclusive)`。默认 1~31 因此正确得到上一自然月 1 日~月末，26~25 则得到跨月周期。
**归属时间口径**：`belong_basis=APPROVED_TIME`——服务商结算按索赔单「闪欣业务审核通过时间」归属；闪欣结算按「欣旺达审批通过时间」归属。`claim_order`/`xw_claim_order` 无专门的「审批通过时间」列，归属时间本期从 `claim_audit_record` 派生（`biz_type=CLAIM`/`XW`、`audit_stage=业务审核`/`欣旺达审核`、`audit_result=通过`、取 `audit_time`）。新增快照列属于待 schema 文档项 D-结算-B1，正式文档到位后回填。

**校验**：
- 幂等键 = `settle_type + provider_id(SHANXIN 为空) + settle_period_start + settle_period_end`。同键有效单为 `PENDING_CONFIRM` 时，重跑不新建，而是在行锁下合并本执行周期新出现的未占用明细（含按 D-结算-B6 滚入的迟到项）并重算金额；已进入对账及后续状态时不改旧单，迟到项留给再下一次可执行周期。
- 同服务商同周期同 `settle_type` 至多 1 张有效结算单（Service 层，非物理唯一索引）。
- **迟到数据处置（D-结算-B6 已确认）**：任务另行检出“原归属时间 < 本执行周期起始日且仍未占用”的索赔/抵扣，纳入**下一次可执行结算周期**。不补开原周期单、不重开或回改已生成结算单；生成的结算单写下一执行周期的 `settle_period_start/end`。源单归属日期保持不变：服务商/闪欣索赔仍以原 `claim_audit_record.audit_time` 追溯，抵扣仍以原 `claim_deduct_order.settle_period_belong` 追溯；明细通过 `claim_no/xw_claim_no` 回查源单，列表/导出展示“执行周期 + 原归属日期”。

**服务商结算（settle_type=PROVIDER）逐服务商聚合**：
1. 取候选索赔单：`claim_order WHERE status='PENDING_SETTLE' AND deleted_at IS NULL AND provider_id=:pid AND 归属时间 < period_end_exclusive AND 该 id 不在任何 status<>VOIDED 的 claim_settle_detail 中`。其中归属时间早于本执行周期起始日者是 D-结算-B6 的迟到滚入项，源审核时间不改。
2. 取候选抵扣单：`claim_deduct_order WHERE status='UNSETTLED' AND deleted_at IS NULL AND provider_id=:pid AND settle_period_belong < period_end_exclusive`（`settle_period_belong` 已由索赔链按闪欣业务审核通过日写入，不得为 NULL）；早于本执行周期起始日者滚入本次，但不改原 `settle_period_belong`。
3. `索赔应付合计 = Σ claim_order.apply_total_amount`；`抵扣合计 = Σ claim_deduct_order.deduct_amount`；`payable_amount = 索赔应付合计 − 抵扣合计`【A-30/S9】。

**分支 A：payable_amount ≠ 0 → 生成结算单**（同事务）：
- 写 `claim_settle_order`：`id`=雪花；`settle_no`=编码（见编码规则，SPJS）；`settle_type='PROVIDER'`；`payee_name`=服务商名；`counterpart_name=NULL`；`provider_id`=:pid；`region`=服务商区域；`settle_period_start`/`settle_period_end`=周期；`payable_amount`=上式；`currency='CNY'`；`claim_count`=纳入索赔单数；`status='PENDING_CONFIRM'`；`invoice_permission_flag`=生单时白名单权限快照（有效=1，否则=0，见 OP5）；`force_settle_flag=0`；`recon_confirm_time=NULL`；`invoice_code/invoice_no/invoice_date/invoice_amount/invoice_attachment_id=NULL`；`settle_time=NULL`。
- 每张索赔单写一行 `claim_settle_detail`：`settle_id`=上单 id；`claim_no`=索赔单号；`xw_claim_no=NULL`；`work_order_no`/`claim_type`/`provider_id`/`provider_code`/`provider_name`/`region`=索赔单带出；`claim_total=apply_total_amount`；`labor_fee_total`/`part_mgmt_fee`/`other_fee_total`=索赔单对应；`deduct_amount=0.00`；`fee_owner=NULL`；`currency='CNY'`。
- 每张抵扣单写一行 `claim_settle_detail`：`settle_id`=上单 id；`claim_no`=抵扣单号 `deduct_no`；`xw_claim_no=NULL`；`work_order_no`=抵扣单工单号；`claim_type=NULL`；`provider_*`=服务商；`region`=区域；`claim_total = -claim_deduct_order.deduct_amount`（负号，抵扣冲减）；`labor_fee_total=0.00`/`part_mgmt_fee=0.00`/`other_fee_total=0.00`；`deduct_amount = claim_deduct_order.deduct_amount`（正）；`fee_owner=NULL`；`currency='CNY'`。
- 回写 `claim_deduct_order`：纳入的每张 `status: UNSETTLED → SETTLED`（`WHERE id=? AND status='UNSETTLED' AND version=?`）。
- `claim_order.status` **不改**（仍 PENDING_SETTLE），靠明细占位排除后续扫描。
- 校验 `payable_amount = Σ claim_settle_detail.claim_total`（一致性断言，不等则回滚报错）。

**分支 B：payable_amount = 0 → 0 元核销**（不建单，同事务）【A-34】：
- 不写 `claim_settle_order`、不写 `claim_settle_detail`。
- 参与的每张 `claim_order.status: PENDING_SETTLE → SETTLED`（`WHERE id=? AND status='PENDING_SETTLE' AND version=?`）。
- 参与的每张 `claim_deduct_order.status: UNSETTLED → SETTLED`。
- 无发票、无付款、无质保金动作。

**闪欣结算（settle_type=SHANXIN）聚合**（同事务，统一一张不拆分）【A-35】：
1. 取候选：`xw_claim_order WHERE status='PENDING_SETTLE' AND deleted_at IS NULL AND 归属时间 < period_end_exclusive AND 该 id 不在任何 status<>VOIDED 的 claim_settle_detail 中`。归属时间早于本执行周期起始日者按 D-结算-B6 滚入本次执行周期，原审核时间保持不变供追溯。
2. `payable_amount = Σ 单张核准金额`，核准金额 = `approved_amount`（部分通过时非空）否则 `apply_total_amount`【A-40】；`apply_total_amount` 含备件管理费（=工时费+备件管理费+其它费用，S9），因此闪欣应付计入欣旺达件备件管理费（D-结算-B5/D-向欣-B2 已确认）。闪欣结算无抵扣、恒 ≥0。
3. 写 `claim_settle_order`：`settle_type='SHANXIN'`；`settle_no`=SXJS 编码；`payee_name`='闪欣动力'；`counterpart_name`='欣旺达售后'；`provider_id=NULL`；`region=NULL`；周期/`payable_amount`/`currency='CNY'`/`claim_count`=纳入闪欣索赔单数；`status='PENDING_CONFIRM'`；`invoice_permission_flag=1`（闪欣主体不套服务商白名单）；其余同分支 A 的 NULL/0 默认。
4. 每张写 `claim_settle_detail`：`claim_no=NULL`；`xw_claim_no`=闪欣索赔单号；`work_order_no`/`provider_*`/`region`=xw 单带出；`claim_type`=对应；`claim_total`=核准金额；`labor_fee_total`/`other_fee_total`=xw 单对应；`part_mgmt_fee = xw_claim_order.part_mgmt_fee`（落库、展示并计入金额，不恒置 0）；`deduct_amount=0.00`；`fee_owner=NULL`（未获得业务确认的自动映射规则，不从 QMS 判责枚举臆测映射）；`currency='CNY'`。
5. 若 `payable_amount=0`：同分支 B，`xw_claim_order.status: PENDING_SETTLE → SETTLED`，不建单。

**失败回滚**：结算单+全部明细+抵扣单回写在同一事务，任一步失败整体回滚；某个服务商维度失败不影响其他维度（各维度独立事务、独立提交）。

### OP2 手动提前结算（手动合并结算单）

**谁能点**：闪欣索赔专员（P05「+ 手动合并结算单」）。**前置**：选定 `settle_type`+服务商（或欣旺达↔闪欣）+周期。

### OP3a 编辑明细（增/删索赔单据）

**谁能点**：`settle_type='PROVIDER'` 为闪欣索赔专员；`settle_type='SHANXIN'` 为欣旺达索赔专员。**前置**：`status='PENDING_CONFIRM'`（仅待确认可增删）。
**服务商结算分支**：删除 `claim_settle_detail`（软删该行）后，对应 `claim_order` 恢复可被扫描；新增时必须是 `claim_order.status='PENDING_SETTLE'` 且不在其他有效结算明细中，插入字段同 OP1 服务商索赔行。
**闪欣结算分支**：删除后对应 `xw_claim_order` 恢复可被扫描；新增时必须是 `xw_claim_order.status='PENDING_SETTLE'` 且不在其他有效结算明细中，插入字段同 OP1 SHANXIN 明细行，`claim_total=COALESCE(approved_amount,apply_total_amount)`、`part_mgmt_fee=xw_claim_order.part_mgmt_fee`。欣旺达索赔专员必须对每行人工选择 `fee_owner ∈ {XW,SHANXIN,INTERNAL_DEPT}`；QMS Mock 判责仅作参考、本期不做无据自动映射。选 `INTERNAL_DEPT` 时关联 `xw_claim_order.internal_dept` 必须非空。
**公共回写**：增删后重算 `claim_count=未删除索赔行数` 与 `payable_amount=Σ claim_total`。
**边界（D-结算-B2 已确认）**：增删后 `payable_amount` 变号时，按新符号重新分流正数线/负数线；变为 0 时立即走 0 元核销——当前 `PENDING_CONFIRM` 结算单置 `VOIDED`、明细软删，剩余参与索赔单置 `SETTLED`，相关抵扣单保持/置 `SETTLED`，不进入对账、开票、付款或质保金流程。上述写入同一事务，任一步失败整体回滚。
**事务**：删/增行与两个汇总字段回写同事务。乐观锁校验 `claim_settle_order.version`。

### OP3b 确认结算

**谁能点**：闪欣索赔专员（服务商结算）/ 欣旺达索赔专员（闪欣结算，P08「确认结算」）。**前置**：`status='PENDING_CONFIRM'`。
**校验**：`payable_amount<>0`；至少 1 行索赔明细。`settle_type='SHANXIN'` 时所有未删除明细 `fee_owner` 必须非空且在字典内；任一行缺失则拦截确认。
**写入**：`claim_settle_order.status: PENDING_CONFIRM → PENDING_RECON`（乐观锁）。无其他表写入。**触发下游**：推送对账给收款方（服务商结算=服务商；闪欣结算=闪欣），本期站内消息/列表可见，无外呼。

### OP4 作废

**谁能点**：闪欣索赔专员（服务商结算）/ 欣旺达索赔专员（闪欣结算）。**前置**：`status='PENDING_CONFIRM'`（仅待确认可作废，终态）。
**写入**（同事务）：
- `claim_settle_order.status: PENDING_CONFIRM → VOIDED`。
- 删该单全部 `claim_settle_detail`（软删 `deleted_at=NOW()`）。
- 回滚抵扣单：该单明细中 `deduct_amount<>0` 对应的 `claim_deduct_order.status: SETTLED → UNSETTLED`。

**失败回滚**：三步同事务。

### OP5 对账确认（正负分流）

**谁能点**：服务商结算=服务商售后专员；闪欣结算=闪欣索赔专员（收款方对账，P06/P08「对账确认」）。**前置**：`status='PENDING_RECON'`。
**公共写入**：`recon_confirm_time=NOW()`。
**分支·应付>0（正数线）**：
- `settle_type='PROVIDER'`：使用生单时已写入的 `invoice_permission_flag` 快照【A-16/A-32】；快照生成规则为 `invoice_whitelist.provider_id=:providerId AND enabled=1 AND deleted_at IS NULL AND COALESCE(effective_from,'1000-01-01') <= DATE(claim_settle_order.created_at)` 则 1，否则 0。权限作为新单快照，后续取消不回改已生成单。快照为 0 时挂起，为 1 时可开票；两者均 `status: PENDING_RECON → PENDING_INVOICE`，挂起恢复见 OP6。
- `settle_type='SHANXIN'`：PRD 的白名单仅为服务商开票权限配置，不套用闪欣主体；直接置 `invoice_permission_flag=1`，`status: PENDING_RECON → PENDING_INVOICE`，由闪欣开票。
**分支·应付<0（负数线，仅服务商结算）**【A-33】：
- 关闭开票权限：`invoice_permission_flag=0`；`force_settle_flag=1`；`status: PENDING_RECON → PENDING_FORCE`。
**拦截**：闪欣结算恒正数，不进负数分支；若应付=0（编辑所致）拦截并提示先处理明细。

### OP6 白名单恢复开票（挂起恢复）

**谁能点**：索赔业务管理员配置白名单后，闪欣索赔专员对挂起的服务商结算单重新校验（或系统在白名单新增时批量重扫）。**前置**：`settle_type='PROVIDER' AND status='PENDING_INVOICE' AND invoice_permission_flag=0`；闪欣结算不进入本操作。
**校验**：该主体当前 `invoice_whitelist.enabled=1 AND deleted_at IS NULL AND COALESCE(effective_from,'1000-01-01') <= CURRENT_DATE`。这是 A-32 对“新增白名单后恢复挂起单”的显式例外；`effective_from` 在未到日期前不得提前解挂。
**写入**：`invoice_permission_flag: 0 → 1`（`status` 不变，仍 `PENDING_INVOICE`，此后可开票）。这是 A-16「配置不回溯」对已挂起单的显式例外——A-32 规定加入白名单后可继续。

### OP7 开票上传发票

**谁能点**：开票方——服务商结算=服务商；闪欣结算=闪欣（P06/P08「开票」）。**前置**：`status='PENDING_INVOICE' AND invoice_permission_flag=1`。
**校验**：发票代码/号码非空；发票附件必传（`attachment` 表 `biz_type='SETTLE'`、`biz_id`=结算单 id，返回 `attachment.id`）；`invoice_amount` 必须等于 `payable_amount`，不等则阻断提交，不允许差额（D-结算-B3 已确认）。
**写入**（同事务）：`claim_settle_order`：`invoice_code`=录入；`invoice_no`=录入；`invoice_date`=录入；`invoice_amount`=录入；`invoice_attachment_id`=附件 id；`status: PENDING_INVOICE → PENDING_VERIFY`。

### OP8 发票确认

**谁能点**：核验方——服务商结算=闪欣索赔专员；闪欣结算=欣旺达索赔专员（发票确认）。**前置**：`status='PENDING_VERIFY'`。

### OP9 申请付款（仅服务商结算，触发付款链）

**谁能点**：闪欣索赔专员（P05/P06 已核验「申请付款」）【A-28】。**前置**：`settle_type='PROVIDER' AND status='VERIFIED'`。
**校验**：该结算单尚无有效 `payment_apply`（一结算单仅 1 张有效付款申请，付款链约束）。
**写入**：结算单状态**保持 VERIFIED**（不变），仅**触发下游付款链**——由付款链 LLD 基于本结算单生成 `payment_apply` 草稿（带出 `settle_id`/`settle_no`/`payee_name`/`fee_owner`/`apply_amount=payable_amount`/发票信息）。本链不写 `payment_apply`（跨链边界）。
**说明**：付款审批与线下付款在付款链内完成。

### OP10 付款完成回写（仅服务商结算，VERIFIED → SETTLED）

- `claim_settle_order`：`status: VERIFIED → SETTLED`；`settle_time=NOW()`。
- 该单明细中每张索赔单 `claim_order.status: PENDING_SETTLE → SETTLED`（`WHERE id=? AND status='PENDING_SETTLE' AND version=?`）。
- 抵扣单已在生成时为 `SETTLED`，无需再动。
**失败回滚**：结算单末态与索赔单末态同事务，任一失败整体回滚（防「结算已完但索赔单未销」）。

### OP0 质保金账户模型与建账 / 入账（INIT / RECHARGE）· 见第九节待确认登记

> 背景：OP11 步骤 B 读账户 `balance` 执行强扣，PRD S8.9/S10/A-31 明确「本模块新建服务商质保金账户实体（余额/流水/归属）」，但**未定义账户何时建、初始余额从哪来、是否有充值入口**。本节忠实落 schema 可派生的账户模型与流水字段口径；建账触发时机、入账/充值来源属产品未决项，**不臆造**，列第九节待确认，产品拍板前本期不实现建账/充值/冻结的写入路径。

**账户模型（`warranty_deposit_account`，schema 真列）**：一服务商一账户（`uk_provider` 唯一）；`balance`=质保金余额、`frozen_amount`=冻结金额、`status ∈ {NORMAL,FROZEN}`。OP11 强扣以「该 `provider_id` 账户已存在」为前置，账户不存在报 `SET_4009`，**不隐式建账**（避免臆造入账来源）。

**建账初值（一旦建账，按 schema 默认忠实取值）**：`balance=0.00`、`frozen_amount=0.00`（NOT NULL 无默认，建账须显式给 0）、`status='NORMAL'`、`version=0`；审计列按通用约定。`frozen_amount` 本期恒 0——无冻结/解冻业务路径，OP11 扣减不读/不写 `frozen_amount`（冻结逻辑见第九节待确认，本期不落）。

**INIT / RECHARGE 流水字段口径（`warranty_deposit_txn`，schema 真列；建账/充值动作本身待确认，此处仅固定字段映射避免下游落库无据）**：
- `INIT`（初始入账，随建账写一条）：`biz_type='INIT'`；`account_id`=账户 id；`provider_id`=服务商；`biz_no=NULL`（非结算触发）；`amount`=首次入账额；`balance_after`=入账后余额；`shortfall_carryover=0.00`；`voucher_attachment_id`/`remark`=可空。
- `RECHARGE`（充值入账）：字段同 INIT，`biz_type='RECHARGE'`。
- 说明：本链只主动写 `DEDUCT`（OP11）。`INIT`/`RECHARGE` 的**触发入口与发起方 PRD 未定**（接口文档 4.7 已标「非本链写入」），字段口径先行固定，实际写入待产品定义建账/充值功能点后落地。

### OP11 强制结算 + 质保金扣减（仅服务商结算负数，两步）

**步骤 A · 索赔专员发起**：闪欣索赔专员在 P05/P06 点「强制结算」，前置为 `settle_type='PROVIDER' AND status='PENDING_FORCE' AND payable_amount<0 AND force_settle_flag=1`【A-31/A-33】。必须上传扣减依据/凭证附件（`attachment.biz_type='SETTLE', biz_id=结算单 id`）；状态保持 `PENDING_FORCE`，不扣质保金、不写扣减流水，等待财务二次确认。

**步骤 B · 财务确认并执行**：财务人员核验扣减依据后确认执行；前置同上且扣减依据附件存在。只有本步骤执行以下扣减与结算完结（D-结算-B4 已确认）。
**账户前置**：该 `provider_id` 的 `warranty_deposit_account` 须已建账存在（建账/入账见 OP0 及第九节待确认），不存在则报 `SET_4009`、不隐式建账。
**分布式锁**：对 `warranty_deposit_account` 按 `account_id`（或 `provider_id`）加锁，防并发扣减。
**计算**：`扣需 = |payable_amount|`；读账户 `balance`；`实扣 = MIN(扣需, balance)`；`shortfall = 扣需 − 实扣`（≥0，余额不足的差额挂账结转）。`frozen_amount` 本期恒 0、不参与扣减计算（无冻结占用）。
**账户 status 分支**：本期账户恒 `NORMAL`，无置 `FROZEN` 的迁移路径；「账户处于 `FROZEN` 时本步强扣是否放行/拦截」PRD 未定，列第九节待确认，本期不实现放行/拦截分支（不臆造资金动作）。
**写入**（**同事务**）：
- 写 `warranty_deposit_txn`：`id`=雪花；`account_id`=账户 id；`provider_id`=服务商；`biz_type='DEDUCT'`；`biz_no`=结算单号 `settle_no`；`amount = -实扣`（负）；`balance_after = balance − 实扣`（余额充足=`balance−扣需`；余额不足=0）；`shortfall_carryover = shortfall`（充足时=0.00）；`voucher_attachment_id`=扣减凭证附件 id（`attachment.biz_type='SETTLE'`）；`remark`=可空。
- 回写账户 `warranty_deposit_account`：`balance = balance − 实扣`（乐观锁 `WHERE id=? AND version=?`）；`status` 不变（NORMAL）。
- `claim_settle_order`：`status: PENDING_FORCE → SETTLED`；`settle_time=NOW()`；`update_by`=财务确认人。
- 该单明细中每张 `claim_order.status: PENDING_SETTLE → SETTLED`。
**失败回滚**：账户扣减、流水、结算单末态、索赔单末态**必须同一事务**，任一步失败整体回滚——这是本链资金一致性的硬约束。
**边界**：`amount=-实扣`（余额充足即 `-|payable|`；不足即 `-balance`）；差额永远记 `shortfall_carryover`，本期不生成二次追扣单（挂账结转，V2 追）。

### OP12 闪欣结算付款（财务共享 Mock，发起+人工回填两步）

**步骤 A·发起**：欣旺达索赔专员在 P07/P08 点“申请付款”；前置 `settle_type='SHANXIN' AND status='VERIFIED'`。系统把 `settle_no/payable_amount/currency` 及 `fee_owner` 分账清单展示给财务人员办理，本期不调真实财务共享接口，不伪造同步成功回执，也不改业务状态（仍 `VERIFIED`）。可重复查看/发起，不生成 `payment_apply`。

**步骤 B·人工回填成功**：财务人员核对线下付款完成后，在内建 Mock 回填页点“付款成功”；前置同上，可选上传付款凭证到 `attachment(biz_type='SETTLE',biz_id=结算单 id)`。同一事务写 `claim_settle_order.status: VERIFIED → SETTLED`、`settle_time=NOW()`、`update_by=回填财务人员`，并把明细中每张 `xw_claim_order.status: PENDING_SETTLE → SETTLED`。重复成功回填因前置状态不符而幂等返回“已处理”。

**失败处理**：线下付款未完成或核对失败时不点成功回填，结算单停留 `VERIFIED`，整改后重试。回填事务任一更新失败整体回滚。

## 六、编码规则

结算单号 `settle_no = <前缀>-yyyyMMdd-4位日内流水`，全局唯一。

- 服务商结算：前缀 `SPJS`，示例 `SPJS-20260701-0012`。
- 闪欣结算：前缀 `SXJS`，示例 `SXJS-20260701-0005`。
- `yyyyMMdd` = 结算单生成日期（定时=gen_day 当天；手动=操作当天）。
- 4 位流水 = 该前缀该日内自增序列，按已确认的 `doc_no_sequence` 同事务行锁方案取号，唯一索引 `uk_settle_no` 兜底；序列表 DDL 待研发组长审定。
- 质保金流水、抵扣单不由本链编号（抵扣单号 `DKD-…` 由责任判定链生成，本链只读引用）。

## 七、导入 / 导出规格

- **导出**（P05/P07 列表「⬇ 导出」）：按当前筛选条件导出结算单列表。导出列与来源逐列对齐第二节列表映射（结算单号/收款方或对方主体/区域/周期/应付总金额/责任抵扣/状态/对账确认时间/结算时间），大数据量异步导出+进度（S12）。明细导出（P06/P08）逐行导出 `claim_settle_detail` 全列；迟到滚入项另带源单原归属日期（服务商/闪欣索赔取原审核通过记录 `audit_time`，抵扣取 `settle_period_belong`），不改写源值。
- **导入**：本链**无导入**（结算单由生成/编辑产生，不支持外部导入）。

## 八、并发 / 幂等 / 回滚

- **乐观锁**：`claim_settle_order`、`warranty_deposit_account`、`claim_order`、`claim_deduct_order` 所有更新带 `version`（`WHERE …AND version=?`，`version=version+1`），冲突返回「数据已变更请刷新」（S10.2/S12）。
- **分布式锁**：质保金扣减（OP11）按 `account_id` 加锁，串行化同账户扣减。
- **生成幂等**：OP1 幂等键 = `settle_type+provider_id+period_start+period_end`；同键待确认单合并本执行周期新到的未占用明细（含 D-结算-B6 迟到滚入项），不重复生单；同键单已进入对账及后续状态时不回改，迟到项留给再下一次可执行周期。
- **同事务边界（硬约束）**：
  - OP1 生成：结算单 + 全部明细 + 抵扣单回写 同事务；0 元核销：索赔单 + 抵扣单末态 同事务。
  - OP4 作废：结算单末态 + 明细删除 + 抵扣单回滚 同事务。
  - OP10 正数完结：结算单末态 + 索赔单末态 同事务。
- **OP11 财务确认执行：质保金账户扣减 + 流水 + 结算单末态 + 索赔单末态 同事务**（索赔专员发起只上传依据、不扣款；D-结算-B4）。
  - OP12 闪欣完结：仅财务人工回填成功后，结算单末态 + 闪欣索赔单末态同事务。
- **失败回滚**：任一事务内步骤失败整体回滚，不产生半完成状态（如「质保金已扣但结算单未 SETTLED」或「结算已 SETTLED 但索赔单未销」）。
- **软删除**：所有查询附 `deleted_at IS NULL`。
