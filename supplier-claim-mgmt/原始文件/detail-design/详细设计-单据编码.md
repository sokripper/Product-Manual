## 编码总览

统一格式：`<前缀>-yyyyMMdd-<4位日内流水>`，全局唯一。yyyyMMdd 取生成时点服务器当日（本期仅 CNY、单时区，无跨时区问题）；4 位流水按「前缀 + 日期」二维在日内从 0001 递增、零填充；跨天由日期段天然区隔，流水重置从 0001 起。

| 单据 | 前缀 | 落库列 | 唯一索引 | 生成时机 | 关联功能 |
|---|---|---|---|---|---|
| 服务商索赔单 | SPCO | claim_order.claim_no | uk_claim_no | 新建索赔草稿保存时 | F-CLAIM-01 |
| 闪欣向欣旺达索赔单 | SXCO | xw_claim_order.claim_no | UNIQUE | 业务审核通过（欣旺达件）逐单自动生成时 | F-XW-01 / F-CLAIM-07 |
| 服务商结算单 | SPJS | claim_settle_order.settle_no | uk_settle_no | 服务商结算单生成时（定时或手动） | F-SET-01 |
| 闪欣结算单 | SXJS | claim_settle_order.settle_no | uk_settle_no | 闪欣结算单生成时 | F-SET-06 |
| 付款申请单 | JSFK | payment_apply.apply_no | uk_apply_no | 闪欣索赔员在已核验结算单点「申请付款」、保存草稿时 | F-PAY-01 |
| 索赔抵扣单 | DKD | claim_deduct_order.deduct_no | UNIQUE | 责任判定=服务商责任自动生成抵扣单时 | F-CLAIM-08 |

> SPJS 与 SXJS 均落 `claim_settle_order.settle_no`（同表按 settle_type 区分前缀）；两者流水各自独立（前缀不同即不同号段）。

## 生成时机（逐单据）

- SPCO：`F-CLAIM-01 新建索赔`保存草稿、`INSERT claim_order`（status=DRAFT）时取号写入 claim_no。同一操作内一次性生成，草稿即占号。
- SXCO：`F-CLAIM-07 业务审核`通过且主责件=欣旺达件时，逐单自动 `INSERT xw_claim_order`（status=DRAFT）时取号写入 claim_no。
- SPJS：`F-SET-01 自动生成结算单`（默认每月 1 日 0 点定时，或手动提前结算）按服务商维度汇总生成 `claim_settle_order`（settle_type=PROVIDER）时取号；应付=0 不生成、不占号（A-34）。
- SXJS：`F-SET-06 闪欣结算`汇总生成 `claim_settle_order`（settle_type=SHANXIN）时取号。
- JSFK：`F-PAY-01`由闪欣索赔员在已核验（VERIFIED）服务商结算单上点「申请付款」、进入新建页保存草稿的同一 `INSERT payment_apply` 内取号（人工触发、系统带出信息，非结算 VERIFIED 系统自动；见 `详细设计-付款申请.md` 操作 1）。
- DKD：`F-CLAIM-08 责任判定`结果=服务商责任时，自动 `INSERT claim_deduct_order` 时取号写入 deduct_no。

各单据编码在其「首次落库 INSERT」的同一逻辑里生成，不预分配、不预留。

## 流水维度与并发唯一方案（设计选型）

**并发唯一采用「序列表 + 同事务行锁」方案**（D-编码-D1 已确认）：本期 MySQL 8 单库、无强制引入 Redis，序列表方案零外部依赖、可落地、天然与业务事务一致；Redis INCR 本期不采用。

新增序列表（本表由 SDD 产出建表 SQL 交付物 `docs/db-schema/ddl/`，标「待组长审定」；未在既有表结构中登记，此处按新建表设计）：

`doc_no_sequence`（单据流水序列表）

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | VARCHAR(32) | PRIMARY KEY | 雪花ID |
| prefix | VARCHAR(8) | UNIQUE(uk_prefix_date) 组合 | 单据前缀（SPCO/SXCO/SPJS/SXJS/JSFK/DKD） |
| seq_date | CHAR(8) | UNIQUE(uk_prefix_date) 组合 | 日期段 yyyyMMdd |
| current_seq | INT | NOT NULL DEFAULT 0 | 当日已发出的最大流水 |

（含通用审计列 create_by/created_at/update_by/updated_at/deleted_at/version）。唯一键 `uk_prefix_date(prefix, seq_date)` 保证每前缀每日仅一行序列。

取号流程（与业务单据 INSERT 同一数据库事务内）：
1. `INSERT INTO doc_no_sequence(id,prefix,seq_date,current_seq) VALUES(?,?,?,1) ON DUPLICATE KEY UPDATE current_seq=current_seq+1` —— 首次当日该前缀建行即发出 `current_seq=1`，之后原子自增；InnoDB 对命中的唯一键行加行锁，并发取号串行化到（前缀+日期）行级，互不阻塞其它前缀 / 其它日期。
2. `SELECT current_seq FROM doc_no_sequence WHERE prefix=? AND seq_date=?` 读回自增后的值（首行为 1）。
3. 拼装 `编号 = 前缀 + '-' + seq_date + '-' + LPAD(current_seq,4,'0')`。
4. 在同一事务内 `INSERT` 业务单据，编号列写入该值；业务表唯一索引（uk_claim_no / uk_settle_no / uk_apply_no / UNIQUE 等）作为最终兜底，理论冲突（例如手工插数据）触发唯一约束异常并回滚重试。

并发唯一保证：同一（前缀+日期）的取号被序列行的行锁串行化，`current_seq` 单调递增不重复；不同前缀 / 不同日期落不同行、不互锁。业务表唯一索引兜底任何绕过序列表的写入。

## 跨天重置

流水按（前缀 + seq_date）二维隔离，`seq_date` 取当日 yyyyMMdd。零点跨天后，新日期在 `doc_no_sequence` 中不存在对应行，取号走 `INSERT ... ON DUPLICATE KEY UPDATE` 的 INSERT 分支建新行、current_seq 从 1 起，天然重置；旧日期行保留（供审计 / 幂等参照），不清理。无需定时清零任务。

---

## 回滚与占号

**采用「提交成功才占号、回滚退号、无跳号」策略**（D-编码-D2 已确认）：取号（序列表自增）与业务单据 INSERT 处于同一数据库事务。同一（前缀+日期）序列行在事务内被行锁持有，并发取号严格串行——后一取号必须等前一事务提交或回滚后才能拿到锁。因此：
- 业务事务提交成功 → 序列自增生效、编号落库，占号成立。
- 业务事务回滚 → 序列自增随事务一并回滚，`current_seq` 恢复，该号未消耗、可被下一取号复用，不产生空洞（无跳号）。
- 行锁串行化保证并发下不会两笔事务拿到同一 `current_seq`，回滚退号也不会造成重复。

不采用 Redis INCR 的原因：Redis INCR 在业务库事务之外自增，业务回滚时号已消耗不回收，会「占号不回收、产生跳号」，且需额外维护 Redis 与跨天 key 过期，本期无强需求，故不引入（备选，标设计选型）。

## 边界与告警

- 日内流水上限：4 位 = 单前缀单日最多 9999 张。超过时 `ON DUPLICATE KEY UPDATE` 自增至 10000，LPAD 会超 4 位——`current_seq>9999` 时取号抛业务异常并记告警日志，拒绝生成（D-编码-B2 已确认；本期各前缀单日业务量远不及此阈值）。如需扩容为 5 位须走变更登记，同步改格式与列长评估（各编号列现为 VARCHAR(32)，容量充足）。
- 时区 / 币种：本期单时区、仅 CNY，yyyyMMdd 取服务器本地日期，无跨时区歧义（美元禁用 A-29）。
