## 模块范围（scope）

本付款申请链**只覆盖闪欣付服务商侧的付款**（`settle_type=PROVIDER` 服务商结算），走本系统内建分级审批链（A-17）。**欣旺达→闪欣结算（`settle_type=SHANXIN`）的付款不进本模块**：按 S6.5/A-18 走财务共享系统 Mock（欣旺达发起后线下办理、财务人工回填成功），不生成本系统 `payment_apply`、不走内建审批链。故本 LLD 全部操作只针对服务商结算派生的付款申请，`fee_owner` 不含欣旺达侧审批分支。

## 业务流转总览

付款申请链是正数服务商结算完成后的资金出口。上游服务商结算单在闪欣发票确认通过、迁入 `claim_settle_order.status=VERIFIED`（已核验，由结算 LLD 归口）后，**结算单停在已核验态，不自动生成付款草稿**；由闪欣索赔员在该已核验结算单上人工点「申请付款」（原型 06 结算详情按钮 / F-SET-04 / S6.3 / S17.2），系统随即按该结算单带出收款方、费用归属、应付金额、结算单号，生成一张 `payment_apply` 草稿（`PENDING_SUBMIT` 待提交）——「系统带出信息自动填充草稿」（A-23）与「人工触发」（A-28）并存，触发是人工、带出是系统。银行开户行 / 账号 / 账户名 / 付款内容为空，待索赔员在草稿上人工补录。负数结算不进本链（走待强制结算 + 质保金扣减），故本链只处理 `apply_amount>0` 的付款。

索赔专员补齐付款信息后提交：系统按 `apply_amount` 到 `payment_approval_flow_config` 匹配金额档 `PAY_GRADE`（<5万 L1 / 5万≤x<20万 L2 / ≥20万 L3，A-24，档位与链路可在结算基础配置维护），取该档有序节点链 `node_chain`，逐节点生成 `payment_approval_node` 流水，单据迁入 `APPROVING` 审批中，当前节点指向链首。审批人沿链逐节点处理：通过则前进到下一节点，末节点通过则单据迁入 `APPROVED`；任一节点驳回时单据直接回到 `PENDING_SUBMIT`，不以 `REJECTED` 作为独立驻留态，修改后重新提交并重建节点链（D-付款-D1 已确认）。

审批通过后由财务线下付款、回录付款凭证，单据迁入 `PAID` 已付款，并触发关联结算单迁入 `SETTLED` 已结算（其名下索赔单置已结算归结算 LLD）。草稿态可作废，`VOIDED` 为终态。全链以 `version` 乐观锁 + 单据状态双条件更新防并发，一结算单全程仅允许 1 张非作废付款申请。

表间数据流转顺序：`claim_settle_order`（VERIFIED）→ 人工点「申请付款」生成 `payment_apply`（PENDING_SUBMIT，系统带出信息）→ 人工补录并提交 → 匹配 `payment_approval_flow_config` → 批量生成 `payment_approval_node` → 逐节点回写节点流水并推进 `payment_apply.current_node/status` → 付款回写 `payment_apply` + `attachment` + 回写 `claim_settle_order`。

## 数据来源映射（人工申请付款生成草稿时的字段带入）

| payment_apply 列 | 取值来源 | 说明 |
|---|---|---|
| settle_id | claim_settle_order.id | 触发源结算单 |
| settle_no | claim_settle_order.settle_no | 只读带出 |
| payee_name | claim_settle_order.payee_name | 收款方全称，只读带出 |

| currency | claim_settle_order.currency | 恒 CNY（A-29） |
| invoice_info_id | NULL（本期 Mock/待文档） | 无独立发票实体；本期直接校验结算单 `invoice_*` 字段与发票附件，正式发票实体文档到位后回填（D-付款-B3） |

## 状态机（payment_apply，本期 5 个有效驻留态）

本期有效驻留态：PENDING_SUBMIT 待提交 / APPROVING 审批中 / APPROVED 审批通过 / PAID 已付款 / VOIDED 已作废。表结构登记仍保留 `REJECTED` 兼容码，但 D-付款-D1 已确认本期不写入该驻留态；审批驳回直接回 `PENDING_SUBMIT`。

合法迁移与触发：
- PENDING_SUBMIT →（提交，校验通过 + 匹配档位 + 生成节点）→ APPROVING
- PENDING_SUBMIT →（作废）→ VOIDED（终）
- APPROVING →（非末节点通过）→ APPROVING（推进 current_node）
- APPROVING →（末节点通过）→ APPROVED
- APPROVING →（任一节点驳回）→ PENDING_SUBMIT（退回待提交，D-付款-D1）
- PENDING_SUBMIT →（修改后重新提交，重新匹配 + 重建节点）→ APPROVING
- APPROVED →（线下付款 + 录凭证）→ PAID（终，触发结算单 SETTLED）

非法迁移拦截（Service 层，乐观锁双条件 `WHERE id=? AND status=? AND version=?`）：
- 非 PENDING_SUBMIT 不得作废、编辑或提交；非 APPROVING 不得审批；非当前节点 / 已处理节点不得审批；非 APPROVED 不得付款；PAID/VOIDED 为终态不接受任何写操作。

## 操作规格（每项自包含）

### 操作 1：人工申请付款生成草稿（F-PAY-01）

- 谁能点：闪欣索赔员，在已核验的服务商结算单上点「申请付款」（原型 06 结算详情按钮 / F-SET-04 / S6.3 / S17.2）——触发是人工，非结算状态迁移的自动副作用。
- 前置：源结算单 settle_type=PROVIDER、status=VERIFIED 且 payable_amount>0；该 settle_id 下不存在 status≠VOIDED 的 payment_apply（一结算单仅一张有效付款申请；生成前查重，命中则跳转到既有单据、不重复生成）。结算单可停在已核验而无付款草稿，直到有人点「申请付款」。
- 校验：settle_type=PROVIDER；payable_amount>0（=0 或负数不进本链）；无有效付款申请。
- 写入 `payment_apply`（一行 INSERT，系统按 A-23 带出结算收款信息填充草稿）：
  - id = 雪花ID
  - apply_no = JSFK 编码（编码规则见 `详细设计-单据编码.md`，生成时机=本操作保存草稿的同一 INSERT 内取号）
  - settle_id = claim_settle_order.id
  - settle_no = claim_settle_order.settle_no
  - payee_name = claim_settle_order.payee_name
  - fee_owner = 'SHANXIN'（本模块只覆盖服务商结算，D-付款-B2 已确认）
  - apply_amount = claim_settle_order.payable_amount
  - currency = claim_settle_order.currency
  - invoice_info_id = NULL（本期 Mock/待文档；不伪造发票实体 id）
  - has_invoice = 1
  - payee_bank = NULL、payee_account = NULL、payee_account_name = NULL、pay_content = NULL
  - grade_level = NULL、current_node = NULL
  - applicant = NULL、apply_dept = NULL、apply_time = NULL（提交时写）
  - status = 'PENDING_SUBMIT'
  - pay_finish_time = NULL、voucher_attachment_id = NULL
  - create_by = 当前操作人（闪欣索赔员）、created_at = NOW()、version = 0、deleted_at = NULL
- 新状态：PENDING_SUBMIT。
- 触发下游：无（等待人工补录付款信息、提交）。
- 事务/回滚：本操作为独立事务（生成草稿即取号入库，与结算单状态无耦合，结算单不因此改状态）；查重 + apply_no 取号（见编码 LLD）+ payment_apply INSERT 同事务，失败整体回滚。

### 操作 2：编辑付款信息（F-PAY-01）

- 谁能点：索赔专员（本单据可编辑角色）。
- 前置：status = PENDING_SUBMIT。
- 校验：银行账号 payee_account 仅数字（`^[0-9]+$`，≤64）；开户行 payee_bank ≤100；账户名 payee_account_name ≤128 且与 payee_name 比对，不一致时弹风险提示需二次确认（不阻断，原型 11）；付款内容 pay_content ≤300。基础信息（settle 带出的 settle_no/payee_name/fee_owner/apply_amount/currency）只读，不接受修改。
- 写入 `payment_apply`：payee_bank=录入值（空写 NULL）、payee_account=录入值、payee_account_name=录入值、pay_content=录入值、update_by=当前用户、updated_at=NOW()、version=version+1；status 不变。
- 附件（可选）：写入 `attachment`（biz_type='PAYMENT'、biz_id=payment_apply.id、file_path=服务器相对路径、upload_by、upload_time=NOW()）。
- 新状态：不变。
- 事务/回滚：payment_apply 更新 + 附件写入同事务；乐观锁冲突提示刷新。

### 操作 3：提交 / 分级审批发起（F-PAY-02）

- 谁能点：索赔专员。
- 前置：status = PENDING_SUBMIT。
- 校验（全量）：payee_bank、payee_account（仅数字）、payee_account_name、pay_content 均非空；账户名与 payee_name 一致性（不一致需已二次确认）；apply_amount>0；`has_invoice=1` 时校验关联结算单的 `invoice_code/invoice_no/invoice_date/invoice_amount/invoice_attachment_id` 齐全。本期不以 `invoice_info_id` 非空作为校验条件。
- 匹配档位：`SELECT grade_level, node_chain FROM payment_approval_flow_config WHERE enabled=1 AND amount_min<=apply_amount AND (amount_max IS NULL OR apply_amount<amount_max)`（边界左闭右开，=5万落 L2，A-24）；匹配不到档位则拦截报错「未配置金额档」。
- 写入 `payment_apply`：grade_level=匹配档、status='APPROVING'、current_node=node_chain[0] 节点名、applicant=当前用户、apply_dept=当前用户部门、apply_time=NOW()、update_by=当前用户、version=version+1。
- 写入 `payment_approval_node`（按 node_chain 顺序逐条 INSERT，n 条）：
  - id=雪花ID、payment_id=payment_apply.id
  - node_seq = 1..n（元素序号）
  - node_name = node_chain[i] 节点名、approver_role = node_chain[i] 角色
  - approver = NULL、approve_time = NULL、approve_result = 'PENDING'、opinion = NULL、attachment_ids = NULL
  - create_by=当前用户、created_at=NOW()、version=0
- 重提场景（付款单已存在被驳回节点且当前已回 `PENDING_SUBMIT`）：本期 Mock 约定先将上一轮 `payment_approval_node.deleted_at=NOW()` 后重建；schema 无轮次列，正式历史保留方案待研发文档 D-付款-B1 回填。
- 新状态：APPROVING。
- 触发下游：无。
- 事务/回滚：payment_apply 更新 + 全部节点 INSERT（+ 旧节点软删）同一事务，任一失败整体回滚。

### 操作 4：逐节点审批通过（F-PAY-03）

- 谁能点：当前节点 approver_role 对应审批人。
- 前置：payment_apply.status=APPROVING；目标节点 approve_result=PENDING 且 node_name=payment_apply.current_node（仅当前节点可审）。
- 校验：操作人角色匹配该节点 approver_role。
- 写入 `payment_approval_node`（当前节点）：approve_result='PASS'、approver=当前用户、approve_time=NOW()、opinion=录入意见（空写 NULL）、attachment_ids=附件列表（空写 NULL）、version=version+1。
- 写入 `payment_apply`：
  - 存在下一节点（node_seq+1）：current_node=下一节点 node_name、version=version+1，status 仍 APPROVING。
  - 当前为末节点：status='APPROVED'、current_node=NULL、version=version+1。
- 新状态：APPROVING（非末）或 APPROVED（末）。
- 触发下游：无（付款为独立操作）。
- 事务/回滚：节点回写 + payment_apply 推进同事务。

### 操作 5：节点审批驳回（F-PAY-03）

- 谁能点：当前节点审批人。
- 前置：payment_apply.status=APPROVING；目标节点为当前节点、approve_result=PENDING。
- 校验：驳回意见 opinion 建议必填（沿用审批意见列）。
- 写入 `payment_approval_node`（当前节点）：approve_result='REJECT'、approver=当前用户、approve_time=NOW()、opinion=驳回原因、version=version+1；后续未处理 PENDING 节点保留至重新提交时统一按本期 Mock 约定软删。
- 写入 `payment_apply`：status='PENDING_SUBMIT'、current_node=NULL、update_by=当前用户、version=version+1。
- 新状态：PENDING_SUBMIT（退回待提交，不驻留 REJECTED；D-付款-D1）。
- 触发下游：无。
- 事务/回滚：节点回写 + payment_apply 迁移同事务。重提走操作 3。

### 操作 6：付款（录凭证）（F-PAY-04）

- 谁能点：财务 / 出纳（付款节点角色）。
- 前置：status=APPROVED。
- 校验：付款凭证附件必传。
- 写入 `payment_apply`：status='PAID'、voucher_attachment_id=凭证附件id、pay_finish_time=NOW()、update_by=当前用户、version=version+1。
- 写入 `attachment`：biz_type='PAYMENT'、biz_id=payment_apply.id、file_name/file_path/file_type/file_size、upload_by=当前用户、upload_time=NOW()。
- 触发下游：同一事务内写 `claim_settle_order`（该 settle_id）：status='SETTLED'、settle_time=NOW()、version=version+1，并把其名下 `claim_settle_detail` 对应的 `claim_order.status` 批量置 `SETTLED`；批量置位由结算 LLD 的 OP10 归口执行（D-付款-B4 已确认）。
- 新状态：PAID（终）。
- 事务/回滚：payment_apply + attachment + claim_settle_order + 名下 claim_order 批量末态同事务，任一失败整体回滚。

### 操作 7：作废（F-PAY-04）

- 谁能点：索赔专员。
- 前置：status=PENDING_SUBMIT（仅草稿可作废，A-25）。
- 校验：无。
- 写入 `payment_apply`：status='VOIDED'、update_by=当前用户、version=version+1。
- 新状态：VOIDED（终）。作废后该 settle_id 可再次人工点「申请付款」生成新草稿（一结算单仅 1 张非 VOIDED 有效单，VOIDED 不计入）。
- 事务/回滚：单行更新。

### 操作 8：查看（F-PAY-05）

- 谁能点：索赔专员 / 审批人（数据范围内）。
- 前置：全状态可查看。
- 读取：`payment_apply` 主档 + `payment_approval_node`（按 node_seq 升序，构成审批时间线）+ `attachment`（biz_type='PAYMENT'）；关联结算单号可跳 `claim_settle_order`。银行账号展示脱敏（原型 10）。
- 写入：无。

> 查询 / 导出 / 批量审核（F-PAY-05）沿用列表页通用规格（多维筛选、按筛选导出、批量部分成功部分失败返回明细），本 LLD 不逐字段展开。

## 并发 / 幂等 / 回滚

- 一结算单一有效单去重：人工点「申请付款」时先对 settle_id 查「存在 status≠VOIDED 的 payment_apply 则跳转既有单、不新建」；配合结算单行锁串行化并发点击，避免重复生成（MySQL 不支持带条件的部分唯一索引，条件唯一由 Service 层 + 结算单单行序列化保证）。
- 乐观锁：所有更新携 `WHERE id=? AND status=? AND version=?`，命中 0 行判为并发冲突，提示刷新重试（PRD S12）。
- 单据编码并发：apply_no 唯一性与取号并发方案见 `详细设计-单据编码.md`；payment_apply.apply_no 有唯一索引 uk_apply_no 兜底。
- 事务边界：提交（payment_apply 更新 + 节点批量插入 [+ 旧节点软删]）、末节点通过（节点 + payment_apply）、付款（payment_apply + attachment + 结算单 + 名下索赔单末态）分别在各自单一数据库事务内完成，任一步失败整体回滚。

## D4 决策与挂起项指针

D-1 已确认驳回直接回 `PENDING_SUBMIT`；B-2 已确认 `fee_owner='SHANXIN'`；B-4 已确认付款完成与结算单/索赔单末态同事务。B-1（审批轮次历史）与 B-3（发票实体）仍待研发文档，本期分别采用“软删旧节点后重建”和 `invoice_info_id=NULL` 的 Mock 约定，详见 `.sdd/traceability/付款申请.md` C 表。
