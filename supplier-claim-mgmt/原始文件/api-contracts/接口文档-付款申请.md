## 2. 接口详情

### 2.1 申请付款生成草稿

- **接口名称**：申请付款生成草稿
- **请求路径**：`/payment/apply/create-from-settle`
- **请求方式**：`POST`
- **描述**：闪欣索赔员在已核验服务商结算单上点「申请付款」，系统按结算单带出信息生成一张付款申请草稿（LLD 操作 1）。

#### 请求参数（Body - JSON）

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `settleId` | String | 是 | 源服务商结算单 id（`claim_settle_order.id`） | `1830000000000000001` |

#### 请求示例

```json
{ "settleId": "1830000000000000001" }
```

#### 业务处理逻辑（写操作）

1. **前置校验**：源结算单 `settle_type=PROVIDER`、`status=VERIFIED`、`payable_amount>0`；该 `settle_id` 下不存在 `status≠VOIDED` 的 `payment_apply`（一结算单仅一张有效付款申请）。命中已有有效单 → 返回既有单 id、不重复生成（`code=409`）；结算类型 / 状态 / 金额不满足 → `code=400`。
2. **落库** `INSERT payment_apply`（一行）：`apply_no`=JSFK 编号（同事务取号，见 conventions S7）；`settle_id`/`settle_no`/`payee_name`/`apply_amount`(=payable_amount)/`currency` 从结算单带出；`fee_owner`=常量 `'SHANXIN'`（D-付款-B2）；`has_invoice`=1；`invoice_info_id`=NULL（本期 Mock，D-付款-B3）；`payee_bank`/`payee_account`/`payee_account_name`/`pay_content`/`grade_level`/`current_node`/`applicant`/`apply_dept`/`apply_time`/`pay_finish_time`/`voucher_attachment_id`=NULL；`status`='PENDING_SUBMIT'；`create_by`=当前操作人、`created_at`=NOW()、`version`=0。
3. **状态流转**：无 → PENDING_SUBMIT。
4. **副作用**：无（结算单不改状态、不耦合）。
5. **事务边界**：查重 + 取号 + `payment_apply` INSERT 同一事务，任一失败整体回滚。

#### 响应参数

| 参数名 | 类型 | 说明 | 示例 |
| :--- | :--- | :--- | :--- |
| `success` | Boolean | 业务是否成功 | `true` |
| `code` | String | 业务码 | `"0"` |
| `message` | String | 提示信息 | `操作成功` |
| `data` | JSON | 业务数据 | |
| ├ `id` | String | 新付款申请 id | `1840000000000000001` |
| ├ `applyNo` | String | 付款申请单号 | `JSFK-20260714-0001` |
| ├ `status` | String | 单据状态 | `PENDING_SUBMIT` |
| `traceId` | String | 链路追踪 ID | `a1b2c3d4e5f6` |

#### 响应示例（成功）

```json
{
  "success": true,
  "code": "0",
  "message": "操作成功",
  "data": { "id": "1840000000000000001", "applyNo": "JSFK-20260714-0001", "status": "PENDING_SUBMIT" },
  "traceId": "a1b2c3d4e5f6"
}
```

### 2.2 编辑付款信息

- **接口名称**：编辑付款信息
- **请求路径**：`/payment/apply/{id}`
- **请求方式**：`PUT`
- **描述**：索赔专员在草稿态补录收款银行信息与付款内容（LLD 操作 2）。

#### 请求参数

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | String | 是 | 付款申请 id（Path） | `1840000000000000001` |
| `payeeBank` | String | 否 | 开户行（≤100，空写 NULL） | `招商银行深圳分行` |
| `payeeAccount` | String | 否 | 银行账号（仅数字 `^[0-9]+$`，≤64） | `6225880000000000` |
| `payeeAccountName` | String | 否 | 账户名（≤128；与 payee_name 不一致时前端二次确认，不阻断） | `深圳某服务商有限公司` |
| `payContent` | String | 否 | 付款内容（≤300） | `2026年6月索赔结算付款` |
| `attachmentIds` | String[] | 否 | 附件 id 列表（可选上传，空写 NULL） | `["1850000000000000001"]` |

> 基础信息（`settleNo`/`payeeName`/`feeOwner`/`applyAmount`/`currency`）只读带出、不接受修改，不在本请求体。

#### 请求示例

```json
{
  "payeeBank": "招商银行深圳分行",
  "payeeAccount": "6225880000000000",
  "payeeAccountName": "深圳某服务商有限公司",
  "payContent": "2026年6月索赔结算付款",
  "attachmentIds": ["1850000000000000001"]
}
```

#### 业务处理逻辑（写操作）

1. **前置校验**：`status=PENDING_SUBMIT`（乐观锁由后端管理：`WHERE id=? AND status=?` status 双条件 + MyBatis-Plus @Version 事务内自读当前版本自增，接口不传 version；命中 0 行 → `code=409` / `PAY_4004` 提示刷新）。
2. **字段校验**：`payeeAccount` 仅数字且 ≤64；`payeeBank`≤100；`payeeAccountName`≤128；`payContent`≤300。不满足 → `code=400`。
3. **落库** `UPDATE payment_apply`：`payee_bank`/`payee_account`/`payee_account_name`/`pay_content`=录入值（空写 NULL）、`update_by`=当前用户、`updated_at`=NOW()、`version`=version+1；`status` 不变。
4. **副作用**：传入 `attachmentIds` 时写入 `attachment`（`biz_type='PAYMENT'`、`biz_id=payment_apply.id`、`file_path`=服务器相对路径、`upload_by`=当前用户、`upload_time`=NOW()）；未传附件时无副作用。
5. **事务边界**：单表更新。

#### 响应参数

| 参数名 | 类型 | 说明 | 示例 |
| :--- | :--- | :--- | :--- |
| `success` | Boolean | 业务是否成功 | `true` |
| `code` | String | 业务码 | `"0"` |
| `message` | String | 提示信息 | `操作成功` |
| `data` | JSON | 业务数据 | |
| ├ `id` | String | 付款申请 id | `1840000000000000001` |
| `traceId` | String | 链路追踪 ID | `a1b2c3d4e5f6` |

#### 响应示例（成功）

```json
{ "success": true, "code": "0", "message": "操作成功", "data": { "id": "1840000000000000001" }, "traceId": "a1b2c3d4e5f6" }
```

### 2.3 提交 / 发起分级审批

- **接口名称**：提交 / 发起分级审批
- **请求路径**：`/payment/apply/{id}/submit`
- **请求方式**：`POST`
- **描述**：索赔专员提交草稿，系统按金额匹配审批档位、生成节点链，单据进入审批中（LLD 操作 3）。

#### 请求参数

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | String | 是 | 付款申请 id（Path） | `1840000000000000001` |

> 无请求体参数（仅路径参数 `id`）。

#### 请求示例

```
POST /payment/apply/1840000000000000001/submit
{}
```

#### 业务处理逻辑（写操作）

1. **前置校验**：`status=PENDING_SUBMIT`（乐观锁由后端管理：`WHERE id=? AND status=?` status 双条件 + MyBatis-Plus @Version 事务内自读当前版本自增，接口不传 version；并发冲突前端靠 status + `code=409`/`PAY_4004` 感知）。
2. **全量校验**：`payee_bank`/`payee_account`(仅数字)/`payee_account_name`/`pay_content` 均非空；账户名与 `payee_name` 一致（不一致需前端已二次确认）；`apply_amount>0`；`has_invoice=1` 时校验关联结算单 `invoice_code`/`invoice_no`/`invoice_date`/`invoice_amount`/`invoice_attachment_id` 齐全（本期不校验 `invoice_info_id`）。不满足 → `code=400`。
3. **匹配档位**：`SELECT grade_level,node_chain FROM payment_approval_flow_config WHERE enabled=1 AND amount_min<=apply_amount AND (amount_max IS NULL OR apply_amount<amount_max)`（左闭右开，=5万落 L2，A-24）。匹配不到 → 拦截报错「未配置金额档」（`code=400`）。
4. **落库** `UPDATE payment_apply`：`grade_level`=匹配档、`status`='APPROVING'、`current_node`=node_chain[0] 节点名、`applicant`=当前用户、`apply_dept`=当前用户部门、`apply_time`=NOW()、`update_by`=当前用户、`version`=version+1。
5. **落库** `INSERT payment_approval_node`（按 node_chain 顺序逐条，n 条）：`payment_id`=payment_apply.id、`node_seq`=1..n、`node_name`/`approver_role`=node_chain[i]、`approve_result`='PENDING'、`approver`/`approve_time`/`opinion`/`attachment_ids`=NULL、`create_by`=当前用户、`created_at`=NOW()、`version`=0。
6. **重提场景**：付款单曾被驳回、当前回到 PENDING_SUBMIT → 先将上一轮 `payment_approval_node.deleted_at`=NOW() 软删，再重建（本期 Mock 约定，正式轮次历史待 D-付款-B1）。
7. **状态流转**：PENDING_SUBMIT → APPROVING。
8. **事务边界**：`payment_apply` 更新 + 全部节点 INSERT（+ 旧节点软删）同一事务，任一失败整体回滚。

#### 响应参数

| 参数名 | 类型 | 说明 | 示例 |
| :--- | :--- | :--- | :--- |
| `success` | Boolean | 业务是否成功 | `true` |
| `code` | String | 业务码 | `"0"` |
| `message` | String | 提示信息 | `提交成功` |
| `data` | JSON | 业务数据 | |
| ├ `id` | String | 付款申请 id | `1840000000000000001` |
| ├ `status` | String | 单据状态 | `APPROVING` |
| ├ `gradeLevel` | String | 匹配到的审批档 | `L2` |
| ├ `currentNode` | String | 当前审批节点名 | `财务主管审批` |
| `traceId` | String | 链路追踪 ID | `a1b2c3d4e5f6` |

#### 响应示例（成功）

```json
{ "success": true, "code": "0", "message": "提交成功", "data": { "id": "1840000000000000001", "status": "APPROVING", "gradeLevel": "L2", "currentNode": "财务主管审批" }, "traceId": "a1b2c3d4e5f6" }
```

### 2.4 节点审批通过

- **接口名称**：节点审批通过
- **请求路径**：`/payment/apply/{id}/approve`
- **请求方式**：`POST`
- **描述**：当前节点审批人通过；非末节点前进到下一节点，末节点通过则单据审批通过（LLD 操作 4）。

#### 请求参数

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | String | 是 | 付款申请 id（Path） | `1840000000000000001` |
| `opinion` | String | 否 | 审批意见（空写 NULL） | `同意付款` |
| `attachmentIds` | String[] | 否 | 附件 id 列表（空写 NULL） | `["1850000000000000001"]` |

#### 请求示例

```json
{ "opinion": "同意付款", "attachmentIds": ["1850000000000000001"] }
```

#### 业务处理逻辑（写操作）

1. **前置校验**：`payment_apply.status=APPROVING`；目标节点为 `current_node` 且其 `approve_result=PENDING`（非当前 / 已处理节点 → `code=409`）。
2. **权限校验**：操作人角色 = 当前节点 `approver_role`，否则 `code=403`。
3. **落库** `UPDATE payment_approval_node`（当前节点）：`approve_result`='PASS'、`approver`=当前用户、`approve_time`=NOW()、`opinion`=录入（空 NULL）、`attachment_ids`=录入（空 NULL）、`version`=version+1。
4. **落库** `UPDATE payment_apply`：存在下一节点（node_seq+1）→ `current_node`=下一节点名、`status` 仍 APPROVING、`version`=version+1；当前为末节点 → `status`='APPROVED'、`current_node`=NULL、`version`=version+1。
5. **状态流转**：APPROVING →（非末）APPROVING /（末）APPROVED。
6. **副作用**：无。
7. **事务边界**：节点回写 + `payment_apply` 推进同一事务。

#### 响应参数

| 参数名 | 类型 | 说明 | 示例 |
| :--- | :--- | :--- | :--- |
| `success` | Boolean | 业务是否成功 | `true` |
| `code` | String | 业务码 | `"0"` |
| `message` | String | 提示信息 | `审批通过` |
| `data` | JSON | 业务数据 | |
| ├ `id` | String | 付款申请 id | `1840000000000000001` |
| ├ `status` | String | 单据状态 | `APPROVING` |
| ├ `currentNode` | String | 当前节点（末节点通过后为 null） | `出纳复核` |
| `traceId` | String | 链路追踪 ID | `a1b2c3d4e5f6` |

#### 响应示例（成功）

```json
{ "success": true, "code": "0", "message": "审批通过", "data": { "id": "1840000000000000001", "status": "APPROVING", "currentNode": "出纳复核" }, "traceId": "a1b2c3d4e5f6" }
```

### 2.5 节点审批驳回

- **接口名称**：节点审批驳回
- **请求路径**：`/payment/apply/{id}/reject`
- **请求方式**：`POST`
- **描述**：当前节点审批人驳回，单据直接退回待提交（不驻留 REJECTED，D-付款-D1）（LLD 操作 5）。

#### 请求参数

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | String | 是 | 付款申请 id（Path） | `1840000000000000001` |
| `opinion` | String | 否 | 驳回原因，建议填写（空写 NULL） | `账号信息有误，请核对后重报` |

#### 请求示例

```json
{ "opinion": "账号信息有误，请核对后重报" }
```

#### 业务处理逻辑（写操作）

1. **前置校验**：`payment_apply.status=APPROVING`；目标节点为当前节点、`approve_result=PENDING`。
2. **权限校验**：操作人角色 = 当前节点 `approver_role`，否则 `code=403`。
3. **落库** `UPDATE payment_approval_node`（当前节点）：`approve_result`='REJECT'、`approver`=当前用户、`approve_time`=NOW()、`opinion`=驳回原因、`version`=version+1；后续 PENDING 节点保留至重提时按 Mock 约定统一软删。
4. **落库** `UPDATE payment_apply`：`status`='PENDING_SUBMIT'、`current_node`=NULL、`update_by`=当前用户、`version`=version+1。
5. **状态流转**：APPROVING → PENDING_SUBMIT（退回待提交；重提走 2.3）。
6. **事务边界**：节点回写 + `payment_apply` 迁移同一事务。

| `message` | String | 提示信息 | `已驳回` |
| `data` | JSON | 业务数据 | |

| ├ `status` | String | 单据状态 | `PENDING_SUBMIT` |
| `traceId` | String | 链路追踪 ID | `a1b2c3d4e5f6` |

#### 响应示例（成功）


### 2.6 付款（录凭证）

- **接口名称**：付款（录凭证）
- **请求路径**：`/payment/apply/{id}/pay`
- **请求方式**：`POST`
- **描述**：财务 / 出纳线下付款后录入凭证，单据置已付款，并触发关联结算单已结算（LLD 操作 6）。

#### 请求参数

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | String | 是 | 付款申请 id（Path） | `1840000000000000001` |
| `voucherAttachmentId` | String | 是 | 付款凭证附件 id（必传） | `1850000000000000009` |

#### 请求示例

```json
{ "voucherAttachmentId": "1850000000000000009" }
```

#### 业务处理逻辑（写操作）

1. **前置校验**：`status=APPROVED`（乐观锁双条件）；付款凭证附件必传，缺失 → `code=400`。
2. **落库** `UPDATE payment_apply`：`status`='PAID'、`voucher_attachment_id`=凭证附件 id、`pay_finish_time`=NOW()、`update_by`=当前用户、`version`=version+1。
3. **落库** `INSERT attachment`：`biz_type`='PAYMENT'、`biz_id`=payment_apply.id、`file_name`/`file_path`/`file_type`/`file_size`、`upload_by`=当前用户、`upload_time`=NOW()。
4. **状态流转**：APPROVED → PAID（终）。
5. **副作用（同事务）**：`UPDATE claim_settle_order`（该 settle_id）`status`='SETTLED'、`settle_time`=NOW()、`version`=version+1；其名下 `claim_order` 批量置 `SETTLED`——批量置位由结算 LLD OP10 归口执行（D-付款-B4）。
6. **事务边界**：`payment_apply` + `attachment` + `claim_settle_order` + 名下 `claim_order` 批量末态同一事务，任一失败整体回滚。

#### 响应参数

| `code` | String | 业务码 | `"0"` |
| `message` | String | 提示信息 | `付款完成` |
| `data` | JSON | 业务数据 | |
| ├ `id` | String | 付款申请 id | `1840000000000000001` |
| ├ `status` | String | 单据状态 | `PAID` |
| ├ `settleStatus` | String | 关联结算单状态 | `SETTLED` |
| `traceId` | String | 链路追踪 ID | `a1b2c3d4e5f6` |

#### 响应示例（成功）


### 2.7 作废

- **接口名称**：作废
- **请求路径**：`/payment/apply/{id}/void`
- **请求方式**：`POST`
- **描述**：索赔专员作废草稿态付款申请（仅草稿可作废，A-25）（LLD 操作 7）。

#### 请求参数

## 3. 错误码（本模块业务专属）

> 通用错误码见 `docs/conventions.md S6`。

| 错误码 | 说明 | 触发场景 |
| :--- | :--- | :--- |
| `PAY_4001` | 结算单不满足申请付款条件 | 非 `settle_type=PROVIDER` / 非 VERIFIED / `payable_amount<=0` |
| `PAY_4002` | 该结算单已存在有效付款申请 | settle_id 下已有 `status≠VOIDED` 的付款申请（返回既有单 id） |
| `PAY_4003` | 未配置金额档 | 提交时 `apply_amount` 匹配不到 `payment_approval_flow_config` 档位 |
| `PAY_4004` | 单据状态不允许该操作 | 乐观锁双条件命中 0 行 / 状态机非法迁移 |
| `PAY_4005` | 非当前节点或已处理节点 | 审批 / 驳回目标非 `current_node` 或该节点非 PENDING |
| `PAY_4006` | 付款凭证缺失 | 付款操作未上传凭证附件 |

## 4. 附录 · 枚举字典

### 4.1 付款申请状态 `payment_apply.status`

| 值 | 含义 |
| :--- | :--- |
| `PENDING_SUBMIT` | 待提交（草稿） |
| `APPROVING` | 审批中 |
| `APPROVED` | 审批通过 |
| `PAID` | 已付款（终态，触发结算单 SETTLED） |
| `VOIDED` | 已作废（终态） |

> `REJECTED` 为 schema 兼容保留码，本期不作为驻留态写入（D-付款-D1）。

### 4.2 节点审批结果 `payment_approval_node.approve_result`

| 值 | 含义 |
| :--- | :--- |
| `PENDING` | 待审批 |
| `PASS` | 通过 |
| `REJECT` | 驳回 |

### 4.3 审批档位 `grade_level`（A-24）

| 值 | 金额区间（左闭右开） |
| :--- | :--- |
| `L1` | `apply_amount < 5万` |
| `L2` | `5万 ≤ apply_amount < 20万` |
| `L3` | `apply_amount ≥ 20万` |

### 4.4 费用归属 `fee_owner`

| 值 | 含义 |
| :--- | :--- |
| `SHANXIN` | 闪欣付服务商（本模块唯一取值，D-付款-B2） |
