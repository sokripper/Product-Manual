# 接口文档 · 结算链（服务商结算 + 闪欣结算）

> 通用说明（认证 / 基础 URL / `Result<T>` / `PageResult<T>` / 审计列 / 通用错误码 / 单据编号规则）见 `docs/conventions.md`，本文件不重复。
> 上游权威：`详细设计-结算.md`（OP1~OP12，含 OP3a/OP3b 两操作、OP11/OP12 两步）、`docs/db-schema/表结构文档.md`（claim_settle_order / claim_settle_detail / warranty_deposit_account / warranty_deposit_txn / invoice_whitelist / settle_period_config / claim_deduct_order / attachment）。每接口 = LLD 一个真实操作，每字段 = 已登记真列或 PRD 字段。
> 范围：服务商结算（`settle_type=PROVIDER`，P05/P06）与闪欣结算（`settle_type=SHANXIN`，P07/P08）两条主体全覆盖；付款由付款申请链归口，本链只到「已核验 VERIFIED / 触发付款」交接点，付款完成回写 `SETTLED` 由付款链回调（OP10）；欣旺达→闪欣付款走财务共享 Mock（OP12）。工单/服务商侧字段来自上游 `claim_order`/`xw_claim_order`，标「待挂载」。
> 结算单号 `settle_no`：服务商结算前缀 `SPJS`、闪欣结算前缀 `SXJS`，`<前缀>-yyyyMMdd-4位日内流水`，取号规则见 `conventions S7`（`doc_no_sequence` 同事务行锁），不在此重复算法。

## 2. 接口详情

### 2.1 自动生成结算单（定时作业）

- **接口名称**：自动生成结算单
- **触发方式**：`—（定时作业，无独立 REST 接口）`
- **调度**：由 Spring 调度 / XXL-Job 在每月 `settle_period_config.gen_day` 0 点触发，系统内部执行 OP1 落库逻辑（LLD OP1「谁能点：系统定时任务，无人工」）。不对外暴露 REST 端点、无人工触发入口；手动提前结算归口 OP2（见 2.2）。作业按 `settle_type` 两类（PROVIDER / SHANXIN）分别跑。

#### 作业入参（内部调度参数，非 REST）

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `settleType` | String | 否 | 限定本次生成的结算类型（枚举见附录 4.2）；不传=两类都跑 | `PROVIDER` |

#### 业务处理逻辑（写操作）

1. **前置校验**：读 `settle_period_config`（取 `frequency=MONTHLY`、`gen_day`、`period_start_day`、`period_end_day`、`belong_basis=APPROVED_TIME`、`zero_amount_rule=NOT_GENERATE` 的有效配置），任务启动时快照当时生效配置；配置修改只对下一未生成周期生效，不回改已生成单。
2. **周期日期生成**：以触发时点为锚，取严格早于本次触发时间的最近一个 `period_end_day` 为 `settle_period_end`（配置 31 而目标月不足则钳制为该月最后一天）；`period_start_day<=period_end_day` 取同月 start day，否则取上一月 start day；查询用半开区间 `[settle_period_start, period_end_exclusive)`（`period_end_exclusive = settle_period_end + 1 日`）。
3. **归属时间口径**：服务商结算按索赔单「闪欣业务审核通过时间」、闪欣结算按「欣旺达审批通过时间」归属；本期从 `claim_audit_record` 派生（`biz_type=CLAIM`/`XW`、`audit_stage=业务审核`/`欣旺达审核`、`audit_result=通过`、取 `audit_time`），快照列待 schema 文档 D-结算-B1。
4. **幂等**：幂等键 = `settle_type + provider_id(SHANXIN 为空) + settle_period_start + settle_period_end`；同键有效单为 `PENDING_CONFIRM` 时行锁下合并本周期新出现的未占用明细（含 D-结算-B6 迟到滚入项）并重算金额，不新建；同键单已进对账及后续状态时不改旧单，迟到项留下一可执行周期。**迟到数据（D-结算-B6）**：原归属时间 < 本执行周期起始日且仍未占用者纳入下一次可执行周期，不补开 / 不回改原单，源单归属日期不变。
5. **服务商结算聚合（settle_type=PROVIDER，逐服务商）**：
   - 候选索赔：`claim_order WHERE status='PENDING_SETTLE' AND deleted_at IS NULL AND provider_id=:pid AND 归属时间 < period_end_exclusive AND id 不在任何 status<>VOIDED 的 claim_settle_detail`。
   - 候选抵扣：`claim_deduct_order WHERE status='UNSETTLED' AND deleted_at IS NULL AND provider_id=:pid AND settle_period_belong < period_end_exclusive`。
   - `payable_amount = Σ claim_order.apply_total_amount − Σ claim_deduct_order.deduct_amount`。
6. **落库·分支 A（payable_amount≠0 → 建单，同事务）**：
   - 写 `claim_settle_order`：`id`=雪花；`settle_no`=SPJS 编码（conventions S7 同事务取号）；`settle_type='PROVIDER'`；`payee_name`=服务商名；`counterpart_name=NULL`；`provider_id`=:pid；`region`=服务商区域；`settle_period_start`/`settle_period_end`=周期；`payable_amount`=上式；`currency='CNY'`；`claim_count`=纳入索赔单数；`status='PENDING_CONFIRM'`；`invoice_permission_flag`=生单白名单快照（`invoice_whitelist.provider_id=:pid AND enabled=1 AND deleted_at IS NULL AND COALESCE(effective_from,'1000-01-01')<=DATE(created_at)` 则 1 否则 0）；`force_settle_flag=0`；`recon_confirm_time`/`invoice_code`/`invoice_no`/`invoice_date`/`invoice_amount`/`invoice_attachment_id`/`settle_time=NULL`。
   - 每张索赔单写一行 `claim_settle_detail`：`settle_id`=上单 id；`claim_no`=索赔单号；`xw_claim_no=NULL`；`work_order_no`/`claim_type`/`provider_id`/`provider_code`/`provider_name`/`region`=索赔单带出；`claim_total=apply_total_amount`；`labor_fee_total`/`part_mgmt_fee`/`other_fee_total`=索赔单对应；`deduct_amount=0.00`；`fee_owner=NULL`；`currency='CNY'`。
   - 每张抵扣单写一行 `claim_settle_detail`：`claim_no`=抵扣单号 `deduct_no`；`xw_claim_no=NULL`；`work_order_no`=抵扣单工单号；`claim_type=NULL`；`provider_*`=服务商；`claim_total=-claim_deduct_order.deduct_amount`（负）；`labor_fee_total`/`part_mgmt_fee`/`other_fee_total=0.00`；`deduct_amount=claim_deduct_order.deduct_amount`（正）；`fee_owner=NULL`；`currency='CNY'`。
   - 回写纳入抵扣单 `claim_deduct_order.status: UNSETTLED → SETTLED`（`WHERE id=? AND status='UNSETTLED'`，version 由后端 @Version 事务内自读自增）；`claim_order.status` 不改（靠明细占位排除后续扫描）。
   - 断言 `payable_amount = Σ claim_settle_detail.claim_total`，不等回滚。
7. **落库·分支 B（payable_amount=0 → 0 元核销，不建单，同事务）**：不写结算单 / 明细；参与 `claim_order.status: PENDING_SETTLE → SETTLED`、`claim_deduct_order.status: UNSETTLED → SETTLED`；无发票 / 付款 / 质保金动作。
8. **闪欣结算聚合（settle_type=SHANXIN，统一一张不拆分）**：候选 `xw_claim_order WHERE status='PENDING_SETTLE' AND deleted_at IS NULL AND 归属时间 < period_end_exclusive AND 未占用`；`payable_amount=Σ COALESCE(approved_amount,apply_total_amount)`（含备件管理费，恒≥0）；写 `claim_settle_order`：`settle_type='SHANXIN'`、`settle_no`=SXJS 编码、`payee_name='闪欣动力'`、`counterpart_name='欣旺达售后'`、`provider_id=NULL`、`region=NULL`、`invoice_permission_flag=1`、其余同分支 A；每张写 `claim_settle_detail`：`claim_no=NULL`、`xw_claim_no`=闪欣索赔单号、`claim_total`=核准金额、`part_mgmt_fee=xw_claim_order.part_mgmt_fee`（落库计入金额）、`fee_owner=NULL`（待人工在 OP3a/OP3b 补选）；`payable_amount=0` 时同分支 B 置 `xw_claim_order.status=SETTLED`、不建单。
9. **状态流转**：—（不存在）→ `PENDING_CONFIRM`（应付≠0）/ 不建单（应付=0，索赔单直接 SETTLED）。
10. **事务边界**：单服务商 / 闪欣维度的结算单 + 全部明细 + 抵扣单回写为**一个独立事务**，任一步失败整体回滚；各维度独立提交，某维度失败不影响其他维度。

#### 作业运行汇总（内部日志 / 监控输出，非 REST 响应）

| 参数名 | 类型 | 说明 | 示例 |
| :--- | :--- | :--- | :--- |
| `success` | Boolean | 作业是否成功 | `true` |
| `code` | String | 业务码 | `"0"` |
| `message` | String | 提示信息 | `生成完成` |
| `data` | JSON | 生成汇总（运行期聚合，非表列） | |
| ├ `generatedCount` | Integer | 本次生成的结算单数 | `3` |
| ├ `zeroWriteoffCount` | Integer | 0 元核销（不建单）的维度数 | `1` |
| ├ `generatedList` | JSON[] | 已生成结算单 | |
| │  ├ `settleId` | String | 结算单 id | `1830000000000000001` |
| │  ├ `settleNo` | String | 结算单号 | `SPJS-20260701-0012` |
| │  ├ `settleType` | String | 结算类型 | `PROVIDER` |

### 2.2 手动提前结算（手动合并结算单）

- **接口名称**：手动提前结算
- **请求路径**：`/settle/order/manual-generate`
- **请求方式**：`POST`
- **描述**：闪欣索赔专员在 P05「+ 手动合并结算单」发起，指定结算类型 + 主体 + 周期提前生成结算单（LLD OP2）。聚合与写库逻辑同 OP1，触发者为人工、周期可为当前进行中周期。

#### 请求参数（Body - JSON）

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `settleType` | String | 是 | 结算类型（枚举见附录 4.2） | `PROVIDER` |
| `providerId` | String | 否 | 服务商 id；`settleType=PROVIDER` 必填、`SHANXIN` 不传 | `PVD-0007` |
| `settlePeriodStart` | String | 是 | 结算周期起（`yyyy-MM-dd`，含首） | `2026-06-01` |
| `settlePeriodEnd` | String | 是 | 结算周期止（`yyyy-MM-dd`，含尾） | `2026-06-30` |

#### 请求示例

```json
{ "settleType": "PROVIDER", "providerId": "PVD-0007", "settlePeriodStart": "2026-06-01", "settlePeriodEnd": "2026-06-30" }
```

#### 业务处理逻辑（写操作）

1. **前置校验**：`settle_period_config.manual_advance=1`（未开放手动提前 → `code=400`）；`settleType=PROVIDER` 时 `providerId` 非空。
2. **幂等键校验**：幂等键 = `settle_type + provider_id(SHANXIN 为空) + settle_period_start + settle_period_end`；同键已有有效单则拒绝，返回 `SET_4010`（该周期该主体已有有效结算单）。
3. **聚合与落库**：与 OP1 分支 A / 分支 B 完全一致（应付≠0 建单、=0 走 0 元核销），`settle_no` 按 `yyyyMMdd`=操作当天生成（SPJS/SXJS，conventions S7）。
4. **状态流转**：—（不存在）→ `PENDING_CONFIRM`（应付≠0）/ 不建单（应付=0）。
5. **事务边界**：结算单 + 全部明细 + 抵扣单回写同一事务，任一步失败整体回滚。

#### 响应参数

| 参数名 | 类型 | 说明 | 示例 |
| :--- | :--- | :--- | :--- |
| `success` | Boolean | 业务是否成功 | `true` |
| `code` | String | 业务码 | `"0"` |
| `message` | String | 提示信息 | `生成成功` |
| `data` | JSON | 业务数据 | |
| ├ `settleId` | String | 结算单 id（0 元核销时为 null） | `1830000000000000002` |
| ├ `settleNo` | String | 结算单号（0 元核销时为 null） | `SPJS-20260714-0003` |
| ├ `status` | String | 单据状态（0 元核销时为 null） | `PENDING_CONFIRM` |
| ├ `payableAmount` | BigDecimal | 应付总金额 | `0.00` |
| ├ `zeroWriteoff` | Boolean | 是否 0 元核销未建单 | `false` |
| `traceId` | String | 链路追踪 ID | `a1b2c3d4e5f6` |

#### 响应示例（成功）

```json
{ "success": true, "code": "0", "message": "生成成功", "data": { "settleId": "1830000000000000002", "settleNo": "SPJS-20260714-0003", "status": "PENDING_CONFIRM", "payableAmount": 86500.00, "zeroWriteoff": false }, "traceId": "a1b2c3d4e5f6" }
```

### 2.3 新增结算明细

- **接口名称**：新增结算明细
- **请求路径**：`/settle/order/{id}/detail`
- **请求方式**：`POST`
- **描述**：待确认结算单上增补一张索赔单据到明细（LLD OP3a 增分支）。服务商结算由闪欣索赔专员操作，闪欣结算由欣旺达索赔专员操作。

#### 请求参数（Path + Body - JSON）

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | String | 是 | 结算单 id（Path，`claim_settle_order.id`） | `1830000000000000001` |
| `sourceClaimId` | String | 是 | 服务商结算=`claim_order.id`；闪欣结算=`xw_claim_order.id` | `1810000000000000005` |
| `feeOwner` | String | 否 | 费用归属（仅 `settle_type=SHANXIN` 必填，枚举见附录 4.4） | `SHANXIN` |

#### 请求示例

```json
{ "sourceClaimId": "1810000000000000005", "feeOwner": "SHANXIN" }
```

#### 业务处理逻辑（写操作）

1. **前置校验**：结算单 `status='PENDING_CONFIRM'`（乐观锁双条件 `WHERE id=? AND status='PENDING_CONFIRM'`，status 作双条件、version 由后端 MyBatis-Plus @Version 事务内自读自增、接口不传；命中 0 行 → `code=409`）。
2. **来源校验**：`settle_type='PROVIDER'` 时 `claim_order.status='PENDING_SETTLE'` 且不在其他 `status<>VOIDED` 有效结算明细中；`settle_type='SHANXIN'` 时 `xw_claim_order.status='PENDING_SETTLE'` 且未被占用，且 `feeOwner ∈ {XW,SHANXIN,INTERNAL_DEPT}`，选 `INTERNAL_DEPT` 时关联 `xw_claim_order.internal_dept` 必须非空；不满足 → `code=400` / `SET_4004`。
3. **落库** `INSERT claim_settle_detail`：服务商结算插入字段同 OP1 服务商索赔行（`claim_no`/`work_order_no`/`claim_type`/`provider_*`/`region`/`claim_total=apply_total_amount`/`labor_fee_total`/`part_mgmt_fee`/`other_fee_total`/`deduct_amount=0.00`/`fee_owner=NULL`/`currency='CNY'`）；闪欣结算插入字段同 OP1 SHANXIN 明细行（`xw_claim_no`/`claim_total=COALESCE(approved_amount,apply_total_amount)`/`part_mgmt_fee=xw_claim_order.part_mgmt_fee`/`fee_owner`=入参、其余对应）。
4. **公共回写** `UPDATE claim_settle_order`：`claim_count=未删除索赔行数`、`payable_amount=Σ claim_total`、`version=version+1`。
5. **边界（D-结算-B2）**：重算后 `payable_amount` 变号则按新符号在后续对账（OP5）重新分流；变为 0 立即走 0 元核销——当前 `PENDING_CONFIRM` 单置 `VOIDED`、明细软删、剩余参与索赔单置 `SETTLED`、相关抵扣单置 / 保持 `SETTLED`，不进对账 / 开票 / 付款 / 质保金流程（此处仅登记，实际 0 元触发点在增删动作提交时统一判定）。
6. **状态流转**：`PENDING_CONFIRM` 不变（除非重算后 =0 触发 `→ VOIDED` 0 元核销）。
7. **事务边界**：明细 INSERT + 两个汇总字段回写（+ 0 元核销时的作废 / 软删 / 索赔单末态）同一事务，任一步失败整体回滚。

#### 响应参数

| 参数名 | 类型 | 说明 | 示例 |
| :--- | :--- | :--- | :--- |
| `success` | Boolean | 业务是否成功 | `true` |
| `code` | String | 业务码 | `"0"` |
| `message` | String | 提示信息 | `操作成功` |
| `data` | JSON | 业务数据 | |
| ├ `detailId` | String | 新增明细行 id | `1832000000000000009` |

## 4. 附录 · 枚举字典

> 请求 / 响应里出现的枚举，在此列全（值 + 含义）。

### 4.1 结算单状态 `claim_settle_order.status`（`SETTLE_STATUS`，8 态状态机）

| 值 | 含义 | 迁移动作（谁 · 前置） → 新态 |
| :--- | :--- | :--- |
| `PENDING_CONFIRM` | 待确认 | 索赔专员确认结算 → `PENDING_RECON`；索赔专员作废（仅此态） → `VOIDED`；编辑致应付=0 → `VOIDED`（0 元核销） |
| `PENDING_RECON` | 待对账 | 收款方对账确认·应付>0 → `PENDING_INVOICE`；·应付<0（仅服务商结算） → `PENDING_FORCE` |
| `PENDING_INVOICE` | 待开票 | 开票方开票上传发票（`invoice_permission_flag=1`） → `PENDING_VERIFY`；`flag=0` 为挂起标记，OP6 解挂后仍留本态 |
| `PENDING_VERIFY` | 待核验 | 核验方发票确认 → `VERIFIED` |
| `VERIFIED` | 已核验 | 服务商结算：申请付款（触发付款链，保持 VERIFIED）→ 付款链回写付款完成 → `SETTLED`；闪欣结算：财务人工回填付款成功 → `SETTLED` |
| `PENDING_FORCE` | 待强制结算 | 索赔专员发起强制结算（上传依据，保持本态）→ 财务确认并执行扣减 → `SETTLED` |
| `SETTLED` | 已结算 | 终态 |
| `VOIDED` | 已作废 | 终态（仅由 `PENDING_CONFIRM` 作废 / 0 元核销进入） |

> 「账单挂起」非独立状态，是 `PENDING_INVOICE` 下 `invoice_permission_flag=0` 的挂起标记（A-32）。乐观锁由后端管理——`WHERE id=? AND status=?` 以 status 作双条件、version 由 MyBatis-Plus @Version 在事务内自读并自增，**接口请求 / 响应体均不含 version**；命中 0 行（冲突）返回「数据已变更请刷新」（`code=409`），前端靠 status + 冲突错误码感知并发。

### 4.2 结算类型 `claim_settle_order.settle_type`（`SETTLE_TYPE`）

| 值 | 含义 |
| :--- | :--- |
| `PROVIDER` | 服务商结算（闪欣结算给服务商，P05/P06，前缀 SPJS） |
| `SHANXIN` | 闪欣动力结算（欣旺达结算给闪欣，P07/P08，前缀 SXJS，恒正数、无质保金扣减） |

### 4.3 索赔类型 `claim_settle_detail.claim_type`（`CLAIM_TYPE`）

| 值 | 含义 |
| :--- | :--- |
| `WARRANTY` | 保内维修 |
| `SERVICE_ACTIVITY` | 服务活动 |
| `INTERNAL` | 内部服务 |
| `SPECIAL` | 特殊工单 |

### 4.4 费用归属 `claim_settle_detail.fee_owner`（`FEE_OWNER`，闪欣结算必填）

| 值 | 含义 |
| :--- | :--- |
