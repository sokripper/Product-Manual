> 通用说明（认证 / 基础 URL / `Result<T>` / `PageResult<T>` / 审计列 / 通用错误码 / 编号规则）见 `docs/conventions.md`，本文件不重复。
> 上游权威：`详细设计-向欣旺达索赔单.md`（S3 操作详解）、`docs/db-schema/表结构文档.md`（xw_claim_order / xw_claim_detail / claim_audit_record / attachment）、`docs/PRD.md S4.2`（F-XW-01~03）。每接口 = LLD 一个真实操作，每字段 = 已登记真列或 PRD 字段。
> 范围与边界：本链只写 `xw_claim_order` / `xw_claim_detail` / `claim_audit_record`(`biz_type='XW'`)，附件走 `attachment`(`biz_type='XW'`)。
> - **逐单自动生成（LLD S3.1）**：非人工点按，随服务商索赔单业务审核通过（F-CLAIM-07、欣旺达件）在**同一数据库事务**内由系统生成，无独立 REST 接口（见 S2.0）。
> - **欣旺达审核（A-22）**：本期为本系统内建角色操作，无跨系统接口——审核即本系统一个角色点按钮的内部接口。
> - **QMS 判责（A-20）**：本期 Mock / 人工录入，`liability_result` / `internal_dept` / `liability_ratio` 为回显位，由 S2.4 人工回填，不外呼 QMS。
> - **结算**：下游闪欣结算链（F-SET-06）读本链 `PENDING_SETTLE`；`SETTLED` 由结算链写，本链不写。

## 2. 接口详情

### 2.0 逐单自动生成（内部操作，无独立 REST 接口）

- **对应 LLD 操作**：S3.1 逐单自动生成（A-36）
- **触发方式**：**非人工点按**。服务商索赔单在「业务审核」（F-CLAIM-07）被闪欣索赔专员点通过、且该服务商单 `claim_order.claim_target='XW'`（欣旺达件）时，系统在**同一数据库事务**内逐单自动生成一张 `xw_claim_order` + 一条 `xw_claim_detail`（一张服务商单对一张闪欣单一条明细，非汇总）。因不由本链前端发起、无独立请求 / 响应报文，故本链不暴露 REST 接口，仅在此登记其落库逻辑供研发对齐。

#### 业务处理逻辑（写操作）

1. **前置**：服务商单 `claim_order.status` 迁至 `PENDING_SETTLE`（欣旺达件不经判责）、且 `claim_order.claim_target='XW'`。
2. **幂等校验**：同一 `claim_order.id` 只能生成一次——落 `xw_claim_detail.provider_claim_id`（`UNIQUE uk_provider_claim`）挡重复；已存在则跳过生成。
3. **取号**：`xw_claim_order.claim_no` = `SXCO-yyyyMMdd-4位日内流水`，同事务从 `doc_no_sequence` 按「前缀+日期」行锁取号（见 conventions S7），业务 INSERT 成功才占号、回滚退号，表上 `UNIQUE` 兜底。
4. **落库** `INSERT xw_claim_order`（一行）：
   - `id`=雪花；`claim_no`=SXCO 编号；`provider_claim_id`=`claim_order.id`；`provider_claim_no`=`claim_order.claim_no`；`work_order_no`=`claim_order.work_order_no`。
   - `repair_type` / `sub_repair_type` / `battery_sn` / `serial_no` / `vin`=工单库带出（**待挂载/Mock**，未提供写 NULL）；`province_city`=工单/服务商库带出（**待挂载**，未提供写 NULL）。
   - `provider_code` / `provider_name` / `region`=`claim_order` 对应列。
   - `labor_fee_total` / `part_mgmt_fee` / `other_fee_total`=`claim_order` 对应列；`apply_total_amount`=`labor_fee_total + part_mgmt_fee + other_fee_total`（含备件管理费、不乘系数 A-37、等于服务商单总额，D-向欣-B2）。
   - `approved_amount`=NULL；`currency`=`'CNY'`；`status`=`'DRAFT'`；`audit_result`=NULL。
   - `liability_result` / `internal_dept` / `liability_ratio`=NULL（QMS 回显位，本期人工/Mock 回填 A-20）。
   - `fee_transfer_started`=0；`provider_deduct_started`=0；`create_by`=业务审核操作人；`created_at` / `updated_at`=NOW()；`version`=0。
5. **落库** `INSERT xw_claim_detail`（一条）：`id`=雪花；`xw_claim_id`=新单 id；`provider_claim_id`=`claim_order.id`（`uk_provider_claim` 占位）；`provider_claim_no`=`claim_order.claim_no`；`work_order_no`=`claim_order.work_order_no`；`labor_fee_total`=`claim_order.labor_fee_total`；`other_fee_total`=`claim_order.other_fee_total`；`subtotal`=`labor_fee_total + other_fee_total`（明细表未登记备件管理费列）。
6. **状态流转**：无 → `DRAFT`。
7. **副作用**：无（等专员补附件/提交）。
8. **事务边界**：与 F-CLAIM-07 业务审核通过**同一事务**——服务商单置 `PENDING_SETTLE`、闪欣单落 `DRAFT`、明细落一条，三者全成或全滚；生成失败整笔业务审核回滚。

### 2.1 编辑（补附件）

- **接口名称**：编辑（补附件）
- **请求路径**：`/xw/claim/{id}`
- **请求方式**：`PUT`
- **描述**：闪欣索赔专员在草稿 / 已驳回态为闪欣索赔单补充附件（LLD S3.2）。只动附件，不动明细、不动金额。

#### 请求参数（Path + Body - JSON）

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | String | 是 | 闪欣索赔单 id（Path，`xw_claim_order.id`） | `1860000000000000001` |
| `attachments` | JSON[] | 否 | 新增附件列表（本地磁盘存相对路径） | |
| ├ `fileName` | String | 是 | 文件名（≤255） | `质检报告.pdf` |
| ├ `filePath` | String | 是 | 服务器相对路径（≤512） | `xw/2026/07/xxx.pdf` |
| ├ `fileType` | String | 否 | 文件类型（`IMAGE` / `PDF`，空写 NULL） | `PDF` |
| ├ `fileSize` | Long | 否 | 文件大小（字节，空写 NULL） | `204800` |

> 明细、金额（`laborFeeTotal` / `partMgmtFee` / `otherFeeTotal` / `applyTotalAmount`）在 S2.0 生成时一次性定稿，编辑阶段不重算、不接受修改，不在本请求体。`uploadBy` / `uploadTime` 由登录态 / 服务端带入，非前端传参。

#### 请求示例

```json
{
  "attachments": [
    { "fileName": "质检报告.pdf", "filePath": "xw/2026/07/xxx.pdf", "fileType": "PDF", "fileSize": 204800 }
  ]
}
```

#### 业务处理逻辑（写操作）

1. **前置校验**：`status ∈ {DRAFT, REJECTED}`（乐观锁由后端管理：status 双条件 `WHERE id=? AND status=? AND deleted_at IS NULL` + MyBatis-Plus `@Version` 事务内自读版本自增，接口不传 `version`；命中 0 行 → `code=409` 提示刷新，并发冲突前端靠 status + 冲突错误码感知）。其余态编辑拒（`XW_4001`）。已驳回可改后重提（H3）。
2. **落库** `INSERT attachment`（每个附件一行）：`biz_type`=`'XW'`、`biz_id`=本单 id、`file_name` / `file_path` / `file_type` / `file_size`=录入值、`upload_by`=当前用户、`upload_time`=NOW()。
3. **落库** `UPDATE xw_claim_order`：仅刷 `version`=version+1、`update_by`=当前用户、`updated_at`=NOW()；`status` 与金额、明细一律不改（当前登记的 `xw_claim_order` 无备注列，本期不提供独立文本说明编辑，不借未登记列落库）。
4. **状态流转**：不变（`DRAFT`→`DRAFT`，`REJECTED`→`REJECTED`）。
5. **副作用**：无。
6. **事务边界**：`attachment` insert + 头表 `version` 递增同一事务；乐观锁冲突整体回滚。

#### 响应参数

| 参数名 | 类型 | 说明 | 示例 |
| :--- | :--- | :--- | :--- |
| `success` | Boolean | 业务是否成功 | `true` |
| `code` | String | 业务码 | `"0"` |
| `message` | String | 提示信息 | `操作成功` |
| `data` | JSON | 业务数据 | |
| ├ `id` | String | 闪欣索赔单 id | `1860000000000000001` |
| ├ `status` | String | 单据状态（不变） | `DRAFT` |
| `traceId` | String | 链路追踪 ID | `a1b2c3d4e5f6` |

#### 响应示例（成功）

```json
{ "success": true, "code": "0", "message": "操作成功", "data": { "id": "1860000000000000001", "status": "DRAFT" }, "traceId": "a1b2c3d4e5f6" }
```

### 2.2 提交

- **接口名称**：提交
- **请求路径**：`/xw/claim/{id}/submit`
- **请求方式**：`POST`
- **描述**：闪欣索赔专员提交草稿 / 已驳回单，单据进入待审核，推送欣旺达索赔专员待办（LLD S3.3）。

#### 请求参数（Path）

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | String | 是 | 闪欣索赔单 id（Path） | `1860000000000000001` |

> 无请求体参数（仅路径参数 `id`）；乐观锁 `version` 由后端管理，接口不传。

#### 请求示例

```
POST /xw/claim/1860000000000000001/submit
```

#### 业务处理逻辑（写操作）

1. **前置校验**：`status ∈ {DRAFT, REJECTED}`（乐观锁 status 双条件 `WHERE id=? AND status=? AND deleted_at IS NULL` + `@Version` 事务内自读版本自增，接口不传 `version`；不符 → `code=409` / `XW_4001`）。
2. **业务校验**：`xw_claim_detail` 明细 ≥ 1 条，否则拒（`XW_4002`）。
3. **条件必填校验（A-39）**：电池包维修 `battery_sn` 必填、电机电控 `serial_no` 必填、整车 `vin` 17 位（内部场景豁免）。**本期这些字段由工单库 Mock 带出——待挂载**：Mock 未带出时放行不卡；真库接上后按 A-39 改为硬校验（D-向欣-B3 已确认）。
4. **落库** `UPDATE xw_claim_order`：`status`：`DRAFT/REJECTED` → `PENDING_AUDIT`；`version`=version+1、`update_by`=提交人、`updated_at`=NOW()。
5. **状态流转**：`DRAFT/REJECTED` → `PENDING_AUDIT`。
6. **副作用**：把单子推给欣旺达索赔专员待办审核（本系统内建角色，无跨系统调用 A-22）。
7. **事务边界**：单表状态更新，乐观锁一把；冲突回滚。

| `code` | String | 业务码 | `"0"` |
| `message` | String | 提示信息 | `提交成功` |
| `data` | JSON | 业务数据 | |
| ├ `id` | String | 闪欣索赔单 id | `1860000000000000001` |
| ├ `status` | String | 单据状态 | `PENDING_AUDIT` |
| `traceId` | String | 链路追踪 ID | `a1b2c3d4e5f6` |

#### 响应示例（成功）

```json
{ "success": true, "code": "0", "message": "提交成功", "data": { "id": "1860000000000000001", "status": "PENDING_AUDIT" }, "traceId": "a1b2c3d4e5f6" }
```

### 2.3 欣旺达审核（通过 / 驳回 / 拒绝 / 部分通过）

- **接口名称**：欣旺达审核
- **请求路径**：`/xw/claim/{id}/audit`
- **请求方式**：`POST`
- **描述**：欣旺达索赔专员（本系统内建角色 A-22）对待审核单出审核结论，按结果分支流转并写审核记录（LLD S3.4）。**内部角色操作，无跨系统接口。**

#### 请求参数（Path + Body - JSON）

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | String | 是 | 闪欣索赔单 id（Path） | `1860000000000000001` |
| `auditResult` | String | 是 | 审核结果枚举 `XW_AUDIT_RESULT`（见 S4.2） | `PARTIAL` |
| `approvedAmount` | BigDecimal | 条件必填 | 核准金额；`auditResult=PARTIAL` 时必填，须 `0 < approvedAmount ≤ applyTotalAmount` | `3200.00` |
| `remark` | String | 条件必填 | 审批意见 / 驳回原因；`auditResult ∈ {REJECT, DECLINE}` 必填；部分通过时记核减明细说明（≤500） | `SPCO-…0044 工时费核减 ¥520.00` |
| `attachmentIds` | String[] | 否 | 审核说明附件 id 列表（空写 NULL） | `["1870000000000000001"]` |

#### 请求示例

```json
{
  "auditResult": "PARTIAL",
  "approvedAmount": 3200.00,
  "remark": "SPCO-20260712-0044 工时费核减 ¥520.00",
  "attachmentIds": ["1870000000000000001"]
}
```

#### 业务处理逻辑（写操作）

1. **前置校验**：`status=PENDING_AUDIT`（乐观锁双条件；不符 → `code=409` / `XW_4001`，挡重复审核）。
2. **权限校验**：操作人角色 = 欣旺达索赔专员（内建角色），否则 `code=403`。
3. **入参校验**：
   - `auditResult ∈ {PASS, REJECT, DECLINE, PARTIAL}`，否则 `code=400`。
   - `auditResult=PARTIAL` 时 `approvedAmount` 必填且 `0 < approvedAmount ≤ apply_total_amount`（核准金额不得超申请总额、不得为 0/负），否则 `XW_4003`。
   - `auditResult ∈ {REJECT, DECLINE}` 时 `remark` 必填，否则 `XW_4004`。
4. **落库** `UPDATE xw_claim_order`（按结果分支，`version`=version+1、`update_by`=审核人、`updated_at`=NOW()）：

   | 结果 | 头表写 | 新状态 |
   | :-- | :-- | :-- |
   | 通过 `PASS` | `status='PENDING_SETTLE'`、`audit_result='PASS'`、`approved_amount` 保持 NULL（NULL=全额通过，结算按 `apply_total_amount`） | `PENDING_SETTLE` |
   | 部分通过 `PARTIAL`（A-40） | `status='PENDING_SETTLE'`、`audit_result='PARTIAL'`、`approved_amount`=录入核准金额（结算口径以此为准，覆盖 `apply_total_amount`） | `PENDING_SETTLE` |
   | 驳回 `REJECT` | `status='REJECTED'`、`audit_result='REJECT'` | `REJECTED`（可编辑重提） |
   | 拒绝 `DECLINE` | `status='DECLINED'`、`audit_result='DECLINE'` | `DECLINED`（终） |

5. **落库** `INSERT claim_audit_record`（四种结果都写一条）：`id`=雪花、`biz_type`=`'XW'`、`biz_id`=本单 id、`audit_stage`=`'欣旺达审核'`、`auditor`=欣旺达索赔专员、`audit_time`=NOW()、`audit_result`=`通过`/`驳回`/`拒绝`/`部分通过`（中文语义，对齐该列既有取值）、`approved_amount`=部分通过时核准金额其余 NULL、`remark`=审批意见/驳回原因/部分通过核减明细说明（本期文本记录、不建专表 D-向欣-B4）、`attachment_ids`=审核说明附件 id 列表（可空）。
6. **状态流转**：`PENDING_AUDIT` → `PENDING_SETTLE` / `REJECTED` / `DECLINED`。
7. **副作用**：进 `PENDING_SETTLE` 后，闪欣结算链（F-SET-06）汇总本链待结算单，按 `approved_amount`（部分通过）或 `apply_total_amount`（全额通过）取数；结算完成由结算链把本单置 `SETTLED`（**本链不写 SETTLED**）。QMS 判责回显不与本审核事务捆绑（走 S2.4）。
8. **事务边界**：头状态更新 + 审核记录写入同一事务，乐观锁一把；冲突回滚，审核记录不落。

#### 响应参数

| 参数名 | 类型 | 说明 | 示例 |
| :--- | :--- | :--- | :--- |
| `success` | Boolean | 业务是否成功 | `true` |
| `code` | String | 业务码 | `"0"` |
| `message` | String | 提示信息 | `审核完成` |
| `data` | JSON | 业务数据 | |
| ├ `id` | String | 闪欣索赔单 id | `1860000000000000001` |
| ├ `status` | String | 单据状态 | `PENDING_SETTLE` |
| ├ `auditResult` | String | 审核结果 | `PARTIAL` |
| ├ `approvedAmount` | BigDecimal | 核准金额（非部分通过为 null） | `3200.00` |
| `traceId` | String | 链路追踪 ID | `a1b2c3d4e5f6` |

#### 响应示例（成功）

```json
{ "success": true, "code": "0", "message": "审核完成", "data": { "id": "1860000000000000001", "status": "PENDING_SETTLE", "auditResult": "PARTIAL", "approvedAmount": 3200.00 }, "traceId": "a1b2c3d4e5f6" }
```

### 2.4 QMS 判责 Mock 人工回填

- **请求路径**：`/xw/claim/{id}/qms-liability`
- **请求方式**：`POST`
- **描述**：具「QMS Mock 回填」权限的欣旺达索赔专员人工回填 QMS 判责回显位（LLD S3.4a，A-20 Mock）。真实 QMS 契约到位后由适配层回调替换本入口，**本期不外呼 QMS**。

#### 请求参数（Path + Body - JSON）

| 参数名 | 类型 | 必选 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | String | 是 | 闪欣索赔单 id（Path） | `1860000000000000001` |
| `liabilityResult` | String | 是 | 判责结果 `LIABILITY_RESULT`：`SUPPLIER` / `PROVIDER` / `INTERNAL`（见 S4.4） | `INTERNAL` |
| `liabilityRatio` | BigDecimal | 是 | 责任比例（%），`0 < liabilityRatio ≤ 100` | `100.00` |
| `internalDept` | String | 条件必填 | 内部责任部门；`liabilityResult=INTERNAL` 必填，其余显式写 NULL（≤128） | `品质部` |
| `liabilityDesc` | String | 是 | 判责说明（落 `claim_audit_record.remark`，≤500） | `内部装配工艺缺陷，判内部责任` |
| `attachmentIds` | String[] | 否 | 证据附件 id 列表（空写 NULL） | `["1870000000000000009"]` |

#### 请求示例

```json
{
  "liabilityResult": "INTERNAL",
  "liabilityRatio": 100.00,
  "internalDept": "品质部",
  "liabilityDesc": "内部装配工艺缺陷，判内部责任",
  "attachmentIds": ["1870000000000000009"]
}
```

#### 业务处理逻辑（写操作）

1. **前置校验**：先锁 `xw_claim_order` 行并复核状态：允许 `status ∈ {DRAFT, REJECTED, PENDING_AUDIT, PENDING_SETTLE}`，且该单尚未进入任何有效 `claim_settle_detail`；已结算、已作废/拒绝或已纳入结算单时拦截（`XW_4006`），避免分账依据在结算中途被改写。乐观锁双条件命中 0 行 → `code=409` / `XW_4001`。
2. **权限校验**：操作人具「QMS Mock 回填」权限，否则 `code=403`。
3. **入参校验**：`liabilityResult ∈ {SUPPLIER, PROVIDER, INTERNAL}`；`0 < liabilityRatio ≤ 100`；`liabilityResult=INTERNAL` 时 `internalDept` 必填（其余显式 NULL）；`liabilityDesc` 必填。不满足 → `XW_4005`。
4. **落库** `UPDATE xw_claim_order`：写 `liability_result` / `internal_dept`（非 INTERNAL 显式 NULL）/ `liability_ratio`；`liabilityResult=PROVIDER` 时 `provider_deduct_started=1, fee_transfer_started=0`，`SUPPLIER/INTERNAL` 时 `fee_transfer_started=1, provider_deduct_started=0`（本期两标志仅表示「已记录后续抵扣/转嫁意图」，不伪造外部单据；OA/供应商子流程待挂载）；`version`=version+1、`update_by`=回填人、`updated_at`=NOW()。
5. **落库** `INSERT claim_audit_record`：`biz_type`=`'XW'`、`biz_id`=本单 id、`audit_stage`=`'QMS_MOCK'`、`audit_result`=判责枚举（`SUPPLIER`/`PROVIDER`/`INTERNAL`）、`auditor`=回填人、`audit_time`=NOW()、`remark`=判责说明、`attachment_ids`=证据附件 id 列表。
6. **状态流转**：单据状态不变。
7. **副作用**：重复回填新增一条历史并覆盖头表当前值，保留可追溯修改轨迹。
8. **事务边界**：头表与留痕记录同事务，任一失败整体回滚。

## 3. 错误码（本模块业务专属）

> 通用错误码见 `docs/conventions.md S6`（`400` / `401` / `403` / `404` / `409` / `500`）。

| 错误码 | 说明 | 触发场景 |
| :--- | :--- | :--- |
| `XW_4001` | 单据状态不允许该操作 | 编辑/提交非 `DRAFT/REJECTED`；审核非 `PENDING_AUDIT`（重复审核）；作废非 `DRAFT`；乐观锁双条件命中 0 行 / 状态机非法迁移 |
| `XW_4002` | 明细为空 | 提交时 `xw_claim_detail` 明细 < 1 条 |
| `XW_4003` | 核准金额非法 | 部分通过 `approvedAmount` 缺失 / `≤0` / `> apply_total_amount` |
| `XW_4004` | 审批意见缺失 | `auditResult ∈ {REJECT, DECLINE}` 未填 `remark` |
| `XW_4005` | 判责参数非法 | QMS 回填 `liabilityResult` 越枚举 / `liabilityRatio` 不在 `(0,100]` / `INTERNAL` 缺 `internalDept` / 缺判责说明 |
| `XW_4006` | 判责回填不允许 | QMS 回填时单据已结算 / 已作废/拒绝 / 已纳入有效 `claim_settle_detail` |

> 条件必填（A-39：`battery_sn` / `serial_no` / `vin`）本期由工单库 **Mock 带出、待挂载**，未带出时提交放行不卡，暂不设错误码；真库接上后按 A-39 改硬校验并补错误码（D-向欣-B3）。

### 4.1 闪欣索赔单状态 `xw_claim_order.status`（`XW_CLAIM_STATUS`，7 态）

| 值 | 含义 |
| :--- | :--- |
| `DRAFT` | 草稿 |
| `PENDING_AUDIT` | 待审核 |
| `PENDING_SETTLE` | 待结算 |
| `REJECTED` | 已驳回（可编辑重提） |
| `DECLINED` | 已拒绝（终态） |
| `SETTLED` | 已结算（终态，**下游结算链写，本链不写**） |
| `VOIDED` | 已作废（终态） |

**状态机流转**（LLD S1）：

| 现态 | 事件 | 新态 | 触发方 |
| :-- | :-- | :-- | :-- |
| （无） | 服务商单业务审核通过且 `claim_target=XW` → 逐单自动生成 | `DRAFT` | 系统（随 F-CLAIM-07 同事务） |
| `DRAFT` / `REJECTED` | 编辑（补附件，不动明细） | 态不变 | 闪欣索赔专员 |
| `DRAFT` / `REJECTED` | 提交 | `PENDING_AUDIT` | 闪欣索赔专员 |
