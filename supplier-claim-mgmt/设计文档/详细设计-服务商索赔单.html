详细设计 · 服务商索赔单链（后端 LLD）

本文覆盖服务商索赔单从新建到结算入口的完整后端行为，字段级、可开发。范围表：claim_order（主单）、claim_fee_labor（工时费明细）、claim_fee_part（备件明细）、claim_fee_other（其它费用明细）、claim_audit_record（审核记录）、claim_deduct_order（索赔抵扣单）、attachment（附件）。工单库取数一律 Mock、标「待挂载」；向欣旺达索赔单、结算单、质保金账户、供应商索赔、OA 转嫁属于相邻链，本文只写触发点，不写其内部字段。

技术底座：Spring Boot 3 + MyBatis-Plus + MySQL8；主单更新一律带乐观锁（WHERE id=? AND status=? AND version=? AND deleted_at IS NULL，命中 0 行即判为并发冲突/状态已变，抛业务码并整事务回滚）；一个操作涉及的多张表在同一数据库事务内提交，任一步失败全部回滚；所有查询附 deleted_at IS NULL。

一、先约定：这条链里反复用到的东西

  • 谁在操作：接口进来先做 RBAC 鉴权（未登录 401，越权 403）+ 数据范围过滤。服务商售后专员只看/只动本服务商单；区域负责人只动本区域单；索赔专员全国；质量部只动【待责任判定】的自采件单。数据范围过滤在 SQL 里拼 provider_id / region 条件，不靠前端。
  • 状态取值：claim_order.status 只允许 CLAIM_STATUS 九个码值：DRAFT / PENDING_REGION / PENDING_HQ / PENDING_LIABILITY / PENDING_SETTLE / REJECTED / DECLINED / SETTLED / VOIDED。任何写状态的地方都先判「当前状态是否为该操作允许的前置状态」，不是就拦下（见第九节非法迁移）。
  • 单号怎么来：claim_no = SPCO-yyyyMMdd-4位日内流水。落库前按已确认的 D-编码-D1/D2 方案，从 doc_no_sequence 以「前缀+日期」同事务行锁取号，业务 INSERT 成功才占号、回滚退号；uk_claim_no 唯一索引兜底。抵扣单号 deduct_no = DKD-yyyyMMdd-4位流水，同机制。序列表 DDL 仍待研发组长审定（D-编码-B1）。
  • 金额恒等式（落单即算、只读）：apply_total_amount = labor_fee_total + part_mgmt_fee + other_fee_total。其中 labor_fee_total = Σ claim_fee_labor.amount，part_mgmt_fee = Σ claim_fee_part.mgmt_fee，other_fee_total = Σ claim_fee_other.amount。备件索赔金额 claim_fee_part.claim_amount 恒为 0。
  • 系数快照：新建时把「发起时点的服务商索赔系数」写进 claim_order.coefficient_snapshot，之后系数配置变了也不动历史单（A-2）。备件管理费 mgmt_fee = 备件销售价 × coefficient_snapshot。
  • 审计列：每次写主单都要带 update_by / updated_at(NOW()) / version=version+1；新建带 create_by / created_at。
  • 附件挂哪：都进 attachment 表，用 biz_type + biz_id 定位。索赔单主体附件 biz_type=CLAIM, biz_id=claim_order.id；判责录入附件 biz_type=LIABILITY, biz_id=claim_order.id。回运单附件按 A-11 也落 attachment（判定见「提交」一节，B-3）。

二、新建暂存草稿（对应 F-CLAIM-01）

谁能点 + 前置：服务商售后专员，在「新建索赔单」入口。无前置单据状态（是新建）。

校验（新建阶段从宽，只拦硬错）：

  1. 关联工单必须是「本服务商 + 状态=已完工 + 保内」的维修工单（A-8）。工单状态/保内标识/服务商归属都从工单库取——待挂载/Mock，本期用 Mock 数据源判断。
  2. 同一工单已存在一张状态非 DECLINED/VOIDED 的索赔单则拦下（A-15：只有已拒绝/已作废才允许重新发起；REJECTED 仍是可编辑重提的同一张有效单，不能另建）。为避免“先查后插”并发穿透，新建必须使用同一 DB 连接先取工单维度互斥锁 SELECT GET_LOCK(CONCAT('claim-wo-',MD5(?)),5)；取锁失败/超时直接返回“该工单正在发起索赔，请稍后重试”。取锁后在事务内再查 claim_order WHERE work_order_id=? AND status NOT IN (DECLINED,VOIDED) AND deleted_at IS NULL，通过后才插入；提交/回滚后在 finally 中用同连接 RELEASE_LOCK。该锁是连接级而非事务级，不得在取锁与放锁之间归还连接池。
  3. 工时费合计必须 > 0（A-1，S8.2「工时费>0 必有」）；其它费用合计 ≥ 0。
  4. 草稿阶段允许必填项暂缺（附件、claim_target 等留白），提交时再严格校验。

写哪些表哪些字段 = 什么值（同一事务）：

  • claim_order 插一行：
    • id=雪花；claim_no=生成的 SPCO 号；status='DRAFT'；currency='CNY'。
    • work_order_id / work_order_no / work_order_biz_type / provider_id / provider_code / provider_name / region=工单库带出（待挂载/Mock）。
    • claim_type=按 work_order_biz_type 查 claim_type_mapping 映射得到，只读带入（A-9）；映射查不到时留空并报「工单业务类型未配置映射」拦截（不可静默）。
    • coefficient_snapshot=发起时点服务商系数：优先取工单库系数（待挂载），无则查本模块 claim_coefficient（按 provider_id 命中，否则全局默认行）当前生效值。
    • labor_fee_total / part_mgmt_fee / other_fee_total=由三张明细汇总回填；apply_total_amount=三者之和。
    • main_part_id / main_part_name=工单主责件带出（待挂载/Mock），可为空（区域初审前允许）。
    • claim_target=NULL、project_code=NULL（区域初审才录）；liability_*=NULL；supplier_claim_started=0、provider_deduct_started=0。
    • 审计列：create_by=当前用户、created_at=NOW()、version=0。
  • claim_fee_labor 按工单维修项目逐行插：claim_id=主单 id、project_name / labor_hours / labor_price=工单带出（待挂载）、amount=labor_hours × labor_price。
  • claim_fee_part 按工单备件逐行插：claim_id、part_name / part_attr / battery_sn / old_part_code=工单带出（待挂载）、claim_amount=0.00（恒 0）、mgmt_fee=销售价 × coefficient_snapshot。
  • claim_fee_other 插工单带出的其它费用行：claim_id、fee_type / amount / remark；editable=物流费等可人工改的置 1，其余置 0（A-7）。
  • 附件此刻可传可不传；若传，attachment 逐个插（biz_type=CLAIM）。
  • claim_audit_record 插入新建操作事件：biz_type='CLAIM'、biz_id=新单 id、audit_stage='OPERATION'、audit_result='CREATE'、auditor/audit_time=操作人/时间，remark='新建草稿'。

状态变成：DRAFT（草稿）。

触发下游：无。

事务/回滚：主单+三张明细+附件同一事务；任一插入失败整体回滚，不留半张单。

三、提交（对应 F-CLAIM-02，也用于「已驳回」重新提交）

谁能点 + 前置：服务商售后专员；前置状态 DRAFT 或 REJECTED（已驳回可编辑后重提，A-13）。

校验：

  1. 走一遍新建的硬校验（工单仍为已完工保内、条件唯一、工时费>0）。
  2. 旧件回运（A-11）：逐条看 claim_fee_part 是否「需回运」；需回运的必须「已发起回运」才放行，否则拦「存在需回运备件未发起回运，不能提交」。
    • 「是否需回运」判定：读 claim_fee_part.part_attr 是否含「需回运」标识（B-2：当前无结构化布尔位，暂按属性文本判定，建议工单取数补布尔字段）。
    • 「已发起回运」判定：本模块未登记回运单实体/回运状态；按 A-11「售后仓点确认+回运单附件即视为已发起」，暂以「该单存在回运单附件」作为已发起证据（B-3：回运单实体/在途状态待挂载，附件判定为临时口径）。
  3. 维修凭证附件必填（有据·硬校验）：提交时必须至少 1 张维修凭证类附件（维修照片/检测报告等），无维修凭证不可提交（原始 PRD S4.1.1 服务商索赔单字段表明文「维修凭证附件…提交时必填」）。旧件资料附件仍为选填（原始 S4.1.1 标非必填），不随此项一起强制。

写哪些表哪些字段（同一事务，乐观锁）：

  • claim_order：status='PENDING_REGION'；审计列刷新。其余业务字段不动。
  • 提交动作在 claim_audit_record 写一条操作事件：biz_type='CLAIM'、audit_stage='OPERATION'、audit_result='SUBMIT'（已驳回重提为 RESUBMIT）、auditor/audit_time=操作人/时间，remark=操作摘要。

状态变成：PENDING_REGION（待区域初审）。从 REJECTED 重提同样回到 PENDING_REGION（从头走，A-13）。

触发下游：无（等区域初审）。

回滚：单表更新，乐观锁未命中（并发/状态已变）即回滚报冲突。

四、编辑（对应 F-CLAIM-03）

谁能点 + 前置：服务商售后专员；前置状态 DRAFT 或 REJECTED。其它状态一律不可编辑。

能改什么：只能动「其它费用明细」（物流费等，A-7）。工时费明细、备件明细、备件管理费一律不可改（工时费/备件管理费按 S8.2 固定，来源工单+系数）。

  • 允许对 claim_fee_other 中 editable=1 的行做改金额/备注/增行/删行；editable=0 的行禁止改删（D-索赔-B8 已确认）。

写哪些表哪些字段（同一事务）：

  • claim_fee_other：按编辑结果 upsert/删除对应行。
  • claim_order：重算 other_fee_total=Σ claim_fee_other.amount，随后 apply_total_amount=labor_fee_total + part_mgmt_fee + other_fee_total（labor 和 part 两项不变）；审计列刷新。status 不变（草稿仍草稿、已驳回仍已驳回）。
  • claim_audit_record：插入 audit_stage='OPERATION'、audit_result='EDIT'，auditor/audit_time=操作人/时间，remark 记录本次其它费用增删改摘要。

状态变成：不变（编辑不改状态；改完可再走「提交」）。

触发下游：无。

回滚：明细与主单汇总在同一事务，金额必须一致回写，失败全回滚。

五、作废（对应 F-CLAIM-04）

谁能点 + 前置：服务商售后专员；前置状态仅 DRAFT。非草稿点作废一律拦下。

写哪些表哪些字段：

  • claim_order：status='VOIDED'；deleted_at 保持 NULL，审计列刷新。已作废是可查看的业务终态，不以技术软删除隐藏。
  • 明细行 claim_fee_* 与附件保留不删，确保 F-CLAIM-05 的全状态查看、附件和操作留痕仍可访问。同一工单重新发起由条件唯一规则排除 VOIDED 实现，不依赖软删除。

状态变成：VOIDED（已作废，终态）。

触发下游：无。作废后该工单可重新发起新单（条件唯一放行，A-15）。

操作留痕：与头表状态更新同事务写 claim_audit_record：audit_stage='OPERATION'、audit_result='VOID'、auditor/audit_time=操作人/时间，remark=作废原因或“作废”。

六、区域初审（对应 F-CLAIM-06）

谁能点 + 前置：区域负责人（只限本区域）；前置状态 PENDING_REGION。

要核查/要录的：

  1. 核查工单真实性、资料完整性（人工，不入库字段，通过与否体现在结果里）。
  2. 录 claim_target（XW=欣旺达件 / SELF=自采件，即主责件归属，A-6）。
  3. claim_target=XW 时 project_code（项目编码）必填；=SELF 时 project_code 置 NULL。
  4. 电池包 SN 自动预判责任（A-26，可覆盖、非终判）：对含 battery_sn 的备件，系统按 SN 预判一个责任倾向给质量部参考。预判规则来源为 QMS，本期 Mock/待挂载；预判值无专用登记字段，本期写入本次 claim_audit_record.remark，不新增专列；QMS 正式契约到位后再补预判字段与口径（D-索赔-B4 已确认）。此预判不写 claim_order.liability_result（终判专用）。
  5. 汽车电子自由填报工时把关（A-14）：一级分类=汽车电子的单，工时为自由填报，区域负责人在此判合理性，不合理可驳回。

三种结果分别写什么（同一事务，乐观锁）：

  • 通过：
    • claim_order：status='PENDING_HQ'；claim_target=录入值；project_code=录入值(XW) 或 NULL(SELF)；审计列刷新。
    • claim_audit_record 插一行：biz_type='CLAIM'、biz_id=主单 id、audit_stage='区域初审'、auditor=当前用户、audit_time=NOW()、audit_result='通过'、remark=核查意见（含本期 Mock 的 SN 预判倾向，D-索赔-B4）、approved_amount=NULL、attachment_ids=如有。
  • 驳回（退回服务商，A-13）：
    • claim_order：status='REJECTED'；claim_target/project_code 若已录可保留也可留空（不影响，重提会从头再录）；审计列刷新。
    • claim_audit_record：同上但 audit_result='驳回'、remark=驳回原因（必填）。
  • 拒绝（终止，A-15 已关闭=已拒绝）：
    • claim_order：status='DECLINED'；审计列刷新。
    • claim_audit_record：audit_result='拒绝'、remark=拒绝原因（必填）。

状态变成：PENDING_HQ / REJECTED / DECLINED。

触发下游：无（等业务审核）。已驳回可编辑重提（回 PENDING_REGION）；已拒绝终态不可再动。

回滚：主单状态 + 审核记录同一事务，乐观锁保并发。

七、业务审核（对应 F-CLAIM-07）

谁能点 + 前置：闪欣索赔专员（全国）；前置状态 PENDING_HQ。

核心：通过后按主责件归属唯一分流（A-6/A-12/A-36）。分流看 claim_order.claim_target（区域初审已录）。

三种结果（同一事务，乐观锁）：

  • 通过 · claim_target=XW（欣旺达件）：
    • claim_order：status='PENDING_SETTLE'（进待结算）；审计列刷新。
    • claim_audit_record 插：audit_stage='业务审核'、audit_result='通过'、auditor/audit_time/remark。
    • 触发下游：逐单自动生成一张向欣旺达索赔单（A-36，xw_claim_order + xw_claim_detail 草稿，金额=本单对应金额不乘系数 A-37）。该生成动作与主单状态更新放在同一事务保证「审核通过即有下游单」；xw_claim_order 的字段填充规则属向欣旺达链 LLD，本文只登记触发与关联键 provider_claim_id=claim_order.id / provider_claim_no=claim_no。claim_order 侧无回写字段（无 xw 关联列）。
  • 通过 · claim_target=SELF（自采/外采件）：
    • claim_order：status='PENDING_LIABILITY'（进待责任判定，A-12）；审计列刷新。
    • claim_audit_record 插：audit_stage='业务审核'、audit_result='通过'。
    • 触发下游：无自动生成，等质量部判责。
  • 驳回 / 拒绝：同区域初审的驳回/拒绝语义——status='REJECTED'（可重提，回 PENDING_REGION 从头）或 status='DECLINED'（终态）；claim_audit_record 记 audit_stage='业务审核'、audit_result='驳回'/'拒绝'、remark=原因（必填）。

状态变成：PENDING_SETTLE（XW）/ PENDING_LIABILITY（SELF）/ REJECTED / DECLINED。

回滚：主单 + 审核记录 +（XW 时）向欣旺达单生成，全在一个事务；向欣旺达单生成失败则整个审核回滚，主单不置通过（避免「通过了但没生成下游单」）。

八、判责录入（对应 F-CLAIM-08）

谁能点 + 前置：质量部责任判定专员（只限【待责任判定】的自采件单）；前置状态 PENDING_LIABILITY。

要录的：责任判定结果 liability_result（三选一：SUPPLIER 供应商 / PROVIDER 服务商 / INTERNAL 内部，A-27）、责任说明 liability_desc（必填，≤200）、承担比例 liability_ratio（%，DECIMAL(5,2)；校验 0 < ratio ≤ 100，不允许 0；PROVIDER 责任必填，其余按需，D-索赔-B7 已确认）。可传判责附件（attachment，biz_type=LIABILITY）。

公共写入（三种结果都写，同一事务，乐观锁）：

  • claim_order：liability_result=录入值、liability_desc=录入值、liability_ratio=录入值、liability_by=当前用户、liability_time=NOW()、status='PENDING_SETTLE'；审计列刷新。
  • 判责附件若有 → attachment 逐条插（biz_type=LIABILITY, biz_id=claim_order.id）。

按结果分别再做：

  • PROVIDER（服务商责任）→ 自动生成索赔抵扣单：
    • claim_order：provider_deduct_started=1。
    • claim_deduct_order 插一行：id=雪花；deduct_no=生成 DKD 号；claim_id=主单 id；claim_no=主单号；provider_id=主单 provider_id；work_order_no=主单 work_order_no；liability_result='PROVIDER'；liability_ratio=录入比例；claim_liable_amount = claim_order.apply_total_amount（抵扣针对服务商全额应付，D-索赔-D1 已确认）；deduct_amount=claim_liable_amount × liability_ratio / 100（A-30）；settle_period_belong=该索赔单 claim_audit_record 中「业务审核通过」的 DATE(audit_time)（A-30，按闪欣审批通过时间归入同周期，不得写 NULL）；status='UNSETTLED'。
    • 抵扣单被结算链汇总时冲减服务商应付（本链只负责生成，不管结算）。
  • SUPPLIER（供应商责任）→ 触发供应商索赔流程：
    • claim_order：supplier_claim_started=1。
    • 供应商索赔子流程为 V2 范围（PRD S5「9月项」）；本期只置 supplier_claim_started=1、标记待挂载，不生成实际单据或占位单（D-索赔-B5 已确认）。备件费走供应商索赔（S8.3），本期不落单。
  • INTERNAL（内部责任）→ 记录比例，内部转嫁：
    • 仅记录 liability_result/liability_ratio/liability_desc（公共写入已含）；费用内部转嫁交 OA，本期待挂载，不生成转嫁单（B-6，A-19）。

状态变成：三种结果统一 PENDING_SETTLE（判责完成即进待结算，S6.1）。

触发下游：PROVIDER 生成抵扣单（本事务内）；SUPPLIER/INTERNAL 仅置标志/记录，实际下游待挂载。

回滚：主单判责字段 + 抵扣单 + 附件同一事务；抵扣单生成失败则判责整体回滚，主单不置待结算（避免「判了服务商责任却没抵扣单」）。

九、查看（对应 F-CLAIM-05）

AND deleted_at IS NULL。命中 0 行有两种原因：并发导致 version 变了、或状态已不在白名单（如对已结算单点作废、对草稿单点判责、对已拒绝/已作废终态做任何流转）。一律回滚并返回业务码「当前状态不允许该操作，请刷新」（HTTP 200，Result.success=false，S12 乐观锁+二次校验）。终态 DECLINED / VOIDED / SETTLED 不作为任何流转事件的前置态。SETTLED 由结算链负责，本链不主动写。

十一、事务与并发小结

  • 多表同事务的操作：新建（工单互斥下主单+3 明细+附件+操作事件）、编辑（明细+主单汇总+操作事件）、提交/重提（主单+操作事件）、作废（只改主单状态+操作事件，主单/明细/附件均不软删）、各审核（主单+审核记录）、业务审核 XW（主单+审核记录+向欣旺达单）、判责 PROVIDER（主单+抵扣单+附件）。任一步失败整体回滚。
  • 非审核操作事件：新建/编辑/提交/重提/作废分别写 audit_result=CREATE/EDIT/SUBMIT/RESUBMIT/VOID，统一 audit_stage='OPERATION'；审核节点仍使用原审核阶段和 PASS/REJECT/DECLINE/PARTIAL。
  • 乐观锁：claim_order 每次状态/金额更新走 version 比对，未命中即冲突回滚；核心动作（提交/审核/判责）叠加分布式锁按 claim_order.id 串行化，避免同单并发双审（S12、S15）。
  • 幂等：向欣旺达单生成以「一张服务商单仅可纳入一张闪欣单」约束兜底（下游 xw_claim_detail.uk_provider_claim）；抵扣单以 deduct_no 唯一 + 判责仅 PENDING_LIABILITY→PENDING_SETTLE 单向迁移，天然防重复生成。
  • 金额一致性：任何改明细的操作，必须在同事务内把 claim_order 的四个汇总字段重算回写，禁止只改明细不改主单。
