服务商索赔单 详细设计

本文覆盖服务商索赔单从新建到结算入口的完整后端行为，字段级、可开发。按 PRD 功能定义清单 F-CLAIM-01~10 一一对应，一个功能点一节。范围表：claim_order（主单）、claim_fee_labor（工时费明细）、claim_fee_part（备件明细）、claim_fee_other（其它费用明细）、claim_audit_record（审核记录）、claim_deduct_order（索赔抵扣单）、attachment（附件）。工单库取数一律 Mock、标「待挂载」；向欣旺达索赔单、结算单、质保金账户、供应商索赔、OA 转嫁属于相邻链，本文只写触发点，不写其内部字段。

技术底座：Spring Boot 3 + MyBatis-Plus + MySQL8。主单每次更新都带乐观锁：WHERE id=? AND status=? AND version=? AND deleted_at IS NULL，命中 0 行即判为并发冲突或状态已变，抛业务码并整事务回滚。一个操作涉及的多张表在同一数据库事务内提交，任一步失败全部回滚。所有查询附 deleted_at IS NULL。

通用约定

这条链里反复用到、各功能点共享的约定集中放这里，各功能点正文不再重复。

谁在操作（RBAC + 数据范围）

  • 接口进来先做 RBAC 鉴权（未登录 401，越权 403）+ 数据范围过滤。
  • 数据范围：服务商售后专员只看/只动本服务商单；区域负责人只动本区域单；闪欣索赔专员全国；质量部只动【待责任判定】的自采件单。
  • 数据范围过滤在 SQL 里拼 provider_id / region 条件，不靠前端。

状态取值（九个码值）

  • claim_order.status 只允许 CLAIM_STATUS 九个码值：DRAFT / PENDING_REGION / PENDING_HQ / PENDING_LIABILITY / PENDING_SETTLE / REJECTED / DECLINED / SETTLED / VOIDED。
  • 任何写状态的地方都先判「当前状态是否为该操作允许的前置状态」，不是就拦下（见文末迁移总表下方的非法迁移拦截）。
  • 术语归一（来自 PRD S7）：「已关闭」=「已拒绝」DECLINED（A-15）；「审核驳回」=「已驳回」REJECTED（A-4）。

单号怎么来

  • claim_no = SPCO-yyyyMMdd-4位日内流水（PRD S10 编码规则：SPCO-8位年月日-4位流水，全局唯一）。落库前按已确认的 D-编码-D1/D2 方案，从 doc_no_sequence 以「前缀+日期」同事务行锁取号，业务 INSERT 成功才占号、回滚退号；uk_claim_no 唯一索引兜底。
  • 抵扣单号 deduct_no = DKD-yyyyMMdd-4位流水，同机制。
  • 序列表 DDL 仍待研发组长审定（D-编码-B1）。

金额恒等式（落单即算、只读）

  • apply_total_amount = labor_fee_total + part_mgmt_fee + other_fee_total。
  • labor_fee_total = Σ claim_fee_labor.amount；part_mgmt_fee = Σ claim_fee_part.mgmt_fee；other_fee_total = Σ claim_fee_other.amount。
  • 备件索赔金额 claim_fee_part.claim_amount 恒为 0（PRD S8.2）。

系数快照

  • 新建时把「发起时点的服务商索赔系数」写进 claim_order.coefficient_snapshot，之后系数配置变了也不动历史单（A-2）。
  • 备件管理费 mgmt_fee = claim_fee_part.sale_price（备件销售价，落库）× coefficient_snapshot（裁决 D-索赔-B11：销售价落库 sale_price 使 mgmt_fee 可追溯）。

审计列

  • 每次写主单都要带 update_by / updated_at(NOW()) / version=version+1；新建带 create_by / created_at。

附件挂哪

  • 都进 attachment 表，用 biz_type + biz_id 定位。
  • 索赔单主体附件 biz_type=CLAIM, biz_id=claim_order.id；判责录入附件 biz_type=LIABILITY, biz_id=claim_order.id。
  • 回运单附件按 A-11 也落 attachment（判定见 F-CLAIM-02 提交一节，B-3）。

全链状态流转总表（谁触发 + 条件 + 写哪些字段 + 拦截）

各功能点「业务流转说明」是本表对应行的展开，本表作为全链一览保留。

非法迁移拦截（统一策略）：每个写状态的操作，先用乐观锁 SQL 限定「前置状态白名单」——UPDATE ... WHERE id=? AND status IN (<该操作允许的前置态>) AND version=? AND deleted_at IS NULL。命中 0 行有两种原因：并发导致 version 变了、或状态已不在白名单（如对已结算单点作废、对草稿单点判责、对已拒绝/已作废终态做任何流转）。一律回滚并返回业务码「当前状态不允许该操作，请刷新」（HTTP 200，Result.success=false，PRD S12 乐观锁+二次校验）。终态 DECLINED / VOIDED / SETTLED 不作为任何流转事件的前置态。SETTLED 由结算链负责，本链不主动写。

F-CLAIM-01 新建索赔

功能说明：

  • 服务商售后专员关联一张【已完工 + 维修方案已过技术审核 + 保内】的维修工单，自动带出工单基础信息、维修项目、工时标准、备件明细（备件索赔金额=0、备件管理费）、其它费用、维修类型/SN/条码（只读带入，A-10）、主责件；索赔类型由工单业务类型带出只读（A-9）。
  • 保存后生成草稿单并带出费用明细（PRD 完成标准）。
  • 关键限制：仅保内且已过技术审核的已完工工单可发起（A-8，技术审核前置来源 PRD F-CLAIM-01）；同一工单同时只能有一张有效单（A-15）；工时费合计必须 > 0（A-1）。

开发类型：

  • 新增节点。

节点目录：

  • 服务商索赔管理 - 服务商索赔单列表（P01）- 新建索赔单（P02 表单）。

业务流转说明：

  • 服务商售后专员在「新建索赔单」入口发起，无前置单据状态（是新建）。系统先取工单维度互斥锁、再在事务内校验工单准入与唯一性，通过后同一事务内建主单 claim_order（status=DRAFT）+ 三张费用明细 + 系数快照 +（可选）附件 + 一条新建操作留痕，任一步失败整体回滚，不留半张单。落地终态为 DRAFT（草稿），不触发任何下游。

涉及数据表：

  • claim_order（索赔主单）：写入主单全部字段（单号/状态/工单带出信息/系数快照/金额汇总/审计列）—— 表结构与状态字典见 docs/db-schema/。
  • claim_fee_labor（工时费明细）：按工单维修项目逐行写入。
  • claim_fee_part（备件明细）：按工单备件逐行写入。
  • claim_fee_other（其它费用明细）：写入工单带出的其它费用行。
  • attachment（附件）：新建时可传可不传，若传按 biz_type=CLAIM 逐个写入。
  • claim_audit_record（审核记录）：写入一条新建操作事件。
  • 工单库相关取数表：只读、待挂载/Mock，物理列未登记，禁止自造。
  • claim_type_mapping（工单业务类型→索赔类型映射字典）：只读，用于带出 claim_type。
  • claim_coefficient（服务商索赔系数配置）：只读，用于取系数快照回退值。

解决方案：

  • 操作按钮：
    • 新建（表单，逐字段写规则）：
      • 校验（新建阶段从宽，只拦硬错）：
        1. 关联工单必须是「本服务商 + 状态=已完工 + 维修方案已过技术审核 + 保内」的维修工单（A-8；技术审核前置来源 PRD F-CLAIM-01）。工单状态/保内标识/服务商归属/维修方案技术审核结果都从工单库取——待挂载/Mock，本期用 Mock 数据源判断（技术审核结果本期由工单侧 Mock 保证，真库接上后回填判据）。
        2. 同一工单已存在一张状态非 DECLINED/VOIDED 的索赔单则拦下（A-15：只有已拒绝/已作废才允许重新发起；REJECTED 仍是可编辑重提的同一张有效单，不能另建）。为避免「先查后插」并发穿透，新建必须使用同一 DB 连接先取工单维度互斥锁 SELECT GET_LOCK(CONCAT('claim-wo-',MD5(?)),5)；取锁失败/超时直接返回「该工单正在发起索赔，请稍后重试」。取锁后在事务内再查 claim_order WHERE work_order_id=? AND status NOT IN (DECLINED,VOIDED) AND deleted_at IS NULL，通过后才插入；提交/回滚后在 finally 中用同连接 RELEASE_LOCK。该锁是连接级而非事务级，不得在取锁与放锁之间归还连接池。
        3. 工时费合计必须 > 0（A-1，S8.2「工时费>0 必有」）；其它费用合计 ≥ 0。
        4. 草稿阶段允许必填项暂缺（附件、claim_target 等留白），提交时再严格校验。
      • 操作按钮：
        • 保存：写入以下数据（同一事务）：
          • 写入 claim_order（插一行）：
            • id = 雪花
            • claim_no = 生成的 SPCO 号（见「编码规则」）
            • status = 'DRAFT'
            • currency = 'CNY'
            • work_order_id / work_order_no / work_order_biz_type / provider_id / provider_code / provider_name / region = 工单库带出（待挂载/Mock）
            • claim_type = 按 work_order_biz_type 查 claim_type_mapping 映射得到，只读带入（A-9）；映射查不到时留空并报「工单业务类型未配置映射」拦截（不可静默）
            • coefficient_snapshot = 发起时点服务商系数：优先取工单库系数（待挂载），无则查本模块 claim_coefficient 当前生效值。「当前生效值」选取谓词：enabled=1 AND deleted_at IS NULL AND COALESCE(effective_from,'1000-01-01') <= 发起日，先按 provider_id=:providerId 命中、无命中行时回退 provider_id IS NULL（全局默认行）；同一命中范围内多行时按 effective_from DESC, created_at DESC 取最新一行（与白名单 effective_from 同口径）
            • labor_fee_total / part_mgmt_fee / other_fee_total = 由三张明细汇总回填
            • apply_total_amount = 三者之和
            • main_part_id / main_part_name = 工单主责件带出（待挂载/Mock），可为空（区域初审前允许）
            • repair_type / sub_repair_type / serial_no / vin / province_city = 工单库带出的维修类型/二级类型/序列号/整车 VIN/所属省市（只读带入 A-10，待挂载/Mock；裁决 D-索赔-B9 已加列，支撑 A-14 汽车电子把关与 A-39 场景条件必填的分类判据），无值可空
            • claim_target = NULL、project_code = NULL（区域初审才录）
            • liability_* = NULL
            • supplier_claim_started = 0、provider_deduct_started = 0
            • create_by = 当前用户、created_at = NOW()、version = 0
          • 写入 claim_fee_labor（按工单维修项目逐行插）：
            • claim_id = 主单 id
            • project_name / labor_hours / labor_price = 工单带出（待挂载）
            • amount = labor_hours × labor_price
          • 写入 claim_fee_part（按工单备件逐行插）：
            • claim_id = 主单 id
            • part_name / part_attr / battery_sn / old_part_code = 工单带出（待挂载）
            • sale_price = 工单备件销售价（待挂载/Mock 带出，裁决 D-索赔-B11 已加列落库，保金额可追溯）
            • claim_amount = 0.00（恒 0）
            • mgmt_fee = sale_price × coefficient_snapshot（销售价落库于 sale_price，派生可复核，裁决 D-索赔-B11）
          • 写入 claim_fee_other（插工单带出的其它费用行）：
            • claim_id = 主单 id
            • fee_type / amount / remark = 工单带出
            • editable = 物流费等可人工改的置 1，其余置 0（A-7）
          • 写入 attachment（此刻可传可不传；若传逐个插）：biz_type=CLAIM、biz_id=新单 id
          • 写入 claim_audit_record（新建操作事件）：biz_type='CLAIM'、biz_id=新单 id、audit_stage='OPERATION'、audit_result='CREATE'、auditor/audit_time=操作人/时间、remark='新建草稿'
        • 取消：不保存，返回上一页。

编码规则：

  • claim_no = SPCO-yyyyMMdd-4位日内流水，例 SPCO-20260712-0007（依据：PRD S10 编码规则；D-编码-D1/D2 已确认，序列表 DDL 待 D-编码-B1）。取号走序列表 doc_no_sequence(prefix, seq_date, current_seq)：以「前缀+日期」同事务行锁自增（详见 conventions S7）、INSERT 成功才占号、回滚退号，uk_claim_no 兜底。

外部接口：

  • 工单库取数：对接系统为维修工单管理模块；触发时机为进入新建表单带出工单基础信息/维修项目/工时/备件/主责件/系数/技术审核结果/VIN·SN·条码。本期 Mock/待挂载，物理列未登记，真库到位走变更登记回填（A-21，见「留白与取数边界」）。失败处理：Mock 源缺关键判据（技术审核结果等）时按校验规则拦截，不静默放行。

导入：

  • 无（新建单条表单录入；批量导入场景见 F-CLAIM-09 涉及的汽车电子旧件条码 Excel，属相邻能力，不在本节）。

防重复与并发：

  • 工单维度互斥锁 GET_LOCK('claim-wo-'+MD5(work_order_id)) 连接级串行化，防「先查后插」并发穿透造成一工单两张有效单；取锁失败即返回重试。
  • 主单+三张明细+附件+操作事件同一事务，任一插入失败整体回滚。
  • uk_claim_no 唯一索引兜底单号并发重复。

F-CLAIM-02 提交索赔

功能说明：

  • 服务商售后专员把草稿或已驳回单提交，进入区域初审（PRD 完成标准：状态→待区域初审）。
  • 提交时维修凭证附件必填（原始 PRD S4.1.1，硬校验），旧件资料附件选填；校验备件是否需回运，需回运的必须先发起回运（在途即可提交，A-11）。
  • 也用于「已驳回」单编辑后重新提交（A-13，从头走区域初审）。

开发类型：

  • 新增节点。

节点目录：

  • 服务商索赔管理 - 服务商索赔单列表（P01）- 详情/编辑页（P02）- 提交。

业务流转说明：

  • 服务商售后专员对 DRAFT 或 REJECTED（已驳回可编辑后重提，A-13）单点提交。系统重走一遍新建硬校验并叠加回运/维修凭证/条件必填校验，通过后单表更新 claim_order.status='PENDING_REGION' 并写一条提交操作留痕，同一事务、走乐观锁。从 REJECTED 重提同样回到 PENDING_REGION（从头走）。落地终态 PENDING_REGION（待区域初审），不触发下游（等区域初审）。

涉及数据表：

  • claim_order（索赔主单）：更新 status 与审计列，其余业务字段不动；repair_type/sub_repair_type（场景触发）、vin（整车条件必填）只读用于 A-39 硬校验（裁决 D-索赔-B9）。
  • claim_audit_record（审核记录）：写一条提交操作事件。
  • claim_fee_part（备件明细）：只读，用于回运与条件必填校验（battery_sn/old_part_code）。
  • attachment（附件）：只读，用于维修凭证必填与回运单附件判定。

解决方案：

  • 操作按钮：
    • 提交（只有 DRAFT 或 REJECTED 状态的可以提交）：
      • 前置校验：
        1. 走一遍新建的硬校验（工单仍为已完工、维修方案已过技术审核、保内、本服务商、条件唯一、工时费>0；技术审核前置来源 PRD F-CLAIM-01，本期由工单侧 Mock 保证）。
        2. 旧件回运（A-11）：逐条看 claim_fee_part 是否「需回运」；需回运的必须「已发起回运」才放行，否则拦【存在需回运备件未发起回运，不能提交】。
          • 「是否需回运」判定：读 claim_fee_part.part_attr 是否含「需回运」标识（B-2：当前无结构化布尔位，暂按属性文本判定，建议工单取数补布尔字段）。
          • 「已发起回运」判定：本模块未登记回运单实体/回运状态；按 A-11「售后仓点确认+回运单附件即视为已发起」，暂以「该单存在回运单附件」作为已发起证据（B-3：回运单实体/在途状态待挂载，附件判定为临时口径）。
        3. 维修凭证附件必填（有据·硬校验）：提交时必须至少 1 张维修凭证类附件（维修照片/检测报告等），无维修凭证不可提交（原始 PRD S4.1.1 服务商索赔单字段表明文「维修凭证附件…提交时必填」）。旧件资料附件仍为选填（原始 S4.1.1 标非必填），不随此项一起强制。
        4. A-39 条件必填 + 位长校验（硬校验，裁决 D-索赔-B9）（来源 PRD S8.6/A-39）：分类列 claim_order.repair_type/sub_repair_type 已按 B9 落库、场景触发有数据支撑，本期即做硬校验（内部场景豁免）：
          • claim_fee_part.battery_sn（电池包 SN）：电池包维修场景必填、须为 24 位，不满足 → 拦。
          • claim_fee_part.old_part_code（汽车电子旧件条码，Excel 批量场景）：汽车电子场景必填，缺失 → 拦。
          • claim_order.vin（整车 VIN）：整车场景必填、须为 17 位（VIN 列已按 B9 补入本链 claim_order，不再豁免本链校验；内部场景仍豁免，A-39）。
          • 场景触发口径：是否电池包/汽车电子/整车由 claim_order.repair_type/sub_repair_type（B9 已落库，工单库 Mock/待挂载带出值）判定；分类命中对应场景即按上述硬校验拦截，缺失/位长不符统一拦【条件必填/位长校验不通过】。
      • 若选择「是」，则更新以下数据（同一事务，乐观锁）：
        • claim_order：status = 'PENDING_REGION'；审计列刷新；其余业务字段不动
        • claim_audit_record（插一条操作事件）：biz_type='CLAIM'、audit_stage='OPERATION'、audit_result='SUBMIT'（已驳回重提为 RESUBMIT）、auditor/audit_time=操作人/时间、remark=操作摘要
      • 若选择「否」，则返回，不做处理。

F-CLAIM-08 责任判定

功能说明：

  • 质量部对外采件判定；结果三项：供应商责任/服务商责任/内部责任（A-27），录责任说明+承担比例；服务商责任→自动生成索赔抵扣单；供应商责任→触发供应商索赔流程；内部责任→记录比例/费用内部转嫁。
  • 判责完成→待结算（PRD 完成标准，S6.1）。

开发类型：

  • 新增节点。

节点目录：

  • 服务商索赔管理 - 服务商索赔单列表（P01）- 判责录入弹窗 - 责任判定。

业务流转说明：

  • 质量部责任判定专员（只限【待责任判定】的自采件单）对 PENDING_LIABILITY 单录判责结果。系统同一事务写公共判责字段 + 判责操作留痕 +（如有）判责附件，主单统一进 PENDING_SETTLE；按结果再做：PROVIDER 生成索赔抵扣单并置 provider_deduct_started=1；SUPPLIER 仅置 supplier_claim_started=1（供应商索赔子流程 V2 范围，本期不落单）；INTERNAL 仅记录比例/说明（内部转嫁交 OA，本期待挂载）。走乐观锁。

涉及数据表：

  • claim_order（索赔主单）：写判责字段（liability_result/desc/ratio/by/time）、status、启动标志、审计列。
  • claim_audit_record（审核记录）：写一条判责留痕。
  • attachment（附件）：判责附件如有则按 biz_type=LIABILITY 逐条写入。
  • claim_deduct_order（索赔抵扣单）：PROVIDER 结果时插一行。

解决方案：

  • 操作按钮：
    • 判责录入（前置状态 PENDING_LIABILITY，只限质量部对自采件单）：
      • 要录的：责任判定结果 liability_result（三选一：SUPPLIER 供应商 / PROVIDER 服务商 / INTERNAL 内部，A-27）、责任说明 liability_desc（必填，≤200）、承担比例 liability_ratio（%，DECIMAL(5,2)；校验 0 < ratio ≤ 100，不允许 0；PROVIDER 责任必填，其余按需，D-索赔-B7 已确认）。可传判责附件（attachment，biz_type=LIABILITY）。
      • 提交则更新以下数据（三种结果都写的公共写入，同一事务，乐观锁）：
        • claim_order：liability_result=录入值、liability_desc=录入值、liability_ratio=录入值、liability_by=当前用户、liability_time=NOW()、status='PENDING_SETTLE'；审计列刷新
        • 判责操作留痕（F-CLAIM-08 判责是一个独立用户动作，须与新建/提交/审核等动作一致进审核时间线，F-CLAIM-05「全流程留痕/操作日志」）：claim_audit_record 插一条：biz_type='CLAIM'、biz_id=主单 id、audit_stage='LIABILITY'（责任判定阶段码，裁决 D-索赔-B10 已决增该码并落进 schema/DDL）、audit_result=判责结果 SUPPLIER/PROVIDER/INTERNAL（该三值已在 audit_result 列注释登记）、auditor=判定人、audit_time=NOW()、remark=责任说明（liability_desc）、attachment_ids=判责附件 id（如有）
        • 判责附件若有 → attachment 逐条插（biz_type=LIABILITY, biz_id=claim_order.id）
      • 按结果分别再做：
        • PROVIDER（服务商责任）→ 自动生成索赔抵扣单：
          • claim_order：provider_deduct_started=1
          • claim_deduct_order 插一行：id=雪花；deduct_no=生成 DKD 号；claim_id=主单 id；claim_no=主单号；provider_id=主单 provider_id；work_order_no=主单 work_order_no；liability_result='PROVIDER'；liability_ratio=录入比例；claim_liable_amount = claim_order.apply_total_amount（抵扣针对服务商全额应付，D-索赔-D1 已确认）；deduct_amount=claim_liable_amount × liability_ratio / 100（A-30）；settle_period_belong=该索赔单 claim_audit_record 中「业务审核通过」的 DATE(audit_time)（A-30，按闪欣审批通过时间归入同周期，不得写 NULL）；status='UNSETTLED'
          • 抵扣单被结算链汇总时冲减服务商应付（本链只负责生成，不管结算）
        • SUPPLIER（供应商责任）→ 触发供应商索赔流程：
          • claim_order：supplier_claim_started=1
          • 供应商索赔子流程为 V2 范围（PRD S5「9月项」）；本期只置 supplier_claim_started=1、标记待挂载，不生成实际单据或占位单（D-索赔-B5 已确认）。备件费走供应商索赔（S8.3），本期不落单
        • INTERNAL（内部责任）→ 记录比例，内部转嫁：
          • 仅记录 liability_result/liability_ratio/liability_desc（公共写入已含）；费用内部转嫁交 OA，本期待挂载，不生成转嫁单（B-6，A-19）

编码规则：

  • deduct_no = DKD-yyyyMMdd-4位流水（PROVIDER 生成抵扣单时，从 doc_no_sequence 同事务行锁取号，deduct_no 唯一兜底；D-编码-D1/D2）。

外部接口：

  • QMS 判责系统：对接系统 QMS；触发时机为质量部判责录入（本期人工录入，A-20 Mock/待挂载）。契约未提供，V2 走 EXT-01 真实对接。失败处理：本期人工录入，无自动回传。
  • 供应商索赔流程：SUPPLIER 结果触发，本期待挂载仅置 supplier_claim_started=1，不生成单据（D-索赔-B5）。
  • OA 内部转嫁：INTERNAL 结果触发，本期待挂载仅记录比例/说明，不生成转嫁单（B-6/A-19，EXT-04）。

导入：

  • 无。

防重复与并发：

  • 主单判责字段 + 抵扣单 + 附件同一事务；抵扣单生成失败则判责整体回滚，主单不置待结算（避免「判了服务商责任却没抵扣单」）。
  • 前置状态白名单 PENDING_LIABILITY，且只 PENDING_LIABILITY→PENDING_SETTLE 单向迁移，天然防重复生成抵扣单；抵扣单以 deduct_no 唯一兜底。
  • 核心动作叠加按 claim_order.id 分布式锁串行化。
