后端任务验收规则

  • 基础设施按企业多模块拓扑搭建（parent + common + starter + 域 sdk/service，脚手架见 dev-standards/java/研发工程模块与SDK-Service拆分指南.md S7，权威 dev-standards/backend-java.md）：禁止自建 pom / 响应包装 / 异常体系。
  • 建表/改表由人执行、子智能体零 DDL：建表 SQL 是设计侧交付物（docs/db-schema/ddl/）；doc_no_sequence DDL 待研发组长审定并由人执行后，B01 编码组件方可开发。后端任务前置假设"目标表已由人建好"。
  • 基础设施任务由 Agent 自动连续执行，不触发用户门禁；验收：$MVN -q verify 通过（ArchUnit/Spotless/JaCoCo 门禁）、/actuator/health 返回 UP。
  • 业务任务必须在任务内完成真实联调验收（后端真实 API + 前端 VITE_USE_MOCK=false 切真实、页面核心操作可用、不展示 [Mock] 数据、Tester 能证明命中真实后端）。
  • 涉及外部服务（QMS/财务共享/OA/工单）的后端任务：本期一律走 Mock/内建适配层，标记 Mock 验收，不得宣称真实外部联调通过。Mock 必须可被同一适配接口的真实实现替换，不把外部地址/鉴权/正式枚举写死在业务层。

六、开发顺序建议

默认推荐节奏，实际开发以 Planner 产出的 .sdd/tasks.md 为准。

阶段1：前端 MVP（Mock，用户先验收 UI/UX）

  1. P01~P12 全部页面用 Mock 数据完成（Mock 格式对齐 api-contracts）
  2. 验收（用户门禁）：布局/配色/交互与原型一致

阶段2：后端基础设施（自动连续执行，不触发用户门禁）

  1. 按拆分指南 S7 初始化多模块骨架 + 统一响应/异常（Result/BizException + i18n）+ DB 连接（不建表）+ 鉴权拦截器；B01 单据编码组件（依赖 doc_no_sequence 已建）
  2. 验收（Agent 自动）：$MVN -q verify 通过、/actuator/health UP

阶段3：逐功能闭环开发（每个功能完成后触发用户门禁）

  1. 业务依赖顺序：结算基础配置 → 服务商索赔单 → 向欣旺达索赔单 → 结算链 → 付款申请链
  2. 每个功能 = 后端真实 API + 前端 Mock 切真实 + 联调验收
  3. 验收（用户门禁）：用户从前端页面操作验证功能可用

阶段4：E2E 回归测试

  1. 完整流程走通（新建索赔 → 初审 → 业务审核 →（向欣/判责）→ 结算 → 付款）
  2. 验收（用户门禁）：全流程通过，无阻塞性 Bug
