接口文档 · 服务商索赔单链

通用说明（认证 / 基础 URL / Result<T> / PageResult<T> / 审计列 / 通用错误码 / 编号规则）见 docs/conventions.md，本文件不重复。

上游权威：详细设计-服务商索赔单.md（F-CLAIM-01~08）、docs/db-schema/表结构文档.md（claim_order / claim_fee_labor / claim_fee_part / claim_fee_other / claim_audit_record / claim_deduct_order / attachment）。每接口 = LLD 一个操作，每字段 = 已登记真列或 PRD 字段。

范围：只覆盖服务商索赔单从新建到进入待结算的完整链。工单库取数一律 Mock/待挂载；向欣旺达索赔单、抵扣单被结算/质保金消费、供应商索赔、OA 转嫁属相邻链，本文只写触发点与关联键，不写其内部字段。

乐观锁由后端管理，version 不在任何请求/响应体出现（conventions S2/S5）。主单更新走 status 双条件 WHERE id=? AND status IN(<允许前置态>) AND deleted_at IS NULL，MyBatis-Plus @Version 在事务内自读当前版本并自增；命中 0 行即并发/状态已变，回滚返回 CLAIM_4008。并发冲突前端靠 status + 冲突错误码（CLAIM_4008）感知，不传 version。

2. 接口详情

2.1 新建暂存草稿

  • 接口名称：新建暂存草稿
  • 请求路径：/claim/order/create-from-work-order
  • 请求方式：POST
  • 描述：服务商售后专员在「新建索赔单」入口，按已完工保内工单带出信息生成一张索赔单草稿（LLD 第二节 / F-CLAIM-01）。

请求参数（Body - JSON）

参数名类型必选说明示例
workOrderIdString是源维修工单 id（落 claim_order.work_order_id，取数走工单库 Mock/待挂载）WO20260712000001
attachmentIdsString[]否草稿阶段可选上传的索赔单主体附件 id 列表（biz_type=CLAIM，空则不插；提交时再严格校验维修凭证）["1850000000000000001"]

主单业务字段（工单号/服务商/区域/系数/工时·备件·其它三类明细/主责件）由工单库 Mock 带出，非前端传参；claim_no/status/currency/审计列由后端生成。

请求示例

{  "workOrderId": "WO20260712000001",  "attachmentIds": ["1850000000000000001"]}

业务处理逻辑（写操作）

  1. 前置校验：
    • RBAC：登录角色为服务商售后专员，否则 403；数据范围按 provider_id 过滤（只动本服务商单）。
    • 工单准入（工单库 Mock/待挂载）：关联工单须「本服务商 + 状态=已完工 + 保内」，任一不满足 → CLAIM_4001。
    • 工单业务类型映射：按工单 work_order_biz_type 查 claim_type_mapping（enabled=1）得 claim_type；查不到 → CLAIM_4005（不可静默）。
    • 条件唯一：同一 work_order_id 已存在状态非 DECLINED/VOIDED 的 claim_order → 拦下（REJECTED 亦占用，不能另建）。为防并发穿透，先以同一 DB 连接取工单维度互斥锁 SELECT GET_LOCK(CONCAT('claim-wo-',MD5(workOrderId)),5)，取锁失败/超时 → CLAIM_4003；取锁后事务内再查唯一性，finally 用同连接 RELEASE_LOCK（连接级锁，取放锁间不得归还连接池）。
    • 金额硬校验（从宽）：工时费合计 > 0（Σ claim_fee_labor.amount），否则 CLAIM_4004；其它费用合计 ≥ 0。草稿阶段允许 claim_target 等必填项暂缺。
  2. 落库（同一事务）：
    • 取号：按 conventions S7 从 doc_no_sequence（前缀 SPCO）同事务行锁取 claim_no。
    • INSERT claim_order（一行）：id=雪花；claim_no=SPCO 号；status='DRAFT'；currency='CNY'；work_order_id/work_order_no/work_order_biz_type/provider_id/provider_code/provider_name/region=工单库带出（Mock/待挂载）；claim_type=映射结果（只读带入）；coefficient_snapshot=发起时点系数（优先工单库 Mock，无则查 claim_coefficient 按 provider_id 命中否则全局默认行的当前生效 coefficient）；main_part_id/main_part_name=工单主责件带出（Mock/待挂载，可空）；claim_target=NULL、project_code=NULL、liability_result/liability_desc/liability_ratio/liability_by/liability_time=NULL、supplier_claim_started=0、provider_deduct_started=0；labor_fee_total/part_mgmt_fee/other_fee_total=三张明细汇总回填，apply_total_amount=三者之和；create_by=当前用户、created_at=NOW()、version=0。
    • INSERT claim_fee_labor（按工单维修项目逐行）：claim_id=主单 id、project_name/labor_hours/labor_price=工单带出（Mock/待挂载）、amount=labor_hours × labor_price。
    • INSERT claim_fee_part（按工单备件逐行）：claim_id、part_name/part_attr/battery_sn/old_part_code=工单带出（Mock/待挂载）、claim_amount=0.00（恒 0）、mgmt_fee=销售价 × coefficient_snapshot。
    • INSERT claim_fee_other（按工单其它费用行）：claim_id、fee_type/amount/remark；editable=物流费等可人工改置 1，其余置 0（A-7）。
    • 附件（若 attachmentIds 非空）：逐个 INSERT attachment（biz_type='CLAIM'、biz_id=新单 id、upload_by=当前用户、upload_time=NOW()）。
    • INSERT claim_audit_record：biz_type='CLAIM'、biz_id=新单 id、audit_stage='OPERATION'、audit_result='CREATE'、auditor=当前用户、audit_time=NOW()、remark='新建草稿'。
  3. 状态流转：无 → DRAFT。
  4. 副作用：无下游触发。
  5. 事务边界：取号 + 主单 + 三张明细 + 附件 + 操作事件同一事务，任一失败整体回滚（回滚退号、不留半张单）；RELEASE_LOCK 在 finally 执行。

参数名类型说明示例
successBoolean业务是否成功true
codeString业务码"0"
messageString提示信息操作成功
dataJSON业务数据
├ idString新索赔单 id1860000000000000001
├ claimNoString索赔单编号SPCO-20260712-0007
├ statusString单据状态DRAFT
├ applyTotalAmountBigDecimal申请总金额（工时+备件管理+其它，自动）3860.00

2.2 编辑（仅其它费用）

  • 接口名称：编辑（仅其它费用）
  • 请求路径：/claim/order/{id}
  • 请求方式：PUT
  • 描述：服务商售后专员在草稿 / 已驳回态编辑「其它费用明细」（仅可编辑行），系统重算金额汇总（LLD 第四节 / F-CLAIM-03）。

请求参数（Path + Body - JSON）

参数名类型必选说明示例
idString是索赔单 id（Path）1860000000000000001
otherFeesJSON[]是其它费用明细编辑结果全集（仅 editable=1 行；未出现的既有 editable=1 行视为删除；editable=0 行禁止改删，若出现则拦截）
├ idString否明细行 id；有则更新，空则新增行1861000000000000005
├ feeTypeString是费用类型（claim_fee_other.fee_type，如物流费/托车费，≤32）物流费
├ amountBigDecimal是金额（claim_fee_other.amount，≥0）140.00
├ remarkString否备注（≤255，空写 NULL）顺丰到付

工时费明细、备件明细、备件管理费一律不可改（S8.2 固定，来源工单+系数），不在本请求体。

请求示例

{  "otherFees": [    { "id": "1861000000000000005", "feeType": "物流费", "amount": 140.00, "remark": "顺丰到付" },    { "feeType": "托车费", "amount": 200.00 }  ]}

业务处理逻辑（写操作）

  1. 前置校验：
    • RBAC：服务商售后专员 + 本服务商数据范围，否则 403。
    • 状态：前置 DRAFT 或 REJECTED（status 双条件 WHERE id=? AND status IN('DRAFT','REJECTED')，@Version 事务内自读版本自增，命中 0 行 → CLAIM_4008）。
    • 明细校验：每条 feeType 非空、amount≥0；请求内所有目标行必须为 editable=1，若命中 editable=0 行的改/删 → 400。
  2. 落库（同一事务）：
    • claim_fee_other：按 otherFees upsert（带 id 更新 fee_type/amount/remark；无 id 新增，editable=1）；既有 editable=1 行未在集合中出现的删除。editable=0 行不动。
    • UPDATE claim_order：重算 other_fee_total=Σ claim_fee_other.amount，随后 apply_total_amount=labor_fee_total + part_mgmt_fee + other_fee_total（labor/part 两项不变）；update_by=当前用户、updated_at=NOW()、version=version+1；status 不变。
    • INSERT claim_audit_record：audit_stage='OPERATION'、audit_result='EDIT'、auditor/audit_time=操作人/NOW()、remark=本次其它费用增删改摘要。
  3. 状态流转：不变（DRAFT 仍 DRAFT、REJECTED 仍 REJECTED）。
  4. 副作用：无下游触发。
  5. 事务边界：明细 upsert/删除 + 主单汇总回写 + 操作事件同一事务；金额必须一致回写，任一失败整体回滚。

codeString业务码"0"
messageString提示信息操作成功
dataJSON业务数据
├ idString索赔单 id1860000000000000001
├ otherFeeTotalBigDecimal重算后其它费用合计340.00
├ applyTotalAmountBigDecimal重算后申请总金额4060.00
traceIdString链路追踪 IDa1b2c3d4e5f6

响应示例（成功）

2.3 提交 / 重新提交

  • 接口名称：提交 / 重新提交
  • 请求路径：/claim/order/{id}/submit
  • 请求方式：POST
  • 描述：服务商售后专员提交草稿或重提已驳回单，通过硬校验后进入待区域初审（LLD 第三节 / F-CLAIM-02，已驳回重提同路径）。

请求参数（Path + Body - JSON）

参数名类型必选说明示例
idString是索赔单 id（Path）1860000000000000001

请求示例

{}

无请求体参数（幂等操作仅凭 Path id + 后端状态前置态判定）；示例见路径 POST /claim/order/{id}/submit。

业务处理逻辑（写操作）

  1. 前置校验：
    • RBAC：服务商售后专员 + 本服务商数据范围，否则 403。
    • 状态：前置 DRAFT 或 REJECTED（乐观锁双条件，命中 0 行 → CLAIM_4008）。
    • 硬校验（同新建）：工单仍「已完工 + 保内 + 本服务商」（工单库 Mock/待挂载）→ 不满足 CLAIM_4001；条件唯一仍成立；工时费合计 >0 → 否则 CLAIM_4004。
    • 旧件回运（A-11）：逐条看 claim_fee_part「是否需回运」——按 part_attr 文本是否含「需回运」标识判定（B-2：当前无结构化布尔位，暂按属性文本判定，待工单取数补布尔字段）；需回运者须「已发起回运」——按「该单存在回运单附件」判定（B-3：回运单实体/在途状态待挂载，附件为临时口径）；存在需回运未发起 → CLAIM_4006。
    • 维修凭证附件必填（有据·硬校验，原始 PRD S4.1.1）：attachment（biz_type='CLAIM'）中至少 1 张维修凭证类附件，缺失 → CLAIM_4007；旧件资料附件仍选填，不随此项强制。
  2. 落库（同一事务，乐观锁）：
    • UPDATE claim_order：status='PENDING_REGION'、update_by=当前用户、updated_at=NOW()、version=version+1；其余业务字段不动。
    • INSERT claim_audit_record：audit_stage='OPERATION'、audit_result= 由 DRAFT 提交为 'SUBMIT'、由 REJECTED 重提为 'RESUBMIT'、auditor/audit_time=操作人/NOW()、remark=操作摘要。
  3. 状态流转：DRAFT / REJECTED → PENDING_REGION（重提从头再走区域初审）。
  4. 副作用：无（等区域初审）。
  5. 事务边界：主单更新 + 操作事件同一事务；乐观锁未命中即回滚报冲突。

2.4 作废

  • 接口名称：作废
  • 请求路径：/claim/order/{id}/void
  • 请求方式：POST
  • 描述：服务商售后专员作废草稿态索赔单（仅草稿可作废，A-15）（LLD 第五节 / F-CLAIM-04）。

请求参数（Path + Body - JSON）

参数名类型必选说明示例
idString是索赔单 id（Path）1860000000000000001
remarkString否作废原因（写 claim_audit_record.remark，≤500，空则记「作废」）工单信息有误，重新发起

请求示例

{ "remark": "工单信息有误，重新发起" }

业务处理逻辑（写操作）

  1. 前置校验：
    • RBAC：服务商售后专员 + 本服务商数据范围，否则 403。
    • 状态：前置仅 DRAFT（status 双条件 WHERE id=? AND status='DRAFT'，@Version 事务内自读版本自增，命中 0 行（非草稿/并发）→ CLAIM_4008）。
  2. 落库（同一事务）：
    • UPDATE claim_order：status='VOIDED'、update_by=当前用户、updated_at=NOW()、version=version+1；deleted_at 保持 NULL（已作废是可查看业务终态，不做技术软删）。
    • 明细行 claim_fee_labor/claim_fee_part/claim_fee_other 与 attachment 保留不删（供全状态查看与留痕）。
    • INSERT claim_audit_record：audit_stage='OPERATION'、audit_result='VOID'、auditor/audit_time=操作人/NOW()、remark=作废原因或「作废」。
  3. 状态流转：DRAFT → VOIDED（终态）。作废后该工单可重新发起（条件唯一放行 VOIDED，A-15）。
  4. 副作用：无。
  5. 事务边界：主单状态更新 + 操作事件同一事务（明细/附件不动）；乐观锁未命中即回滚。

响应参数

参数名类型说明示例
successBoolean业务是否成功true
codeString业务码"0"
messageString提示信息已作废
dataJSON业务数据
├ idString索赔单 id1860000000000000001
├ statusString单据状态VOIDED
traceIdString链路追踪 IDa1b2c3d4e5f6

响应示例（成功）

{ "success": true, "code": "0", "message": "已作废", "data": { "id": "1860000000000000001", "status": "VOIDED" }, "traceId": "a1b2c3d4e5f6" }

2.5 区域初审

  • 接口名称：区域初审
  • 请求路径：/claim/order/{id}/region-review
  • 请求方式：POST
  • 描述：区域负责人核查工单真实性与资料完整性，录主责件归属并作出通过 / 驳回 / 拒绝三选一（LLD 第六节 / F-CLAIM-06）。

请求参数（Path + Body - JSON）

参数名类型必选说明示例
idString是索赔单 id（Path）1860000000000000001
auditResultString是审核结果，枚举 通过/驳回/拒绝（见附录 4.3）通过
claimTargetString条件主责件归属 XW=欣旺达件 / SELF=自采件（auditResult=通过 时必填；驳回/拒绝可空）XW
projectCodeString条件项目编码（claimTarget=XW 时必填；SELF 置 NULL）PRJ-XW-2026-018
remarkString条件核查意见 / 驳回原因 / 拒绝原因（驳回/拒绝 时必填，≤500）工单资料齐全，予以通过
attachmentIdsString[]否本次审核附件 id 列表（写 claim_audit_record.attachment_ids，空写 NULL）["1850000000000000021"]

请求示例

{ "auditResult": "通过", "claimTarget": "XW", "projectCode": "PRJ-XW-2026-018", "remark": "工单资料齐全，予以通过" }

业务处理逻辑（写操作）

  1. 前置校验：
    • RBAC：区域负责人，且数据范围只限本区域（SQL 拼 region 条件），否则 403。
    • 状态：前置 PENDING_REGION（乐观锁双条件，命中 0 行 → CLAIM_4008）。
    • 结果分支参数：auditResult=通过 时 claimTarget 必填、claimTarget=XW 时 projectCode 必填（缺失 → CLAIM_4009）；驳回/拒绝 时 remark 必填（缺失 → 400）。
    • 汽车电子自由填报工时把关（A-14）：一级分类=汽车电子的单，区域负责人在此判工时合理性，不合理走「驳回」。
  2. 电池包 SN 自动预判责任（A-26，可覆盖·非终判）：对含 battery_sn 的备件，系统按 SN 预判责任倾向供质量部参考。预判规则来源 QMS，本期 Mock/待挂载；预判值无专用登记列，本期并入本次 claim_audit_record.remark（D-索赔-B4：QMS 正式契约到位后再补预判字段），不写 claim_order.liability_result（终判专用）。
  3. 落库（同一事务，乐观锁）——按 auditResult 分三支：
    • 通过：UPDATE claim_order status='PENDING_HQ'、claim_target=录入值、project_code=录入值(XW) 或 NULL(SELF)、审计列刷新；INSERT claim_audit_record：biz_type='CLAIM'、biz_id=主单 id、audit_stage='区域初审'、auditor=当前用户、audit_time=NOW()、audit_result='通过'、remark=核查意见（含本期 Mock 的 SN 预判倾向）、approved_amount=NULL、attachment_ids=录入或 NULL。
    • 驳回：UPDATE claim_order status='REJECTED'、审计列刷新（claim_target/project_code 已录可保留，重提会从头再录）；INSERT claim_audit_record：audit_stage='区域初审'、audit_result='驳回'、remark=驳回原因、attachment_ids=录入或 NULL。
    • 拒绝：UPDATE claim_order status='DECLINED'、审计列刷新；INSERT claim_audit_record：audit_stage='区域初审'、audit_result='拒绝'、remark=拒绝原因。
  4. 状态流转：PENDING_REGION → PENDING_HQ（通过）/ REJECTED（驳回，可重提）/ DECLINED（拒绝，终态）。
  5. 副作用：无下游触发。
  6. 事务边界：主单状态 + 审核记录同一事务；乐观锁保并发。

响应参数

参数名类型说明示例

2.6 业务审核

  • 接口名称：业务审核
  • 请求路径：/claim/order/{id}/business-audit
  • 请求方式：POST
  • 描述：闪欣索赔专员（全国）复核，通过后按主责件归属唯一分流——XW 进待结算并生成向欣旺达索赔单，SELF 进待责任判定（LLD 第七节 / F-CLAIM-07）。

请求参数（Path + Body - JSON）

参数名类型必选说明示例
idString是索赔单 id（Path）1860000000000000001
auditResultString是审核结果，枚举 通过/驳回/拒绝（见附录 4.3）通过
remarkString条件审核意见 / 驳回原因 / 拒绝原因（驳回/拒绝 时必填，≤500）复核通过
attachmentIdsString[]否本次审核附件 id 列表（写 claim_audit_record.attachment_ids，空写 NULL）["1850000000000000031"]

请求示例

{ "auditResult": "通过", "remark": "复核通过" }

业务处理逻辑（写操作）

  1. 前置校验：
    • RBAC：闪欣索赔专员（全国数据范围），否则 403。
    • 状态：前置 PENDING_HQ（乐观锁双条件，命中 0 行 → CLAIM_4008）。
    • 驳回/拒绝 时 remark 必填，缺失 → 400。
  2. 落库（同一事务，乐观锁）——按 auditResult 与 claim_target 分支：
    • 通过 · claim_target=XW：UPDATE claim_order status='PENDING_SETTLE'、审计列刷新；INSERT claim_audit_record：audit_stage='业务审核'、audit_result='通过'、auditor/audit_time/remark。触发下游：逐单自动生成一张向欣旺达索赔单（xw_claim_order + xw_claim_detail 草稿，金额=本单对应金额不乘系数 A-37；字段填充属向欣旺达链 LLD，本文只登记触发与关联键 provider_claim_id=claim_order.id、provider_claim_no=claim_no、编号前缀 SXCO）。生成与主单状态更新同一事务（审核通过即有下游单）；claim_order 侧无 xw 关联回写列。
    • 通过 · claim_target=SELF：UPDATE claim_order status='PENDING_LIABILITY'、审计列刷新；INSERT claim_audit_record：audit_stage='业务审核'、audit_result='通过'。无自动生成，等质量部判责。
    • 驳回：UPDATE claim_order status='REJECTED'、审计列刷新；INSERT claim_audit_record：audit_stage='业务审核'、audit_result='驳回'、remark=原因。可重提（回 PENDING_REGION 从头）。
    • 拒绝：UPDATE claim_order status='DECLINED'、审计列刷新；INSERT claim_audit_record：audit_stage='业务审核'、audit_result='拒绝'、remark=原因。终态。
  3. 状态流转：PENDING_HQ → PENDING_SETTLE（XW 通过）/ PENDING_LIABILITY（SELF 通过）/ REJECTED（驳回）/ DECLINED（拒绝）。
  4. 副作用：XW 通过 → 生成向欣旺达单（本事务内，幂等靠下游 xw_claim_detail.uk_provider_claim 约束「一张服务商单仅纳入一张闪欣单」）；SELF/驳回/拒绝 无下游生成。
  5. 事务边界：主单 + 审核记录 +（XW 时）向欣旺达单生成，全在一个事务；向欣旺达单生成失败则整个审核回滚，主单不置通过（避免「通过了但没生成下游单」）。

2.7 判责录入

  • 接口名称：判责录入
  • 请求路径：/claim/order/{id}/liability
  • 请求方式：POST
  • 描述：质量部责任判定专员对自采件单录责任判定结果，进入待结算；服务商责任自动生成索赔抵扣单（LLD 第八节 / F-CLAIM-08）。

请求参数（Path + Body - JSON）

参数名类型必选说明示例
idString是索赔单 id（Path）1860000000000000002
liabilityResultString是责任判定结果 SUPPLIER/PROVIDER/INTERNAL（见附录 4.4）PROVIDER
liabilityDescString是责任说明（claim_order.liability_desc，≤200）服务商操作不当导致损坏
liabilityRatioBigDecimal条件承担比例（%，DECIMAL(5,2)，0<ratio≤100；PROVIDER 必填，其余按需，D-索赔-B7）80.00
attachmentIdsString[]否判责附件 id 列表（attachment，biz_type=LIABILITY，空则不插）["1850000000000000041"]

请求示例

{ "liabilityResult": "PROVIDER", "liabilityDesc": "服务商操作不当导致损坏", "liabilityRatio": 80.00, "attachmentIds": ["1850000000000000041"] }

业务处理逻辑（写操作）

  1. 前置校验：
    • RBAC：质量部责任判定专员，只限【待责任判定】的自采件单，否则 403。
    • 状态：前置 PENDING_LIABILITY（乐观锁双条件，命中 0 行 → CLAIM_4008）。
    • 字段：liabilityDesc 非空且 ≤200；liabilityRatio 满足 0<ratio≤100（不允许 0），PROVIDER 时必填——不满足 → CLAIM_4010。
  2. 公共写入（三种结果都写，同一事务，乐观锁）：
    • UPDATE claim_order：liability_result=录入值、liability_desc=录入值、liability_ratio=录入值、liability_by=当前用户、liability_time=NOW()、status='PENDING_SETTLE'、审计列刷新。
    • 判责附件若有 → 逐条 INSERT attachment（biz_type='LIABILITY'、biz_id=主单 id、upload_by/upload_time）。
  3. 按结果分别再做：
    • PROVIDER（服务商责任）→ 自动生成索赔抵扣单：UPDATE claim_order provider_deduct_started=1；取号（前缀 DKD，conventions S7）后 INSERT claim_deduct_order：id=雪花、deduct_no=DKD 号、claim_id=主单 id、claim_no=主单号、provider_id=主单 provider_id、work_order_no=主单 work_order_no、liability_result='PROVIDER'、liability_ratio=录入比例、claim_liable_amount=claim_order.apply_total_amount（针对服务商全额应付，D-索赔-D1）、deduct_amount=claim_liable_amount × liability_ratio / 100（A-30）、settle_period_belong=该单 claim_audit_record 中「业务审核 / 通过」的 DATE(audit_time)（A-30，不得写 NULL）、status='UNSETTLED'。抵扣单被结算链消费冲减服务商应付，本链只负责生成。
    • SUPPLIER（供应商责任）→ 触发供应商索赔：UPDATE claim_order supplier_claim_started=1、标记待挂载；供应商索赔子流程为 V2 范围（PRD S5「9月项」），本期不生成实际单据或占位单（D-索赔-B5）。
    • INTERNAL（内部责任）→ 记录比例：仅公共写入的 liability_result/liability_ratio/liability_desc；费用内部转嫁交 OA，本期待挂载，不生成转嫁单（B-6，A-19）。
  4. 状态流转：三种结果统一 PENDING_LIABILITY → PENDING_SETTLE。
  5. 副作用：PROVIDER 生成抵扣单（本事务内，幂等靠 deduct_no 唯一 + 单向迁移天然防重）；SUPPLIER/INTERNAL 仅置标志/记录，实际下游待挂载。
  6. 事务边界：主单判责字段 + 附件 +（PROVIDER 时）抵扣单同一事务；抵扣单生成失败则判责整体回滚，主单不置待结算（避免「判了服务商责任却没抵扣单」）。
