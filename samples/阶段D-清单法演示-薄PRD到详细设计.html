阶段 D 清单法演示：从薄 PRD → 完善详细设计（经销商新能源订单管理）

回答"那份完善设计是怎么从薄 PRD 得来的、清单法能否到达那个精细度"。

输入 = 薄 PRD（代提报只有一句"提交时匹配资源，只匹配现车，全部成功才提交否则拒绝"）。

证明 = 清单能把这一句展开成完整设计的每一个格子，并标清每格答案的来源。

精度公式

完善详细设计 = 薄PRD结构 + 参考已有节点逻辑(燃油车订单管理/审批单审批) + 既有库schema(表名/字段/码)
清单法 = 保证"每一格都被列出 + 每格标清来源"，不漏格、不瞎填

来源记号：【P】薄 PRD 已给｜【参考】需读参考节点(燃油车订单管理/审批单审批)｜【S】需既有库 schema(db-schema 纳管)｜【产品】需产品/OTD 决策

演示：把「代提报」一句话，展开成完整设计的所有格子

薄 PRD 原文只有：

提交订单时匹配资源（只匹配现车），若全部匹配成功，则可以提交，否则直接拒绝；OTD可代提报；增补订单有经销商车系限制，无 AIC 限制。

清单法逐格展开 ↓（右栏是这些格子在完整版设计里最终的答案，用来证明清单确实覆盖到了那个精细度）

格子组 1：可用条件 / 触发

问来源完整版最终答案
哪些状态可代提报？【P】暗示+【参考】仅"新建"
是否二次确认？文案？【参考】是，「您确认要提报选择的数据吗？」

格子组 2：资源匹配的精确条件（薄 PRD 只说"匹配现车"）

问来源完整版最终答案
查哪张库表？【S】车辆主数据 T_M_Z_BD_CAR
"现车"的精确判定条件？【参考】【S】CAR_STATE∈{2,12} && ONECAR_FLAG(No/null) && WAREHOUSE_CODE='A' && EXPECT_SUB_WAREHOUSE=null && LOGISTICS_CAR_FLAG=Yes && IS_SALE_CLOSED=Yes && 整车共享经销商=null && OFFLINE_DATE≠null
还要 join 哪张仓库表、加什么条件？【参考】【S】VLS_T_WH_VIN：ISDISTRIBUTION=未分车 && VLS_STATE=已入库 && CAR_QUALITY_STATUS=正常 && IS_LOGISTICS_SEPARABLE=否 && ISMODEL=否 && ZEBRASTATE=否 && ISLEND∈{空,归还} && INV_FROZEN=否
"经销商车系限制"怎么落？【P】+【S】产品查询车系限制表 T_M_Z_PRODUCT_SERIES_LIMIT（DLR_CODE+SERIES_CODE）
占用顺序？【参考】按 OFFLINE_DATE 升序

格子组 3：分支（薄 PRD 只说"全成功/否则拒绝" → 实际是三分支）

问来源完整版最终答案
分支有几个？⚠️【产品】三个：全匹配 / 部分匹配 / 无匹配（薄 PRD 只写了"全/否则"两个 → 这是薄 PRD 的缺口，清单逼出来）
无匹配怎么办？【P】不更新，提示「系统可用资源不足！」
部分匹配单据置什么状态？【参考】"部分待审批"

格子组 4：副作用——写哪些表 × 每表每字段每分支写什么值

薄 PRD 一个字没提"写表"。清单逼出 5 张表，每张的字段级写入：

表来源全匹配分支最终答案（节选）
T_V_Z_BU_DLR_ORDER【S】【参考】STATE=待审批, SUBMIT_TIME=当前, 修改人/时间
T_V_Z_BU_DLR_ORDER_D【S】【参考】行状态=待审批, 是否分车=Yes, 分车时间, VIN/APPROVE_VIN=匹配VIN, APPROVE_SON←生产计划
T_M_Z_BD_CAR【S】【参考】CAR_STATE=4, DLR_CODE=订单经销商, STOCK_LOCK_STATE=Yes, LOCK_BILL_NO/ID
T_M_Z_BD_CAR_CHANGE_HIS【参考】INSERT 异动类型=增补订单分配, 原状态→新状态4
T_V_Z_BU_PRODUCT_PLAN【S】【参考】ASSIGN_FLAG=Yes, ASSIGN_TYPE=12, 业务单据号/ID

格子组 5：编码

问来源完整版最终答案
采购订单号规则？【参考】参考经销商订单管理编码（考古项，形如 A17997OR...）

格子组 6：逆向（薄 PRD 没提"拒绝要回滚"）

问来源完整版最终答案
OTD 拒绝后，已占的车/生产计划怎么释放？【参考】逐表逆向：CAR 状态回滚(取异动历史原状态)、锁定清空、PRODUCT_PLAN ASSIGN 清空、写"分配拒绝"异动

这份演示证明了什么

  1. 清单能到达那个精细度：代提报从"一句话"被展开成 6 组、约 20 个格子，和完整版设计的结构一一对上（资源匹配条件、三分支、五表字段级、编码、逆向）。没有漏。
  2. 精度来自来源，不来自空想：上表右栏那些具体值，来源几乎都是【参考】+【S】——即参考节点逻辑 + 既有 schema。薄 PRD（【P】）只贡献了结构和少数大逻辑。
  3. 薄 PRD 的真实缺口也被逼出来：如"三分支"（薄 PRD 只写了两个）、"车系限制怎么落表"——这些进【产品】待决。

我要到达完整版精度，需要的输入

输入对应 SDD 机制现状
参考节点(燃油车订单管理/审批单审批)的规格或代码phase-D 第 7 节「依赖既有逻辑(考古项)」❌ 待提供
既有库 schema(表名/字段/枚举码)db-schema 纳管（提案第三节，尚未落地）❌ 待落地
薄 PRD 的少数产品决策(如三分支/车系限制)补充需求清单🟡 少量

给齐前两项，我就能把这份薄 PRD 产出成和样例同精度的详细设计；给不齐，我产出的是"格子全、来源标清、待填项明确"的清单——不编造表名。
