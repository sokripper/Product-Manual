1. 查询规格（市场活动维护主界面）

1.1 查询条件 ✅

活动编码(模糊) / 活动名称(模糊) / 活动类型(全部·品牌曝光度·促进销售) / 活动状态(默认除作废外全部) / 有效期范围(默认当月) / 计划起止日期(默认当月)。来源均为市场活动主表。

1.2 查询结果 ✅

活动编码(超链接→详情) / 名称 / 类型 / 状态 / 有效起止 / 计划起止 / 实际起止 / 创建人 / 最后修改时间。

1.3 数据范围过滤 ❌ B7

  • 硬过滤：活动状态默认排除"作废"。✅
  • 数据权限：「操作人按组织机构，只能看本组织机构下的活动」——❌ 组织机构模型未定义，过滤条件写不了。需产品给出：组织机构树结构 + 活动归属字段（按创建人所属公司？按经销商？）。

2. 表单与明细字段规格

2.1 市场活动 新增/编辑表单 ✅（除 B9）

活动编码(只读自动生成) / 名称(必填≤100) / 描述(≤500) / 类型(必填单选) / 状态(只读系统控) / 计划起止(必填) / 实际起止(选填) / 举办地址(≤200) / 预算金额(必填 ZAR 2位) / 执行人 / 有效起止(必填) / 备注(≤500)。

  • ⚠️ B9「是否大客户专场」：无必填/规则/界面位置，产品需定是否保留及规则。

2.2 活动日历 新增/编辑表单 ❌ B3

  • 年份(必填) + 月份 or 季度？——❌ 主界面按季度、新增按月份，粒度冲突，表单画不出来。
  • 每活动类型的 活动数量(≥0整数) / 活动预算(ZAR 2位)。

2.3 活动日历清单表（明细网格）⚠️ B4

字段规则问题
活动类型单选:品牌曝光度/促进销售✅
活动预算金额PRD 写"整数≥0，对应数量"⚠️ 规则写反，应为"数值 2 位小数 ZAR"
活动数量PRD 写"2位小数 ZAR，对应预算"⚠️ 规则写反，应为"整数≥0"

4. 操作清单（节选卡点）

4.x 操作：审核-通过（总经理节点）❌ B2 + 结算接口 Mock

  • 副作用：更新活动状态 → 下一级；总经理通过后调「南非市场活动结算接口」。
    • ❌ 结算接口无契约（入参/出参/失败处理）→ 用户已选 Mock：定义 SettlementAdapter.settle(活动编码) 桩，返回成功/失败；真实契约待 GMP 提供。
    • ❌ B2：结算成功后置为"待验收"还是"验收完成"？验收环节与结算的先后未定，事务顺序写不了。

4.x 操作：附件上传 ❌ B6

  • 格式/大小校验二选一未定（20M+MP3 vs 10M+PDF）。

4.x 操作：详情页保存/提交 ❌ B8

  • PRD 详情页(2.4)含"保存/提交"，疑似复制错误，需产品确认删除。

6. 并发 / 幂等 / 回滚 ⚠️

  • 并发校验：PRD 多处写"并发校验"✅（保存/提交/审核/验收）——需乐观锁字段，PRD 未指定，建议用最后修改时间/版本号。
  • 逆向：驳回→回到"驳回"态可再编辑提交 ✅；无资源占用类操作，回滚简单（区别于新能源）。

7. 依赖既有逻辑（考古项）

  • 南非市场活动结算接口：❌ 外部 GMP/SAP，无契约 → Mock。
  • 业务单据编码规则：❌ 钉钉附件，未提供。
  • 组织机构/角色/审批人绑定：❌ 依赖平台底座，未定义。

8. 压测结论

模板本身没问题——能填的部分（查询、表单、审核记录、简单状态迁移）都干净装下。但这份 PRD 产不出一份完整详细设计，因为：

  • 4 个硬阻塞（B1 审批链 / B2 结算与验收先后 / B3 日历粒度 / B5 编码规则）直接让状态机、核心副作用、编码这三块写不完整；
  • 另有 5 个需产品澄清（B4/B6/B7/B8/B9）。

这恰恰证明了阶段 D 的价值：它把我第一轮 gap 分析里那些"文档层面的矛盾"，变成了"详细设计里画不下去的具体格子"——也就变成了给开发前，产品必须逐条补齐的清单。这份清单（阻塞点汇总表）就是产品要回去补的 TODO，补完才能冻结 feature-signoff 进开发。
