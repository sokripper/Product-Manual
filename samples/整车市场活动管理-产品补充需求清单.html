整车市场活动管理 — 产品补充需求清单

来源：用 PRD-Sales-75 真跑阶段 D（后端详细设计）时逼出的 9 个卡点（见 detail-design-整车市场活动管理-压测.md「阻塞点汇总」）。

用法：产品逐条在「产品答复」栏作答（可直接确认"我的建议默认"或改）。硬阻塞不补齐，详细设计写不完整，不能冻结 feature-signoff、不能进开发。

分级：🔴 硬阻塞（挡住状态机/副作用/编码核心）；🟡 待澄清（影响局部字段/校验）。

🔴 硬阻塞（4 条，必须先补）

B1　审批链不一致 —— "待市场部审核"没有审批人

  • 位置：状态枚举（表11）、角色权限（表25）、审核界面审批流（3.2）
  • 矛盾：状态枚举里有「待市场部审核」，但审核界面写的审批流是 采购部部长→销售部长→财务部长→总经理，不含市场部审核节点；角色表里市场部经理是申请人，却又挂了一串"待XX审核"。到底完整审批链是几级、每级谁审，对不上。
  • 需产品给出：完整审批链的有序节点 + 每个节点的审批角色。
  • 我的建议默认（供确认）：

录入 →(市场部经理提交)→ 待采购部部长审核 → 待销售部长审核 → 待财务部长审核 → 待总经理审核 →(总经理通过调结算)→ 待验收 → 待验收审核 → 验收完成

即删除"待市场部审核"节点（市场部只提交不自审）。

  • 卡住：状态机迁移边、审核界面按钮可用条件。
  • 产品答复：____

B2　结算与验收的先后矛盾

  • 位置：流程图（image1）、审核界面（3.2）、状态枚举
  • 矛盾：流程图是「总经理通过 → 立即调结算(SAP) → 之后才验收」；但状态里又有「验收完成」在最后。结算发生在验收之前还是之后？
  • 需产品给出：结算接口的调用时点，且明确结算后活动进入哪个状态。
  • 我的建议默认（供确认）：按流程图——总经理审核通过即调结算接口，成功后置「待验收」；验收/验收审批通过后置「验收完成」。（结算在前，验收在后）
  • 卡住：审核-通过操作的副作用顺序、事务边界。
  • 产品答复：____

B3　活动日历粒度 年/季/月 混用

  • 位置：日历主界面（1.1 按年/季度）、新增界面（1.2 按年/月份）、日历主表/清单表（按月份）
  • 矛盾：查询和卡片按季度展示，录入和存储按月份。到底以哪个为准。
  • 需产品给出：日历的统一粒度（年+季度 / 年+月份 / 年+月且卡片按季汇总）。
  • 我的建议默认（供确认）：存储与录入按"年+月份"；主界面卡片按季度汇总展示（一个季度卡片聚合该季 3 个月）。这样两处都能自洽。
  • 卡住：日历表单字段（月份控件是否存在）、卡片列表聚合逻辑。
  • 产品答复：____

B5　活动编码生成规则外置未提供

  • 位置：市场活动主表「活动编码 = 系统自动生成」；规则在钉钉《业务单据编码规则》
  • 缺失：PRD 未附编码规则；原型里是 HD22024000002，规则不明。
  • 需产品给出：活动编码的生成规则（前缀 / 日期段 / 流水位数 / 重置周期）。
  • 我的建议默认（供确认）：参照原型反推 HD2 + YYYY + 6位流水（如 HD22024000002）；请产品确认或给出钉钉文档里的真实规则。
  • 卡住：编码生成算法（详细设计第 5 节）。
  • 产品答复：____

🟡 待澄清（5 条，影响局部）

B4　活动日历清单表字段规则对调

  • 位置：活动日历清单表（表27）字段 4、5
  • 矛盾：预算金额写成"整数≥0，对应数量"；数量写成"2位小数 ZAR，对应预算"——两行规则互换了。
  • 需产品确认：是否即为笔误，纠正为「预算金额=数值2位小数 ZAR；数量=整数≥0」。
  • 我的建议默认：确认为笔误，按上述纠正。
  • 产品答复：____

B6　附件格式/大小规则矛盾

  • 位置：验收界面（2.3 正文：PNG/JPG/JPEG/MP3，单个 20M）vs 原型图（jpg/png/pdf，单个 10M）
  • 需产品确认：以哪套为准（含是否允许 MP3、单文件上限）。
  • 我的建议默认：取并集偏保守——jpg/png/jpeg/pdf，单个 ≤ 10M（如需 MP3 请明确说明用途）。
  • 产品答复：____

B7　组织机构数据权限模型未定义

  • 位置：查询「只能看到本组织机构下的活动」
  • 缺失：组织机构树结构、活动的归属字段（按创建人所属公司？按经销商？）未定义。
  • 需产品给出：组织机构模型 + 活动归属判定字段（本期若只有"南非公司"一个主体，可简化）。
  • 我的建议默认：本期单主体（南非公司），数据权限先按「创建人所属公司 = 当前用户所属公司」过滤；多主体扩展留 V2。
  • 产品答复：____

B8　详情页多余的"保存/提交"按钮

  • 位置：详情界面（2.4 操作区写了"保存/提交/状态=录入"）
  • 判断：疑似从新增页复制错误，详情页应只读。
  • 需产品确认：删除详情页的保存/提交按钮。
  • 我的建议默认：删除，详情页纯查看。
  • 产品答复：____

B9　"是否大客户专场"孤儿字段

  • 位置：市场活动主表字段 14
  • 缺失：无必填/规则/界面出现，全表仅此一处提及。
  • 需产品给出：保留还是删除；保留则给出取值、必填、界面位置、业务含义。
  • 我的建议默认：本期删除（无配套逻辑）；如属未来大客户专场功能，标 V2。
  • 产品答复：____

补齐后的动作

  1. 产品逐条作答上面 9 项 →
  2. 我据此改写 PRD 相应章节 + 补完详细设计（状态机/副作用/编码可写完整）→
  3. 生成并逐条确认 feature-signoff.md 冻结 →
  4. 进开发门禁。

结算接口按用户既定选择走 Mock（SettlementAdapter 桩），真实契约由 GMP 后续提供；此项不阻塞开发，但需在 feature-signoff 标注为降级验收。
